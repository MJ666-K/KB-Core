# Knowledge Core - 知识图谱设计文档

> 在现有 KB-Core（RAG + Agent + Skill + Tool）架构上引入**结构化知识图谱**，让 Agent 同时拥有"语义证据"和"结构化路由"。图存储用 **Neo4j**，前端可视化用 **D3.js**，与向量检索协同工作。

---

# 一、为什么需要知识图谱

## 1.1 现有方案的边界

现有 KB-Core 走 **RAG + Agent Loop** 路线：

```
Query → Hybrid Retrieval（dense + sparse + rerank）→ Agent 自主决策 → Skill/Tool 组合 → Answer
```

这套架构对**语义问答**很强，但对以下问题就吃力：

| 场景 | 纯 RAG 的短板 |
|---|---|
| "劳动调解要走哪些步骤？" | chunks 是文档切片，LLM 要自己拼出 1→2→3→4 流程，容易漏步骤或错序 |
| "调解会谈阶段需要哪些证据？" | chunks 是平铺的，没有"流程节点 ↔ 证据节点"的强关联 |
| "调解失败能直接司法确认吗？" | LLM 不知道"成功→司法确认"和"失败→归档"是分支点，可能答错 |
| "这个案子能赢吗？" | 案例和法条散落在不同 chunks，缺少"案例→证据→法条"闭环 |
| "我们知识库覆盖了哪些业务？" | 没法盘点已有节点、识别空白 |

**根因**：向量是"扁平相似度"，没有"关系"。

## 1.2 图谱能补什么

知识图谱提供**结构化路由**：

```
用户问题
  ↓
  图谱：定位到流程入口节点
  ↓
  沿边遍历：流程节点 → 证据节点 → 法规节点 → 案例节点
  ↓
  向量：每个节点关联的 chunks 提供法条原文 / 案例判旨
  ↓
  联合回答
```

| 能力 | 图谱能给的 |
|---|---|
| 流程路由 | BFS 最短路、子图遍历、步骤排序 |
| 证据清单 | 实线/虚线区分"必需 / 可选" |
| 法条溯源 | 流程节点 ↔ 法规节点 ↔ chunks 三段引用 |
| 案例闭环 | 案例 ↔ 证据 ↔ 法条三角验证 |
| 知识盘点 | 按 type/category 聚合，识别 `blank_nodes_todo` |

## 1.3 与向量检索的关系（不是替代，是互补）

```
          ┌──────────────┐
          │  QueryAgent  │
          └──────┬───────┘
                 │
        ┌────────┴────────┐
        ▼                 ▼
   ┌─────────┐       ┌─────────┐
   │  图谱轨  │       │  检索轨  │
   │ (骨架)   │       │ (血肉)   │
   └────┬────┘       └────┬────┘
        │                 │
   流程 / 清单 / 路径   法条原文 / 判旨
   节点 / 边            chunks
        │                 │
        └────────┬────────┘
                 ▼
           结构化回答 + 引用
```

**双轨并行**（不是串行）：
- Agent 自主决定**是否需要图谱**（问流程/清单/路径才走图谱）
- 一旦走图谱，**并行调用**图谱和检索，最后综合

**设计原则**：
- 图谱不直接给最终答案（避免双重 LLM）
- Skill 返回结构化 `{answer, graphPath, citedChunks, nextSteps, ...}`，前端可渲染成可追溯卡片

# 二、技术选型

## 2.1 图数据库：Neo4j 5.x Community

| 候选 | 优势 | 劣势 | 结论 |
|---|---|---|---|
| **Neo4j** | 原生图存储、Cypher 表达力强、社区成熟、APOC 扩展 | 内存占用、运维稍重 | ✅ **采用** |
| PostgreSQL + 递归 CTE | 复用现有 DB | JOIN 性能差、路径查询表达力弱 | ❌ |
| ArangoDB | 多模型 | 社区比 Neo4j 小、生态弱 | ❌ |
| NebulaGraph | 国产、高性能 | 运维复杂、文档英文为主 | ❌ |
| Memgraph | 性能优秀 | 商业限制、社区小 | ❌ |

**选 Neo4j 的关键理由**：
- Cypher 是图查询的事实标准（类似 SQL 的地位）
- APOC 插件提供 `apoc.create.relationship`，让关系类型可动态化
- Browser UI（7474 端口）开发期直接看图谱，免去自研可视化
- 与 kg-data.json 的"节点-边"模型**一一对应**，零阻抗

## 2.2 前端可视化：D3.js

| 候选 | 优势 | 劣势 | 结论 |
|---|---|---|---|
| **D3.js** | 完全控制样式、力导向成熟、社区文档丰富 | 学习曲线陡 | ✅ **采用** |
| AntV G6 | 业务图谱友好、开箱即用 | 样式定制不如 D3 灵活 | ❌ |
| react-force-graph | 轻量、API 简单 | 样式定制弱 | ❌ |
| Cytoscape.js | 图算法强 | 与 React 集成略繁琐 | ❌ |

**选 D3 的关键理由**：
- 我们要按 `meta.legend` 严格控制**节点形状（矩形/圆/菱形/圆角矩形）+ 颜色 + 边虚实**，D3 SVG 完全可控
- 力导向布局（`d3-force`）天然适合"流程 + 证据 + 法条"这类关系网
- 不引入额外图表库依赖，减少包体积

## 2.3 与现有技术栈的关系

| 层面 | 现有 | 知识图谱新增 |
|---|---|---|
| 运行时 | Bun | — |
| 语言 | TypeScript (strict) | `neo4j-driver`（官方 TS SDK） |
| ORM | Drizzle (Postgres) | Neo4j 用 Cypher，不走 ORM |
| Web 框架 | Hono | 加 `/kg/*` 路由 |
| 向量库 | Postgres + pgvector | 复用，chunks 通过 `kg_node_id` 反向引用图谱节点 |
| 前端 | React + Antd + Vite | 加 D3.js（`d3` + `@types/d3`） |

**关键决策**：Neo4j 不替代 Postgres，二者**各司其职**：
- Neo4j：图结构、关系遍历
- Postgres：文档/chunks/向量/业务数据（用户、角色、权限、会话）

# 四、数据模型

## 4.1 Neo4j 节点类型（对应 kg-data.json 的 type）

```
:Flow      流程节点     stepOrder: int?
:Law       法规节点     lawText: string?
:Evidence  证据节点     description: string?
:Case      案例节点     fact: string?
```

**通用属性**（所有节点都有）：

| 属性 | 类型 | 说明 |
|---|---|---|
| `id` | string | 业务主键（如 `flow_labor_apply`），全局唯一 |
| `label` | string | 显示名（如 `提交调解申请`） |
| `category` | string | 业务分类（如 `劳动调解`、`邻里调解`） |
| `datasetId` | int | 归属数据集，关联 Postgres `datasets.id` |
| `chunkId` | string? | 关联 Postgres `chunks.id`，双向引用 |
| `meta` | json | 扩展字段（duration / outputDoc / output_doc 等） |

**Label 选择理由**：Neo4j Label 是一级公民，`MATCH (n:Flow)` 走索引，比 `MATCH (n) WHERE n.type='Flow'` 快一个数量级。

## 4.2 Neo4j 关系类型（对应 kg-data.json 的 edge）

| 关系类型 | 来源 | 去向 | 语义 |
|---|---|---|---|
| `:NEXT` | :Flow | :Flow | 流程下一步（实线） |
| `:BRANCH_TO` | :Flow | :Flow | 流程分支（如"拒绝→终止"） |
| `:APPLIES_TO` | :Law | :Flow | 法律依据（实线） |
| `:REQUIRES` | :Flow | :Evidence | 需要证据（虚线，必需） |
| `:MAY_REQUIRE` | :Flow | :Evidence | 需要证据（虚线，可选） |
| `:REFERS_TO` | :Case | :Flow | 案例参考某流程 |
| `:KEY_EVIDENCE` | :Case | :Evidence | 案例涉及关键证据 |
| `:CITES` | :Case | :Law | 案例援引法规 |
| `:RELATED` | 任意 | 任意 | 兜底关系 |

**关系属性**：

| 属性 | 类型 | 说明 |
|---|---|---|
| `solid` | bool | true=实线 / false=虚线，前端 `stroke-dasharray` 用 |
| `label` | string | 边的中文标签（`下一步` / `法律依据` / `关键证据`） |
| `article` | string? | 法条引用（如 `第十条`） |

## 4.3 kg-data.json 到 Neo4j 的映射

| JSON 字段 | Neo4j 映射 |
|---|---|
| `nodes[].id` | 节点 `id` |
| `nodes[].type="流程"` | Label `:Flow` |
| `nodes[].type="法规"` | Label `:Law` |
| `nodes[].type="证据"` | Label `:Evidence` |
| `nodes[].type="案例"` | Label `:Case` |
| `nodes[].label` | 节点 `label` |
| `nodes[].category` | 节点 `category` |
| `nodes[].step_order` | 节点 `stepOrder`（仅 Flow） |
| `nodes[].law` / `nodes[].case` / `nodes[].output_doc` | 节点 `meta` json 字段 |
| `edges[].from` / `edges[].to` | 关系的 startNode / endNode |
| `edges[].type="solid"` | 关系属性 `solid=true` |
| `edges[].type="dashed"` | 关系属性 `solid=false` |
| `edges[].label` | 关系属性 `label` |

## 4.4 索引与约束

```cypher
-- 唯一性约束（按 Label 拆开，避免 id 跨类型冲突）
CREATE CONSTRAINT flow_id  IF NOT EXISTS FOR (n:Flow)     REQUIRE n.id IS UNIQUE;
CREATE CONSTRAINT law_id   IF NOT EXISTS FOR (n:Law)      REQUIRE n.id IS UNIQUE;
CREATE CONSTRAINT ev_id    IF NOT EXISTS FOR (n:Evidence) REQUIRE n.id IS UNIQUE;
CREATE CONSTRAINT case_id  IF NOT EXISTS FOR (n:Case)     REQUIRE n.id IS UNIQUE;

-- 业务索引
CREATE INDEX flow_category   IF NOT EXISTS FOR (n:Flow)     ON (n.category);
CREATE INDEX flow_step       IF NOT EXISTS FOR (n:Flow)     ON (n.stepOrder);
CREATE INDEX law_category    IF NOT EXISTS FOR (n:Law)      ON (n.category);
CREATE INDEX ev_category     IF NOT EXISTS FOR (n:Evidence) ON (n.category);
CREATE INDEX case_category   IF NOT EXISTS FOR (n:Case)     ON (n.category);

-- 全文索引（按 label 搜索，Neo4j 5.x）
CREATE FULLTEXT INDEX node_fulltext IF NOT EXISTS
  FOR (n:Flow|Law|Evidence|Case)
  ON EACH [n.label, n.category];
```

## 4.5 Postgres 侧的双向关联

`chunks` 表新增一个字段，把图谱节点和向量切片绑定：

```sql
ALTER TABLE chunks ADD COLUMN kg_node_id VARCHAR(128);
CREATE INDEX chunk_kg_node_idx ON chunks(kg_node_id) WHERE kg_node_id IS NOT NULL;
```

**双向跳转**：
- 图谱节点 → 法条原文：`kg_to_chunk(nodeId)` → `chunks.id` → 原文
- 向量命中 → 反查图谱：`chunks.kg_node_id` → `MATCH (n {id: chunk.kg_node_id})`

# 五、Tool 层（6 个原子操作）

每个 Tool **原子、可独立测试、JSON Schema 声明输入输出**。

## 5.1 Tool 清单

| Tool | 输入 | 输出 | Cypher 核心 |
|---|---|---|---|
| `kg_search_nodes` | `{keyword, type?, category?, limit?}` | `Node[]` | 全文索引 / label 模糊匹配 |
| `kg_get_node` | `{id}` | `Node + 邻接边` | 节点 + 一跳邻居 |
| `kg_neighbors` | `{id, edgeType?, direction, solid?}` | `Neighbor[]` | 一跳邻居，支持过滤方向和关系类型 |
| `kg_path` | `{fromId, toId, maxDepth=5}` | `Path` | `shortestPath` BFS |
| `kg_subgraph` | `{rootIds[], depth=2, category?}` | `{nodes, edges}` | `[*1..depth]` 子图 |
| `kg_to_chunk` | `{nodeId}` | `{chunkId, docTitle, text}` | 反查 Postgres |

## 5.2 Tool 示例：kg_subgraph

```typescript
// app/src/kg/tools/kg_subgraph.ts
import { withSession } from "../client";
import { z } from "zod";

export const kgSubgraphTool = {
  name: "kg_subgraph",
  description:
    "围绕一组根节点拉取指定跳数内的子图。返回节点和边，前端可直接用 D3 渲染。",
  inputSchema: z.object({
    rootIds: z.array(z.string()).min(1).max(10),
    depth: z.number().int().min(1).max(3).default(2),
    category: z.string().optional(),
  }),
  async execute(input: z.infer<typeof this.inputSchema>) {
    return await withSession(async (session) => {
      const result = await session.run(
        `
        MATCH p=(root)-[*1..$depth]-(m)
        WHERE root.id IN $rootIds
          AND ($category IS NULL OR root.category = $category)
        WITH collect(p) AS paths
        UNWIND paths AS p
        UNWIND nodes(p) AS n
        WITH collect(DISTINCT n) AS nodes
        UNWIND nodes AS n1
        UNWIND nodes AS n2
        MATCH (n1)-[r]-(n2)
        RETURN
          [n IN nodes | {
            id: n.id, label: n.label, type: labels(n)[0],
            category: n.category, chunkId: n.chunkId,
            stepOrder: n.stepOrder, meta: n.meta
          }] AS nodes,
          collect(DISTINCT {
            from: startNode(r).id, to: endNode(r).id,
            type: type(r), solid: r.solid, label: r.label
          }) AS edges
        `,
        { rootIds: input.rootIds, depth: input.depth, category: input.category ?? null },
      );
      if (result.records.length === 0) return { nodes: [], edges: [] };
      const record = result.records[0];
      return {
        nodes: record.get("nodes"),
        edges: record.get("edges"),
      };
    });
  },
};
```

# 六、入库流程（kg-data.json → Neo4j）

## 6.1 入库脚本

```typescript
// app/src/kg/ingest.ts
import fs from "node:fs";
import { getNeo4j, withSession } from "./client";

interface KgData {
  meta: any;
  nodes: any[];
  edges: any[];
}

export async function ingestKgData(filePath: string, datasetId: number) {
  const raw = fs.readFileSync(filePath, "utf-8");
  const data: KgData = JSON.parse(raw);

  await withSession(async (session) => {
    // 清空该 dataset 旧数据（开发用，生产应保留）
    await session.run(
      "MATCH (n {datasetId: $did}) DETACH DELETE n",
      { did: datasetId },
    );

    // 按类型分组批量入库（UNWIND 比 N 次 CREATE 快 10 倍+）
    const groups: Array<[string, any[]]> = [
      ["Flow",     data.nodes.filter((n) => n.type === "流程")],
      ["Law",      data.nodes.filter((n) => n.type === "法规")],
      ["Evidence", data.nodes.filter((n) => n.type === "证据")],
      ["Case",     data.nodes.filter((n) => n.type === "案例")],
    ];

    for (const [label, nodes] of groups) {
      if (nodes.length === 0) continue;
      await session.run(
        `
        UNWIND $nodes AS n
        CREATE (node:${label} {
          id: n.id,
          label: n.label,
          category: n.category,
          datasetId: $did,
          stepOrder: n.step_order,
          chunkId: n.chunk_id,
          meta: apoc.convert.toMap(n)
        })
        `,
        { nodes, did: datasetId },
      );
    }

    // 批量入库边（用 apoc.create.relationship 让关系类型可动态化）
    const validEdges = data.edges.filter((e) => e.from && e.to);
    await session.run(
      `
      UNWIND $edges AS e
      MATCH (from {id: e.from})
      MATCH (to   {id: e.to})
      CALL apoc.create.relationship(
        from,
        CASE
          WHEN e.label = '下一步'         THEN 'NEXT'
          WHEN e.label = '同意调解'       THEN 'NEXT'
          WHEN e.label = '达成一致'       THEN 'NEXT'
          WHEN e.label = '申请司法确认'   THEN 'NEXT'
          WHEN e.label = '归档'           THEN 'NEXT'
          WHEN e.label = '法律依据'       THEN 'APPLIES_TO'
          WHEN e.label CONTAINS '关键证据' THEN 'KEY_EVIDENCE'
          WHEN e.label = '需要核对'       THEN 'REQUIRES'
          WHEN e.label = '需要材料'       THEN 'REQUIRES'
          WHEN e.label CONTAINS '需采集'   THEN 'REQUIRES'
          WHEN e.label CONTAINS '需调取'   THEN 'REQUIRES'
          WHEN e.label CONTAINS '需提供'   THEN 'REQUIRES'
          WHEN e.label CONTAINS '参考案例' THEN 'REFERS_TO'
          WHEN e.label CONTAINS '援引'     THEN 'CITES'
          WHEN e.type = 'dashed'          THEN 'REQUIRES'
          ELSE 'RELATED'
        END,
        { solid: (e.type = 'solid'), label: e.label },
        to
      ) YIELD rel
      RETURN count(rel) AS created
      `,
      { edges: validEdges },
    );
  });

  console.log(
    `✅ Ingested ${data.nodes.length} nodes, ${data.edges.length} edges (dataset=${datasetId})`,
  );
}

// CLI 入口
if (import.meta.main) {
  const filePath = process.argv[2];
  const datasetId = parseInt(process.argv[3] ?? "1", 10);
  await ingestKgData(filePath, datasetId);
  await getNeo4j().close();
}
```

## 6.2 关键 Cypher 解释

| 技巧 | 作用 |
|---|---|
| `UNWIND $nodes AS n` | 把列表展开批量插入，比 N 次 CREATE 快一个数量级 |
| `apoc.create.relationship(from, type, props, to)` | 让关系类型可以由数据动态决定，避免写 N 个 CASE WHEN 的 Cypher |
| `apoc.convert.toMap(n)` | 把整个 JSON 节点兜底塞进 `meta` 字段，前端能拿到所有原始数据 |
| `MATCH (n {datasetId: $did}) DETACH DELETE n` | `DETACH` 会同时删除该节点的所有关系，避免悬空边 |

## 6.3 kg-data.json 节点的法条同步入向量

kg-data.json 的法规节点 `law` 字段（如《劳动争议调解仲裁法》第十四条原文）目前只在图谱里。我们还需要把它走 splitter → embedding 入 Postgres chunks，让 `hybrid_search` 能直接命中法条原文。

```
kg-data.json 法规节点
  ↓ 提取 law 字段
  ↓ splitter（RecursiveSplitter）
  ↓ embedding（text-embedding-v3，1024 维）
  ↓ chunks 表写入，回填 kg_node_id
```

**这一步可以放在 ingest 脚本的同一次 pipeline，也可以拆成异步任务**。推荐异步：
- 图谱结构立即可用（流程/证据查询不依赖 chunks）
- 向量索引后台跑，不阻塞图谱功能上线

## 7.1 SKILL.md

```markdown
# mediation-advisor

## 名称
调解业务问答

## 描述
街镇调解业务问答。综合 Neo4j 图谱路由 + Postgres 向量检索，
给出"流程 + 证据 + 法律依据 + 案例"四要素的可追溯回答。

## 输入
- query: 用户问题
- category?: 劳动调解 | 邻里调解（可选，未传则 LLM 路由）

## 工作流
1. router_llm：
   - 识别 category（劳动调解 / 邻里调解）
   - 识别问题类型（流程 / 证据 / 法条 / 案例）
2. parallel_knowledge（双轨并行）：
   - graph_path: kg_subgraph(rootIds=[流程入口], depth=2, category)
   - evidence:   hybrid_search(query, top_k=8)
3. synthesis：
   - 主 LLM 综合 graph_path + chunks
   - 输出 {answer, graphPath, citedChunks, nextSteps, requiredEvidence, legalBasis}

## 输出结构
{
  "answer": string,            // 自然语言回答
  "graphPath": {...},          // 走过的图谱路径（节点 + 边）
  "citedChunks": [...],        // 引用的原文 chunks
  "nextSteps": [...],          // 下一步流程节点
  "requiredEvidence": [...],   // 所需证据
  "legalBasis": [...]          // 法律依据
}

## 终止策略
Skill 返回结构化结果后，主 Agent **直接透传**给前端，
不再二次 LLM 生成（遵循"Agent 不双重生成"原则）。
```

## 7.2 Skill 实现

```typescript
// app/src/skills/mediation-advisor/index.ts
import { kgSubgraphTool } from "../../kg/tools/kg_subgraph";
import { hybridSearchTool } from "../../retrieve/hybrid";

const ROOT_BY_CATEGORY: Record<string, string> = {
  劳动调解: "flow_labor_apply",
  邻里调解: "flow_neighbor_register",
};

export const mediationAdvisorSkill = {
  name: "mediation-advisor",
  description: "调解业务问答：流程指引 + 证据清单 + 法条溯源 + 案例参考",

  async run(ctx: SkillContext, input: { query: string; category?: string }) {
    // 1. 路由分类（LLM 自动判断 category）
    const category = input.category ?? (await inferCategory(ctx.llm, input.query));
    const rootId = ROOT_BY_CATEGORY[category] ?? ROOT_BY_CATEGORY["劳动调解"];

    // 2. 双轨并行（图谱 + 检索）
    const [subgraph, chunks] = await Promise.all([
      kgSubgraphTool.execute({ rootIds: [rootId], depth: 2, category }),
      hybridSearchTool.execute({ query: input.query, topK: 8, datasetIds: [ctx.datasetId] }),
    ]);

    // 3. 综合（一次 LLM 生成）
    const prompt = buildSynthesisPrompt(input.query, subgraph, chunks);
    const llmOut = await ctx.llm.generate(prompt);

    // 4. 结构化输出（前端可渲染成卡片）
    return {
      answer: llmOut.text,
      graphPath: subgraph,
      citedChunks: chunks.map((c) => ({
        id: c.id,
        docTitle: c.docTitle,
        text: c.text,
      })),
      nextSteps: extractFlowSteps(subgraph, rootId),
      requiredEvidence: subgraph.edges
        .filter((e) => e.type === "REQUIRES")
        .map((e) => subgraph.nodes.find((n) => n.id === e.to))
        .filter(Boolean),
      legalBasis: subgraph.edges
        .filter((e) => e.type === "APPLIES_TO" || e.type === "CITES")
        .map((e) => subgraph.nodes.find((n) => n.id === e.from))
        .filter(Boolean),
    };
  },
};

async function inferCategory(llm: LLMService, query: string): Promise<string> {
  const r = await llm.generate(
    `判断下面问题属于哪个调解类型，只输出"劳动调解"或"邻里调解"：\n${query}`,
  );
  return r.text.trim();
}
```

## 7.3 Agent Loop 路由策略

在 MainAgent 的 System Prompt 里固化图谱路由策略（参考 `docs/知识库设计.md` 第十三章）：

```
你拥有以下 Tool，其中 kg_* 是知识图谱工具：
```

## 8.3 渲染核心代码

```tsx
// status/src/pages/KnowledgeGraph/GraphCanvas.tsx
import { useEffect, useRef } from "react";
import * as d3 from "d3";
import type { KgNode, KgEdge } from "../../api/kgApi";

interface Props {
  data: { nodes: KgNode[]; edges: KgEdge[] };
  onNodeClick: (n: KgNode) => void;
  onNodeDoubleClick: (n: KgNode) => void;
}

export function GraphCanvas({ data, onNodeClick, onNodeDoubleClick }: Props) {
  const svgRef = useRef<SVGSVGElement>(null);

  useEffect(() => {
    if (!svgRef.current || data.nodes.length === 0) return;
    const svg = d3.select(svgRef.current);
    const { width, height } = svgRef.current.getBoundingClientRect();
    svg.selectAll("*").remove();

    // 1. 力导向布局
    const simulation = d3.forceSimulation(data.nodes as any)
      .force("link", d3.forceLink(data.edges).id((d: any) => d.id).distance(120))
      .force("charge", d3.forceManyBody().strength(-300))
      .force("center", d3.forceCenter(width / 2, height / 2));

    // 2. 箭头定义（solid / dashed 两套）
    svg.append("defs").selectAll("marker")
      .data(["solid", "dashed"])
      .enter().append("marker")
      .attr("id", (d) => `arrow-${d}`)
      .attr("viewBox", "0 -5 10 10").attr("refX", 20).attr("refY", 0)
      .attr("markerWidth", 6).attr("markerHeight", 6).attr("orient", "auto")
      .append("path").attr("d", "M0,-5L10,0L0,5").attr("fill", "#666");

    // 3. 边
    const links = svg.append("g").selectAll("line")
      .data(data.edges).enter().append("line")
      .attr("stroke", "#999").attr("stroke-width", 1.5)
      .attr("stroke-dasharray", (d) => (d.solid ? "0" : "5,5"))
      .attr("marker-end", (d) => `url(#arrow-${d.solid ? "solid" : "dashed"})`);

    // 4. 边标签
    const linkLabels = svg.append("g").selectAll("text")
      .data(data.edges).enter().append("text")
      .text((d) => d.label || "")
      .attr("font-size", 10).attr("fill", "#666");

    // 5. 节点容器
    const nodeGroups = svg.append("g").selectAll("g")
      .data(data.nodes).enter().append("g")
      .style("cursor", "pointer")
      .on("click", (_, d) => onNodeClick(d))
      .on("dblclick", (_, d) => onNodeDoubleClick(d))
      .call(d3.drag()
        .on("start", (event, d: any) => {
          if (!event.active) simulation.alphaTarget(0.3).restart();
          d.fx = d.x; d.fy = d.y;
        })
        .on("drag", (event, d: any) => { d.fx = event.x; d.fy = event.y; })
        .on("end", (event, d: any) => {
          if (!event.active) simulation.alphaTarget(0);
          d.fx = null; d.fy = null;
        }) as any);

    // 法规：蓝色矩形
    nodeGroups.filter((d: any) => d.type === "Law")
      .append("rect")
      .attr("width", (d: any) => Math.max(80, d.label.length * 14))
      .attr("height", 40).attr("rx", 4).attr("fill", "#1890ff");

    // 案例：绿色圆形
    nodeGroups.filter((d: any) => d.type === "Case")
      .append("circle").attr("r", 25).attr("fill", "#52c41a");

    // 证据：橙色菱形
    nodeGroups.filter((d: any) => d.type === "Evidence")
      .append("polygon")
      .attr("points", "0,-25 25,0 0,25 -25,0").attr("fill", "#fa8c16");

    // 流程：紫色圆角矩形
    nodeGroups.filter((d: any) => d.type === "Flow")
      .append("rect")
      .attr("width", (d: any) => Math.max(80, d.label.length * 14))
      .attr("height", 40).attr("rx", 12).attr("fill", "#722ed1");

    // 节点文字
    nodeGroups.append("text")
      .text((d: any) => d.label)
      .attr("text-anchor", "middle").attr("dy", 4)
      .attr("fill", "white").attr("font-size", 12);

    // 6. tick 更新位置
    simulation.on("tick", () => {
      links
        .attr("x1", (d: any) => d.source.x).attr("y1", (d: any) => d.source.y)
        .attr("x2", (d: any) => d.target.x).attr("y2", (d: any) => d.target.y);
      linkLabels
        .attr("x", (d: any) => (d.source.x + d.target.x) / 2)
        .attr("y", (d: any) => (d.source.y + d.target.y) / 2);
      nodeGroups.attr("transform", (d: any) => `translate(${d.x},${d.y})`);
    });

    return () => { simulation.stop(); };
  }, [data, onNodeClick, onNodeDoubleClick]);

  return <svg ref={svgRef} width="100%" height="100%" />;
}
```

# 十、安全与权限

## 10.1 多租户隔离

每个 dataset 下的图谱节点和 chunks 都带 `datasetId`：

```cypher
-- 所有 Tool 调用都强制带 datasetId 过滤
MATCH (n {datasetId: $datasetId, id: $id}) ...
```

**Tool 层强制注入**（不是 LLM 传，而是从会话上下文取）：

```typescript
async execute(input, ctx: ToolContext) {
  // ctx.datasetIds 来自 JWT 解析后的用户权限
  const cypher = `MATCH (n {datasetId: $datasetId, id: $id}) ...`;
  return await withSession((s) => s.run(cypher, { datasetId: ctx.datasetIds[0], id: input.id }));
}
```

## 10.2 角色权限

| 角色 | 图谱权限 |
|---|---|
| `admin` | 全部 dataset 的 `/kg/*` |
| `editor` | 自己 dataset 的 `/kg/*` + `/kg/ingest` |
| `viewer` | 自己 dataset 的只读 `/kg/*` |

**实现**：复用现有 RBAC（参考 `docs/知识库设计.md` 第十四章 Hooks），加一个 `kg:access` Hook 在 Tool 入口拦截：

```typescript
// app/src/hooks/kg_access.ts

};
```

# 十五、风险与未来

## 15.1 当前风险

| 风险 | 缓解 |
|---|---|
| kg-data.json 节点 id 与 chunks 无对应（`chunkId` 字段空） | 步骤 12 走一遍法条原文的 splitter + embedding 回填 |
| 关系类型枚举可能遗漏（kg-data.json 新增标签） | `apoc.create.relationship` 的 `ELSE 'RELATED'` 兜底 |
| Neo4j 内存占用大 | Community 版够用（< 10 万节点级别），大了再考虑 Enterprise |
| 双向同步（Postgres 删 chunk / Neo4j 删节点）易遗漏 | 引入 `chunks.kg_node_id` 后，删除 chunk 时同步触发 Neo4j 清理 |
| LLM 自主决定是否调图谱，可能误判 | system prompt 给出明确路由策略 + Hook 监控调用率 |

## 15.2 未来扩展

| 方向 | 内容 |
|---|---|
| 自动抽取 | 用 LLM 从 chunks 自动抽取节点和边（kg-data.json seed → 增量抽取） |
| 图算法 | 中心度分析（哪些法规被引用最多）、社区发现（哪些业务域高度耦合） |
| 时间维度 | 加 `effectiveAt` / `expiredAt` 属性，支持"现行有效 / 已废止" |
| 多模态节点 | 节点 `meta` 加 `imageUrl`，D3 渲染缩略图 |
| 跨图谱 | 多个 dataset 的图谱联邦查询（`CALL { USE ... }`） |
| 图谱版本 | 加 `:Version` 节点，记录每次 kg-data.json 入库的快照，支持回滚 |

## 15.3 与 kg-data.json `blank_nodes_todo` 的关系

kg-data.json 已经预留了"待补充节点"清单（商事合规、财税审计、招投标等）：

```json
"blank_nodes_todo": [
  "  - flow_commerce_* 系列节点（企业合规审查流程）",
  "  - flow_hr_audit_* 系列节点（用工风险排查流程）",
  ...
]
```

**这套设计天然支持扩展**：
- 新增 `flow_commerce_*` 节点 + 边，按现有 schema 入库即可
- 前端 KnowledgeGraph 左侧栏按 category 折叠，新加的 "商事合规" 自动出现
- Skill `mediation-advisor` 的 `ROOT_BY_CATEGORY` 加一行就能支持
