# Knowledge Core - 架构设计文档

> Agent 自主驱动。Skill 以 **DB `skill_definitions` 为运行时真相**，`SKILL.md` 仅作种子源。主 Agent 可调度多个子 Agent。

---

# 一、核心理念

**传统 RAG**：`Query → 检索 → 生成`（固定管线，所有问题走同一条路）

**本设计**：`Query → Agent（LLM）自主决策 → 按需调用 Skill / Tool → 自动编排组合 → 回答`

```
用户："对比 Docker 和 K8s 在微服务场景的优缺点"

Agent 自主思考:
  1. 这是对比类问题
  2. 调用 compare skill
  3. SkillExecutor 读 compare/SKILL.md 指令，LLM 按步骤执行
  4. 信息够了 → 生成对比表格

用户："你好"

Agent 自主思考:
  1. 闲聊，不需要检索
  2. 直接回答

分层原则：Agent 层灵活（LLM 自主决策 + 多 Agent 调度），Skill 层固化（instructions 定义最佳实践流程），Tool 层原子可测。
```

# 三、整体架构

```text
┌──────────────────────────────────────────────────────────┐
│              MainAgent（主调度 Agent）                     │
│  构建含子 Agent 列表的 System Prompt → 委托 QueryAgent     │
│  完成后异步 generateFollowUpSuggestions()                  │
└──────────────────────────┬───────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────┐
│              QueryAgent（核心 Agent Loop）                 │
│                                                          │
│  System Prompt（Skill + Tool + 子 Agent 描述）            │
│       +                                                  │
│  Function Definitions（Skill + Tool 的 JSON Schema）     │
│       ↓                                                  │
│  Agent Loop:                                             │
│    THINK  → 我该调什么？                                  │
│    CALL   → function calling（Skill / Tool / call_agent）│
│    OBSERVE → 看结果                                       │
│    DECIDE → 继续调？还是回答？                            │
│    ... 循环直到终止 ...                                   │
│                                                          │
│  三种终止路径：                                            │
│    ① Skill 返回 answer → 直接透传（不二次生成）            │
│    ② 只调了 Tool → 最后做一次 synthesis 合成               │
│    ③ LLM 直接回答（没调任何 function）                     │
└──────────┬─────────────────────────┬─────────────────────┘
           │                         │
     调用 Skill                直接调用 Tool / call_agent
           │                         │
┌──────────▼──────────────────┐  ┌───▼──────────────────────┐
│   Skill 层（DB 驱动）        │  │   Tool 层（6 个原子操作）  │
│                             │  │                          │
│  SkillLoader 从 DB 加载      │  │  search_knowledge        │
│  skill_definitions          │  │  get_document            │
│  启动时 DB 空 → seed SKILL.md│  │  get_chunk               │
│                             │  │  list_documents          │
│  SkillExecutor 执行时：      │  │  summarize_text          │
│    system = instructions    │  │  call_agent（仅主 Agent） │
│    LLM 按指令逐步调 Tool     │  │                          │
│                             │  │  ← SkillExecutor 内部调用 │
│  基础设施：                  │  └──────────────────────────┘
│  loader.ts / executor.ts    │
│  registry.ts（支持 reload） │
│                             │  ┌──────────────────────────┐
│  7 个内置 Skill（6 对外）    │  │   SubAgentRegistry       │
│  followups 为内部 Skill     │  │   DB agents + models       │
└─────────────────────────────┘  │   子 Agent 独立 QueryAgent │
                                 └──────────────────────────┘
                                 ┌──────────────────────────┐
                                 │   Hooks（横切拦截）       │
                                 │   audit / rate-limit      │
                                 │   异常隔离（不影响主流程） │
                                 └──────────────────────────┘
                                 ┌──────────────────────────┐
                                 │   Data 层                 │
                                 │   PostgreSQL + pgvector   │
                                 │   + tsvector + Redis      │
                                 └──────────────────────────┘
```

**关键设计**：
- Agent 没有"意图→Skill"硬编码映射。System Prompt 告诉 LLM 有哪些 Skill/子 Agent，LLM 自己判断。
- Skill 运行时以 **DB `skill_definitions`** 为真相；`SKILL.md` 仅在 DB 为空时作一次性种子。
- 主 Agent 通过 `call_agent` Tool 调度子 Agent；子 Agent 有独立 systemPrompt、model、datasetIds、skillNames 白名单。
- Agent 调用 Skill 时，SkillExecutor 读 instructions 作为 LLM 系统指令，LLM 按步骤执行。

## 5.3 chunks（Parent-Child + 双索引 + 外键修复）

```typescript
export const embeddingStatusEnum = pgEnum('embedding_status', ['pending', 'done', 'failed']);

export const chunks = pgTable('chunks', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id')
    .references(() => documents.id, { onDelete: 'cascade' }).notNull(),

  // 修复 #10：datasetId 有外键约束
  datasetId: uuid('dataset_id')
    .references(() => datasets.id, { onDelete: 'cascade' }).notNull(),

  parentId: uuid('parent_id'),        // self-FK 通过 migration SQL 添加

  // 修复 #12：拆为两个字段
  parentChunkIndex: integer('parent_chunk_index').notNull(),
  childIndexWithinParent: integer('child_index_within_parent'),  // null = parent 自身

  content: text('content').notNull(),
  contentHash: text('content_hash').notNull(),
  tokenCount: integer('token_count').notNull(),
  embedding: vector('embedding', { dimensions: 1024 }),
  embeddingStatus: embeddingStatusEnum('embedding_status').notNull().default('pending'),
  scope: text('scope').notNull().default('platform'),
  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => [
  index('chunk_embedding_idx').using('hnsw', table.embedding.op('vector_cosine_ops')),
  index('chunk_document_idx').on(table.documentId),
  index('chunk_parent_idx').on(table.parentId),
  index('chunk_dataset_idx').on(table.datasetId),
  uniqueIndex('chunk_doc_parent_child_uniq')
    .on(table.documentId, table.parentChunkIndex, table.childIndexWithinParent),
]);
```

# 六 ~ 十、数据层（Parser / Splitter / Embedding / Retrieve）

> 基础设施层，与实现一致。摘要：

| 模块 | 职责 |
|---|---|
| **Parser** | 文件 → Document（当前仅 TXT/MD，入库 pipeline 硬编码 `TxtParser`）|
| **Splitter** | Document → Parent(1200t) + Child(300t)，递归结构化切分 |
| **Embedding** | Child chunks → 1024 维向量（OpenAI 兼容 API），批量 + TTL/LRU 缓存 |
| **Retrieve** | Dense(pgvector) + Sparse(ILIKE + tsvector) → RRF → Reranker → Parent 去重 |

# 十一、Tool 系统

Tool 是原子操作，注册到 ToolRegistry，Agent 和 SkillExecutor 都可以调用。

| Tool | 描述 | 参数 | 可见范围 |
|---|---|---|---|
| `search_knowledge` | 混合检索知识库（支持多 datasetIds）| query, topK? | 全部 Agent |
| `get_document` | 获取文档元数据 | documentId | 全部 Agent |
| `get_chunk` | 获取 chunk + parent 上下文 | chunkId | 全部 Agent |
| `list_documents` | 列出文档 | limit? | 全部 Agent |
| `summarize_text` | LLM 文本摘要 | text, instruction? | 全部 Agent |
| `call_agent` | 调度子 Agent 执行问答 | agent_name, question; top_k?, max_iterations? | **仅主 Agent** |

> 文档管理（上传/删除/重入库）走 HTTP `/ingest` 和 `/api/documents`，不作为 Tool 暴露给 Agent。

# 十二、Skill 系统 ★（DB 驱动 + SKILL.md 种子）

## 12.1 设计理念

| 维度 | Phase 1（纯文件）| 当前实现 |
|---|---|---|
| 运行时真相 | SKILL.md 文件 | **DB `skill_definitions`** |
| 文件作用 | 唯一真相 | **种子源**（DB 为空时一次性导入）|
| 管理入口 | 改文件 + 重启 | **Admin API + 前端 Skills 页** |
| 热更新 | 不支持 | `SkillRegistry.reload()`（API 改后需手动 reload 或重启）|

## 12.2 加载流程

```
启动 → createSkillRegistry() → SkillRegistry.reload()
     → SkillLoader.loadAll() → SELECT skill_definitions WHERE enabled
     → 若 DB 空 → seedSkills() → 扫描 SKILL.md → INSERT（onConflictDoNothing）
```

## 12.3 SKILL.md 格式（种子源）

```markdown
---
name: qa
description: "基于知识库的事实问答。检索相关资料，LLM 生成带引用标注的回答。"
tools:
  - search_knowledge
parameters:
  type: object
  properties:
    query:
      type: string
      description: "用户的事实性问题"
  required:
    - query
---

# QA Skill

你的任务是根据知识库内容，回答用户的实际问题。

## 执行步骤

1. 从参数中取出 `query`
2. 调用 `search_knowledge` 工具，参数：{ "query": query, "topK": 10 }
3. 检查检索结果，如果为空回复"资料不足"
4. 根据资料生成回答，用 [1][2] 标注引用
5. 输出最终回答
```

**两部分**：
- **YAML frontmatter**（`---` 之间）→ SkillMetadata（name + description + tools + parameters）
- **Markdown 正文**（`---` 之后）→ instructions（LLM 执行指令）

```typescript
class SkillExecutor {
  async execute(
    instructions: string,          // SKILL.md 正文
    allowedTools: readonly string[], // frontmatter 声明的 tools 白名单
    ctx: SkillContext,
  ): Promise<SkillResult> {
    // 1. SKILL.md 正文 → system prompt
    // 2. params → user message
    // 3. 只暴露 allowedTools 声明的 Tool
    // 4. mini Agent Loop:
    //    LLM 按指令调 Tool → 看结果 → 继续调或回答
    // 5. 返回 SkillResult
  }
}
```

**执行流程**：

```
Agent 调用 skill "qa"
  ↓
SkillExecutor.execute(qa/SKILL.md 正文, ['search_knowledge'], ctx)
  ↓
  system: "你的任务是根据知识库回答问题。步骤：1. 调 search_knowledge..."
  user:   "query: Docker 是什么"
  tools:  [search_knowledge]
  ↓
  LLM 读指令 → 调 search_knowledge({ query: "Docker 是什么", topK: 10 })
  ↓
  结果返回给 LLM
  ↓
  LLM 按步骤 4 生成带 [1][2] 引用的回答
  ↓
  返回 SkillResult { answer, citations: [], toolCalls }
```

# 十三、Agent ★（MainAgent + QueryAgent + 三种终止路径）

## 13.1 多智能体架构

| 类 | 文件 | 职责 |
|---|---|---|
| `MainAgent` | `agent/main-agent.ts` | 主调度：构建含子 Agent 列表的 system prompt，委托 `QueryAgent`；完成后异步 `generateFollowUpSuggestions()` |
| `QueryAgent` | `agent/query-agent.ts` | 核心 Agent Loop：LLM function calling → Skill/Tool 执行 → 三种终止路径 |
| `SubAgentRegistry` | `agent/sub-agent-registry.ts` | 从 DB `agents` + `models` 加载子 Agent 实例，支持 `reload()` |
| `buildSystemPrompt` | `agent/system-prompt.ts` | 默认 / 主调度 / 子 Agent 三种 prompt 模板 |

子 Agent 通过 `call_agent` Tool 被主 Agent 调度。每个子 Agent 有独立的 systemPrompt、model、datasetIds、skillNames 白名单。子 Agent 的 Tool Registry **不含** `call_agent`（防递归）。

## 13.2 核心原则

Agent 是 **LLM + System Prompt + Function Calling Loop**。LLM 自主决策调什么 Skill/Tool/子 Agent。

## 13.3 三种终止路径（修复 #1 双重生成）

| 路径 | 条件 | 行为 |
|---|---|---|
| **skill** | LLM 调了 Skill | Skill 返回 answer，Agent **直接透传**（不二次生成） |
| **synthesis** | LLM 只调了 Tool（没调 Skill） | Agent 最后做一次合成生成 |
| **direct** | LLM 没调任何 function | 用 LLM 的 content 作为 answer |

## 13.4 Skill 结果处理（修复 #8 截断）

当 LLM 调用 Skill 时：
- SkillExecutor 返回 `{ answer, citations, toolCalls }`
- Agent **不** JSON.stringify 截断
- 只给 LLM 发摘要："Skill qa 完成，答案 1234 字，3 个引用"
- Agent 直接使用 Skill 的 answer 作为最终答案（不二次生成）

## 13.5 Agent Loop（核心代码结构）

```typescript
class QueryAgent {
  async execute(query, options): Promise<QueryResult> {
    // Agent Loop
    for (let iter = 0; iter < maxIterations; iter++) {
      const response = await llm.chat({ messages, tools: [...skills, ...tools] });

      if (!response.tool_calls?.length) {
        // Case C: LLM 直接回答
        directAnswer = response.content;
        break;
      }

      for (const call of response.tool_calls) {
        if (skillRegistry.has(call.name)) {
          // Case A: 调 Skill → SkillExecutor 执行
          const skill = skillRegistry.get(call.name);
          const executor = new SkillExecutor();
          const result = await executor.execute(
            skill.instructions,
            skill.metadata.tools,
            ctx,
          );
          skillResults.push(result);
          // 给 LLM 只发摘要（不截断原始内容）
          messages.push({ role: 'tool', content: summarize(result) });
        } else {
          // Case B: 调 Tool
          const result = await tool.execute(params, ctx);
          messages.push({ role: 'tool', content: JSON.stringify(result) });
        }
      }
    }

    // 决定最终答案
    if (skillResults.length > 0) {
      return { answer: skillResults[last].answer, termination: 'skill' };
    }
    if (directAnswer !== null) {
      return { answer: directAnswer, termination: 'direct' };
    }
    // synthesis 路径
    const final = await llm.chat({ messages: [...messages, "请给出最终回答"] });
    return { answer: final.content, termination: 'synthesis' };
  }
}
```

## 13.6 自主编排示例（含子 Agent 调度）

```
用户问："员工迟到，可以扣工资吗？"

[Iter 0] 主 Agent LLM CALL: call_agent({ agent_name: "mediation", question: "..." })

  → SubAgentRegistry 取 mediation 子 Agent
  → 绑定其 datasetIds + skillNames 白名单

  → 子 Agent 调 qa Skill → search_knowledge → 返回带引用回答

  → RESULT: { answer: "...", citations: [...], termination: 'skill' }

  主 Agent 透传子 Agent 结果。

[完成] 异步 generateFollowUpSuggestions() 生成推荐追问。
```

```typescript
// HookRegistry 内部用 try/catch 包裹每个 hook
```
