# 多 Agent 协作协议

## 为什么需要协议

单个 Agent 工作时，按 PROJECT.md → AGENTS.md 流程即可。但多个 Agent 同时工作时，
没有协议就会出现：两个 Agent 改同一个文件、上下文丢失、重复劳动。

这份协议解决三个问题：
1. 谁做什么（角色分工）
2. 做完怎么交给下一个人（交接协议）
3. 同时改同一文件怎么办（冲突处理）

---

## 角色定义

| 角色 | 职责 | 产出的文档 |
|------|------|-----------|
| **Architect** | 设计系统架构、功能方案、技术选型 | `openspec/add/`、`openspec/change/` |
| **Implementer** | 按设计文档实现代码 | 源代码、单测 |
| **QA** | 测试验证、回归测试、边界用例 | 测试报告 |
| **Doc** | 维护文档、同步规则、写 changelog | changelog、AGENTS.md 更新、docs/ |
| **Reviewer** | 代码审查、规范检查、一致性验证 | 审查意见 |

> 角色可兼任：小项目一个人身兼多角色是正常的。
> 关键是职责边界清晰，而不是每个角色必须不同的人。

---

## 入驻协议

Agent 进入项目的标准流程：

```
1. 读 PROJECT.md        → 项目全局：模块、目标、技术栈
2. 读 AGENTS.md         → 工作规则：铁律、测试命令、约定
3. 读最近 5 个 changelog → 变更脉络：最近改了什么、为什么
4. 读相关 docs/          → 业务规则：表结构、业务语义等
5. 读目标源码            → 具体实现细节
```

**不可跳级**——跳过 AGENTS.md 直接读源码会导致 Agent 对项目约定一无所知。

## 交接协议

### 场景：前一阶段完成，交给下一阶段

**完成者** 在交接时输出交接摘要（格式见下），**接手者** 必须先读完摘要再开始工作。

### 交接摘要格式

```markdown
## 交接摘要

- **阶段**：设计 / 实现 / 测试 / 审查
- **完成者**：[Agent 角色标识]
- **日期**：YYYY-MM-DD
- **完成了什么**：[1-2 句话概括]
- **改动/新建的文件**：
  - `path/to/file1` — [简要描述]
  - `path/to/file2` — [简要描述]
- **接手者注意**：
  - [需要特别注意的技术决策或坑]
- **未解决问题**：
  - [如有，列出待处理的问题]
- **下一步建议**：[建议下一步做什么]
```

### 交接注意事项

- 交接摘要可以追加到 changelog 末尾，也可以写在对应的 openspec 文档中
- 接手者如果看不懂前手的交接摘要，应该先查相关代码理解上下文，而不是跳过
- 交接摘要不是形式——它是防止"Agent B 不知道 Agent A 做了什么导致返工"的核心机制

## 并行规则

### 分工原则

按文件或模块分工，确保不同 Agent 的工作区域不重叠：

```
✅ 好：Agent A 负责 backend/auth，Agent B 负责 frontend/pages
❌ 坏：Agent A 和 Agent B 都在改 backend/auth.ts
```

### 文件冲突处理

当两个 Agent 确实需要同改一个文件时：

1. **先到先得**：先开始的 Agent 优先完成，后开始的等待
2. **追加不覆盖**：后改的 Agent 先读取前者的改动，以追加形式修改
3. **分歧仲裁**：如果两人的方案产生分歧，**呈现两个方案给用户决定**，不要自行覆盖

### 工作区域声明

并行工作前，每个 Agent 声明自己的工作区域：

```
🔒 我正在修改：
- src/auth/login.ts（重构登录逻辑）
- tests/auth.test.ts（新增测试）

✅ 安全区域（其他人可以使用）：
- src/auth/register.ts
- src/api/routes.ts
```

## 生命周期工作流

```
┌────────┐     ┌─────────┐     ┌──────┐     ┌──────────┐
│Architect│     │Implementer│     │  QA  │     │Doc+Review│
│ 设计    │ ──→ │ 实现     │ ──→ │ 测试 │ ──→ │ 文档+审查│
│         │     │          │     │      │     │          │
│产出：    │     │产出：     │     │产出： │     │产出：     │
│openspec │     │代码+测试  │     │测试   │     │changelog │
│设计文档  │     │          │     │报告   │     │AGENTS.md │
└────────┘     └─────────┘     └──────┘     └──────────┘
                     ↑              │
                     └──────────────┘
                     测试失败：交回实现
```

### 各环节交接点

| 环节 | 触发条件 | 交接给 | 交接内容 |
|------|---------|--------|---------|
| 设计完成 | 用户确认 openspec 文档 | Implementer | 设计文档 + 实现要点 |
| 实现完成 | 代码实现 + 自测通过 | QA | 文件列表 + 测试建议 |
| 测试通过 | 所有测试 pass | Doc + Reviewer | 变更文件 + 测试报告 |
| 测试失败 | 存在失败的测试 | Implementer | 失败用例 + 报错信息 |
| 文档完成 | changelog + 规则同步完成 | 用户 | 变更总结 + 验证清单 |
| 审查通过 | 代码和文档质量达标 | 用户 | 完成报告 |
| 审查不通过 | 发现问题 | Implementer | 问题列表 + 修改建议 |

## 冲突仲裁规则

当出现以下冲突时，按规则处理：

| 冲突类型 | 处理方式 |
|---------|---------|
| 两个 Agent 同改一个文件 | 先声明工作区域者优先；后者追加 |
| 设计分歧 | 列出方案对比，交给用户决定 |
| 规范不明确 | 追加到 AGENTS.md，形成新约定 |
| 测试标准不一致 | 以 AGENTS.md 中定义的测试命令为准 |
| 文档与代码不匹配 | **以代码实际行为为准**，然后修正文档 |
