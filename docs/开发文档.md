## 0.2 设计原则（含修复）

### 原则 1：分层 —— Agent 灵活，Skill 固化（修复 #7）

旧版设计文档说"没有硬编码流程"，这与 Skill 内部固定 `retrieve→generate→format` 矛盾。

**正确表述**：
- **Agent 层**：LLM 自主决策，不硬编码"问题→Skill"映射
- **Skill 层**：固化某类任务的最佳实践流程（可以由工程师调优，不需要 LLM 现场编排）
- **Tool 层**：纯原子操作，无流程

这不是矛盾，而是合理的关注点分离：让 LLM 做它擅长的（理解意图、组合 Skill），让代码做它擅长的（执行确定性的多步流程）。

### 原则 2：Skill 是一等公民（修复 #2）

Skill 自己声明完整的 JSON Schema，LLM 可以传结构化参数（不止 `query`）。例如：

```typescript
metadata: {
  name: 'compare',
  description: '...',
  parameters: {                  // ← Skill 自己声明
    type: 'object',
    properties: {
      query: { type: 'string' },
      entities: { type: 'array', items: { type: 'string' } },
      depth: { type: 'string', enum: ['brief', 'detailed'] }
    },
    required: ['query']
  }
}
```

### 原则 3：Agent 与 Skill 的契约（修复 #1 双重生成）

旧版：Skill 内部 `llm.generate` 出 `answer`，被 JSON.stringify 截断到 4000 字喂回 Agent，Agent 又调一次 `llm.chat` 重写一遍 —— 双重生成、引用丢失。

**新版契约**：
- Skill 返回 `SkillResult { answer, citations, toolCalls, raw?: RetrievalResult[] }`
- 如果 LLM 在 Agent Loop 中**调用了 Skill**：Agent 直接透传 Skill 的 `answer`，不二次生成
- 如果 LLM 在 Agent Loop 中**只调用了 Tool**（没调 Skill）：Agent 最后做一次合成生成

详见 Part 6。

### 原则 4：类型安全

`strict: true`，**禁止** `as any`、`@ts-ignore`、`@ts-expect-error`。需要类型断言时用 `as unknown as T` + 注释说明为什么类型系统推不出来。

### 原则 5：批量优先（修复 #4）

数据库操作能用批量就不用循环。一份 1000 chunk 的文档，旧版要 2000 次 round-trip，新版应该 ≤ 5 次。

### 原则 6：安全边界（修复 #5、#19）

- 所有 HTTP body 用 zod schema 校验
- 所有用户控制的字符串（文件名、路径）经过 sanitize
- JWT 认证 + RBAC 权限控制（10 项权限）；`requirePermission()` 用于用户/角色管理 API

## Step 1.2: tsconfig.json

**目标**：开启 strict + 现代 module 解析。

写 `tsconfig.json`：

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "esModuleInterop": true,
    "skipLibCheck": true,

    "outDir": "./dist",

    "declaration": true,
    "resolveJsonModule": true,
    "allowImportingTsExtensions": true,
    "noEmit": true,
    "types": ["bun-types"]
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

**每项配置的意义**：

| 配置 | 作用 | 为什么这样选 |
|---|---|---|
| `target: ES2022` | 编译目标 JS 版本 | Bun 支持 ES2022 全部特性（top-level await、class fields、`Error#cause`） |
| `module: ESNext` | 模块系统 | Bun 原生 ESM，不需要 CommonJS |
| `moduleResolution: bundler` | 模块解析策略 | Bun 像打包器一样解析（支持 `exports` 字段、子路径导入） |
| `strict: true` | 开启所有严格检查 | 项目核心约束，**不允许任何 `as any`** |
| `noUncheckedIndexedAccess` | 数组/对象索引访问返回 `T \| undefined` | 防止 `arr[0]` 假设一定存在 |
| `allowImportingTsExtensions` | 允许 `import './foo.ts'` | 显式写后缀，避免运行时找不到文件 |
| `noEmit: true` | 不输出 .js | Bun 直接运行 .ts，tsc 只做类型检查 |
| `types: ["bun-types"]` | 全局类型 | 让 `Bun.file`、`process` 等有类型 |

**常见坑**：
- `module: ESNext` + `moduleResolution: node`（旧版）会冲突，必须用 `bundler`
- 不要写 `lib`，让 `target` 自动推断
- `noUncheckedIndexedAccess` 会让旧代码大量出现 `possibly undefined`，写代码时要主动处理

**验证**：`bunx tsc --noEmit` 不报错（虽然现在 src 还空）。

## Step 1.9: src/utils/hash.ts

**目标**：稳定的哈希函数（用于文件查重、chunk 去重）。

写 `src/utils/hash.ts`：

```typescript
import { createHash } from 'crypto';

/** 对字符串做 SHA-256，返回 hex */
export function sha256(text: string): string {
  return createHash('sha256').update(text, 'utf8').digest('hex');
}

/** 对文件做 SHA-256，返回 hex（流式读取大文件不会爆内存） */
export async function fileHash(filePath: string): Promise<string> {
  const stream = Bun.file(filePath).stream();
  const hasher = createHash('sha256');
  for await (const chunk of stream) {
    hasher.update(chunk as Buffer);
  }
  return hasher.digest('hex');
}
```

**性能改进**：旧版用 `await Bun.file(filePath).arrayBuffer()` + `Buffer.from(file)`，对大文件会一次性读到内存。新版用 stream，恒定内存占用。

**为什么用 Node 的 `crypto` 而不是 `Bun.hash`**：`Bun.hash` 返回数字/特定格式，要 hex 还得自己转；`crypto.createHash` 是标准库 API，跨运行时通用，性能差异在 RAG 场景可忽略。

```typescript
import { config } from '../config';

type LogLevel = 'debug' | 'info' | 'warn' | 'error';

const LEVEL_PRIORITY: Record<LogLevel, number> = {
  debug: 10,
  info: 20,
  warn: 30,
  error: 40,
};

function log(level: LogLevel, msg: string, data?: unknown): void {
  if (LEVEL_PRIORITY[level] < LEVEL_PRIORITY[config.logLevel]) return;

  const timestamp = new Date().toISOString();
  const prefix = `[${timestamp}] [${level.toUpperCase()}]`;
  const fn = level === 'error' ? console.error : console.log;

  if (data !== undefined) {
    fn(prefix, msg, data);
  } else {
    fn(prefix, msg);
  }
}

  debug: (msg: string, data?: unknown) => log('debug', msg, data),
  info: (msg: string, data?: unknown) => log('info', msg, data),
  warn: (msg: string, data?: unknown) => log('warn', msg, data),
  error: (msg: string, data?: unknown) => log('error', msg, data),
};
```

## Step 1.11: src/cache/ttl-cache.ts（修复 #11：加 LRU）

**目标**：一个既有 TTL 又有 LRU 淘汰的缓存。旧版只有 lazy TTL，长尾查询会无限堆积导致内存泄漏。

写 `src/cache/ttl-cache.ts`：

```typescript
/**
 * TTL + LRU 混合缓存。
 * - TTL：超过时间自动失效（lazy，get 时检查）
 * - LRU：达到 maxSize 时淘汰最久未访问的
 *
 * 利用 JS Map 的插入顺序特性实现 LRU：
 * - get 时 delete + set，让被访问的 key 移到末尾
 * - 淘汰时删 Map.keys().next().value（头部 = 最旧）
 */
export class TTLCache<K, V> {
  private store = new Map<K, { expiresAt: number; value: V }>();
  private readonly ttlMs: number;
  private readonly maxSize: number;

  constructor(ttlMs: number, maxSize: number = 1000) {
    if (maxSize <= 0) throw new Error('maxSize must be positive');
    if (ttlMs <= 0) throw new Error('ttlMs must be positive');
    this.ttlMs = ttlMs;
    this.maxSize = maxSize;
  }

  get(key: K): V | undefined {
    const entry = this.store.get(key);
    if (!entry) return undefined;

    // 过期了：删除并返回 undefined
    if (Date.now() > entry.expiresAt) {
      this.store.delete(key);
      return undefined;
    }

    // LRU：delete + set 把 key 移到 Map 末尾（=最近访问）
    this.store.delete(key);
    this.store.set(key, entry);
    return entry.value;
  }

  set(key: K, value: V): void {
    // 已存在先删除，保证 set 后在末尾
    this.store.delete(key);

    // 容量超限：淘汰 Map 头部（最旧、最久未访问）
    while (this.store.size >= this.maxSize) {
      const oldestKey = this.store.keys().next().value;
      if (oldestKey === undefined) break;
      this.store.delete(oldestKey);
    }

    this.store.set(key, { expiresAt: Date.now() + this.ttlMs, value });
  }

  delete(key: K): void {
    this.store.delete(key);
  }

  clear(): void {
    this.store.clear();
  }

  get size(): number {
    return this.store.size;
  }
}
```

**修复说明（#11）**：

| 旧版问题 | 新版做法 |
|---|---|
| 只有 TTL，无大小上限 | 加 `maxSize` 参数，超限淘汰 |
| Lazy expiration 导致过期项仍占内存 | `get` 时主动 `delete`；`set` 时按 LRU 淘汰 |
| 无访问顺序追踪 | 利用 JS Map 的插入顺序，get 时 delete+set 移到末尾 |

**为什么用 Map 而不是对象**：
- Map 保持插入顺序（对象不保证）
- Map 的 key 可以是任意类型
- Map 删除是 O(1)

**LRU 的核心技巧**：
- `this.store.delete(key); this.store.set(key, entry);` 让"被访问"的 key 移到 Map 末尾
- `this.store.keys().next().value` 拿到 Map 头部（最旧的）
- 整个 LRU 不需要双向链表，靠 Map 的迭代顺序实现

**验证**：写 `tests/ttl-cache.test.ts`：

## Step 2.3: src/db/schema/document.ts

**目标**：文档元数据表。

写 `src/db/schema/document.ts`：

```typescript
import { pgTable, uuid, text, integer, timestamp, pgEnum } from 'drizzle-orm/pg-core';
import { datasets } from './dataset';

/**
 * 文档状态机：
 *   pending → parsing → chunking → embedding → ready
 *                                  ↘ failed
 */
export const documentStatusEnum = pgEnum('document_status', [
  'pending',     // 刚创建，等待 worker 拾取
  'parsing',     // 正在解析文件
  'chunking',    // 正在切分
  'embedding',   // 正在生成向量
  'ready',       // 可被检索
  'failed',      // 入库失败
]);

export const documents = pgTable('documents', {
  id: uuid('id').primaryKey().defaultRandom(),
  datasetId: uuid('dataset_id').references(() => datasets.id).notNull(),

  title: text('title').notNull(),
  docType: text('doc_type').notNull().default('general'),     // general | law | tutorial ...
  sourcePath: text('source_path').notNull(),                  // 原始文件路径
  fileHash: text('file_hash').notNull(),                      // 文件级 SHA-256（查重用）
  contentHash: text('content_hash'),                          // 文本内容 SHA-256（变更检测）
  fileSize: integer('file_size').notNull(),

  status: documentStatusEnum('status').notNull().default('pending'),
  errorMsg: text('error_msg'),                                // 失败原因
  embeddingModel: text('embedding_model'),                   // 用哪个模型 embedding 的（便于以后换模型）

  scope: text('scope').notNull().default('platform'),         // platform | tenant_xxx（多租户预留）
  ownerId: text('owner_id'),                                  // 上传者 ID（认证后填）

  createdAt: timestamp('created_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
  deletedAt: timestamp('deleted_at'),                         // 软删时间戳
});
```

**关键设计**：

1. **状态枚举**：用 `pgEnum` 而不是 `text`，类型安全（修 ingest_jobs 的一致性 #）——见 Step 2.5
2. **`fileHash` vs `contentHash`**：
   - `fileHash` 是文件二进制的 hash，**上传时**计算，用于秒级查重
   - `contentHash` 是解析后纯文本的 hash，**入库时**计算，用于"同一文件内容改了没"
3. **`embeddingModel`**：记录每个文档用的 embedding 模型。将来换模型时（如 BGE-M3 → BGE-M5），可以批量找出旧模型 embedding 的文档重新生成
4. **`deletedAt` 软删**：document 是软删（不丢历史），chunks 是硬删（占空间大）—— Part 8 会处理这个一致性

## Step 2.4: src/db/schema/chunk.ts（修复 #10：datasetId 加外键）

**目标**：Parent-Child 分块表，含向量列。

写 `src/db/schema/chunk.ts`：

```typescript
import {
  pgTable, uuid, text, integer, timestamp, vector, pgEnum, index, uniqueIndex,
} from 'drizzle-orm/pg-core';
import { documents } from './document';
import { datasets } from './dataset';

export const embeddingStatusEnum = pgEnum('embedding_status', [
  'pending',
  'done',
  'failed',
]);

export const chunks = pgTable('chunks', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id')
    .references(() => documents.id, { onDelete: 'cascade' })
    .notNull(),

  // 修复 #10：datasetId 加外键
  // 旧版只有 .notNull()，可以指向不存在的 dataset
  datasetId: uuid('dataset_id')
    .references(() => datasets.id, { onDelete: 'cascade' })
    .notNull(),

  // parent_id：自引用。parentId = null 表示这是 parent chunk
  // 不能用 references() 自引用（Drizzle 限制），通过 migration SQL 补外键
  parentId: uuid('parent_id'),

  // 修复 #12：chunkIndex 改为 (parentChunkIndex, childIndexWithinParent) 联合语义
  // - parent chunk: parentChunkIndex = 自己序号, childIndexWithinParent = null
  // - child chunk: parentChunkIndex = 父 parent 的序号, childIndexWithinParent = 在父内的序号
  parentChunkIndex: integer('parent_chunk_index').notNull(),
  childIndexWithinParent: integer('child_index_within_parent'),  // null = 自身是 parent

  content: text('content').notNull(),
  contentHash: text('content_hash').notNull(),
  tokenCount: integer('token_count').notNull(),

  // embedding 仅 child 有（parent 不参与向量检索）
  embedding: vector('embedding', { dimensions: 1024 }),
  embeddingStatus: embeddingStatusEnum('embedding_status').notNull().default('pending'),

  scope: text('scope').notNull().default('platform'),

  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => [
  // 向量检索：HNSW + 余弦距离
  index('chunk_embedding_idx').using('hnsw', table.embedding.op('vector_cosine_ops')),

  // 按 documentId 查所有 chunk（删文档、查文档分块用）
  index('chunk_document_idx').on(table.documentId),

  // 按 parentId 查所有 child（parent-child 还原用）
  index('chunk_parent_idx').on(table.parentId),

  // 按 datasetId 查（检索时过滤用，绝大多数查询的 WHERE 条件）
  index('chunk_dataset_idx').on(table.datasetId),

  // 修复 #12：联合唯一约束，保证同一 document 内 (parent_chunk_index, child_index_within_parent) 不重复
  uniqueIndex('chunk_doc_parent_child_uniq').on(
    table.documentId,
    table.parentChunkIndex,
    table.childIndexWithinParent,
  ),
]);
```

**修复 #10 的解释**：

旧版：
```typescript
datasetId: uuid('dataset_id').notNull(),   // ❌ 可以指向不存在的 dataset
```

新版：
```typescript
datasetId: uuid('dataset_id')
  .references(() => datasets.id, { onDelete: 'cascade' })   // ✅ 外键约束
  .notNull(),
```

加上 `onDelete: 'cascade'`：删 dataset 时自动删其下所有 chunks（document 表也有同样的 cascade，删 dataset 会级联清干净）。

**修复 #12 的解释**：

旧版：
```typescript
chunkIndex: integer('chunk_index').notNull(),   // 全局递增，丢失 child 在 parent 内的序号
```

新版用两个字段：
- `parentChunkIndex`：parent 的全局序号（parent 自己和它的所有 child 共享这个值）
- `childIndexWithinParent`：child 在 parent 内的序号（parent 自己为 null）

这样：
- 查 parent 下所有 child：`WHERE parent_chunk_index = X AND child_index_within_parent IS NOT NULL`
- 查所有 parent：`WHERE child_index_within_parent IS NULL`

**为什么 chunkIndex 字段不加外键自引用**：Drizzle 暂不支持同一表内的 `references()` 自引用。Part 2.7 的 migration SQL 会补这个外键。

**为什么 tsvector 不在这里加**：Drizzle 的 schema 描述能力有限，tsvector `GENERATED ALWAYS AS` 列通过 migration SQL 添加（Step 2.7）。

更新 `src/db/schema/index.ts`：

```typescript

export { documents, documentStatusEnum } from './document';
export { chunks, embeddingStatusEnum } from './chunk';
```

## Step 2.7: tsvector 列 + 自引用外键 migration

**目标**：补 Drizzle 写不了的 SQL —— tsvector 生成列、GIN 索引、parent_id 自引用外键。

写 `src/db/migrations/0001_add_tsvector_and_fkeys.sql`：

```sql
-- ===== 1. tsvector 生成列（稀疏检索核心） =====
-- 用 'simple' 分词器：中文按字符切，英文按空格切
-- 中文真正分词需要 pg_jieba 或 zhparser 扩展（Phase 2）
ALTER TABLE chunks ADD COLUMN IF NOT EXISTS tsv tsvector
  GENERATED ALWAYS AS (to_tsvector('simple', coalesce(content, ''))) STORED;

CREATE INDEX IF NOT EXISTS chunk_tsv_idx ON chunks USING GIN(tsv);


-- ===== 2. parent_id 自引用外键（Drizzle 写不了自引用，这里补） =====
-- onDelete CASCADE：删 parent 时自动删其下所有 child
ALTER TABLE chunks
  ADD CONSTRAINT chunks_parent_id_fkey
  FOREIGN KEY (parent_id) REFERENCES chunks(id)
  ON DELETE CASCADE;


-- ===== 3. tsvector 中文支持占位（Phase 2 启用 pg_jieba 时改这里） =====
-- 当前注释掉，避免没装扩展时报错
-- ALTER TABLE chunks ADD COLUMN tsv_jieba tsvector
--   GENERATED ALWAYS AS (to_tsvector('jieba', coalesce(content, ''))) STORED;
-- CREATE INDEX chunk_tsv_jieba_idx ON chunks USING GIN(tsv_jieba);
```

**为什么 tsvector 用 STORED 生成列**：
- `GENERATED ALWAYS AS (...) STORED` 让 PostgreSQL 在 INSERT/UPDATE 时自动计算 tsvector 列
- 应用代码不需要管，永远不会出现 content 改了但 tsv 没更新的情况
- 占额外磁盘，但查询快（GIN 索引直接走 tsv 列）

**为什么中文用 `simple` 分词器**：
- PostgreSQL 默认的 `simple` 分词器按非字母数字字符切分，中文按字符切（每个汉字一个 token）
- 准确度不如真正的中文分词（pg_jieba），但 Phase 1 够用，且不需要装扩展
- Phase 2 换 pg_jieba 时改这一行 SQL + 重建索引即可

### Step 3A.4: ParserRegistry

**目标**：按文件扩展名路由到对应 Parser。Phase 2 加 PDF Parser 时不用改入库逻辑。

写 `src/parser/registry.ts`：

```typescript
import type { BaseParser } from './base-parser';

export class ParserRegistry {
  private parsers = new Map<string, BaseParser>();

  /** 注册一个 Parser，会自动注册它支持的所有扩展名 */
  register(parser: BaseParser): void {
    for (const type of parser.supportedTypes) {
      if (this.parsers.has(type)) {
        throw new Error(`Parser for type '${type}' already registered`);
      }
      this.parsers.set(type, parser);
    }
  }

  /** 按扩展名取 Parser，找不到抛错 */
  getParser(fileType: string): BaseParser {
    const parser = this.parsers.get(fileType.toLowerCase());
    if (!parser) {
      throw new Error(
        `No parser for type '${fileType}'. Supported: ${[...this.parsers.keys()].join(', ')}`,
      );
    }
    return parser;
  }

  /** 列出所有支持的扩展名（用于 API 响应） */
  supportedTypes(): string[] {
    return [...this.parsers.keys()];
  }
}
```

**改进**：
- 注册时如果重复，旧版静默覆盖，新版抛错（避免误注册两个 Parser 处理同一扩展名）
- `getParser` 找不到时返回明确错误信息（包含所有支持的类型）
- `fileType.toLowerCase()` 防御性处理（用户可能传 `.TXT`）

### Step 3B.2: 分隔符层级

**目标**：定义递归切分的优先级表。

写 `src/splitter/separators.ts`：

```typescript
/**
 * 5 级分隔符，从粗到细。
 * 切分时优先用 L0（段落），文本太长降级到 L1（句子），以此类推。
 * 绝不切半句话：找不到句子分隔符就降级到词，最后才硬切字符。
 */
export const SEPARATOR_LEVELS: string[][] = [
  // L0: 段落
  ['\n\n', '\r\n\r\n'],

  // L1: 句子（中英文标点）
  ['。', '！', '？', '；', '\n', '. ', '! ', '? ', '; '],

  // L2: 子句
  ['，', '、', '：', ', ', ': '],

  // L3: 词
  [' '],

  // L4: 字符（兜底，空字符串表示逐字符切）
  [''],
];
```

**为什么这样设计**：
- L0 段落优先：保持段落完整，向量编码语义最完整
- L1 句子：段落太大时降级到句子，绝不切半句
- L2 子句：句子还太大时降级（中文逗号、顿号）
- L3 词：英文按空格切
- L4 字符：最后兜底，按单字符硬切

**中文设计的考虑**：
- 中文用全角标点（`。！？；`）
- 英文用半角 + 空格（`. ` 而不是 `.`），避免切到缩写（如 `e.g.` `Mr.`）
- 中文顿号 `、` 单独一级，因为它切的是并列短语，不是完整子句

### Step 3B.3: token 计数

**目标**：用启发式估算 token 数，避免装 tiktoken。

写 `src/splitter/token-counter.ts`：

```typescript
/**
 * 中英文混合 token 估算。
 * 启发式：中文字符约 1.5 字/token，英文字符约 4 字/token。
 * 这是经验值，和真实 BPE 分词器（如 tiktoken）误差 ±20%。
 *
 * 为什么不直接用 tiktoken：
 * - tiktoken 是 wasm 包，Bun 兼容性偶尔有问题
 * - 启发式够用：切分只需要"够准"的长度估计，差 20% 不影响 chunk 质量
 * - 0 依赖、纯函数、可独立测试
 *
 * 如果以后需要精确 token 数（如算 LLM 上下文成本），可以替换为 tiktoken
 */
export function countTokens(text: string): number {
  if (text.length === 0) return 0;

  // 统计中文字符数（CJK 统一汉字 + 扩展 A）
  let cn = 0;
  for (let i = 0; i < text.length; i++) {
    const c = text.charCodeAt(i);
    // CJK Unified Ideographs (常用汉字)
    if (c >= 0x4e00 && c <= 0x9fff) {
      cn++;
      continue;
    }
    // CJK Extension A (生僻字)
    if (c >= 0x3400 && c <= 0x4dbf) {
      cn++;
    }
  }
  const en = text.length - cn;

  // 中文约 1.5 字/token，英文约 4 字/token
  return Math.ceil(cn / 1.5 + en / 4);
}
```

### Step 3B.4: RecursiveSplitter —— splitBySeparators

**目标**：实现最底层的"按给定分隔符切文本"，作为后续递归的基础。

写 `src/splitter/recursive-splitter.ts`（先只加这一个方法）：

```typescript
import type { SplitConfig } from './types';

export class RecursiveSplitter {
  constructor(protected config: SplitConfig) {}

  /**
   * 按给定的一组分隔符切文本。
   * - 优先用第一个能匹配的分隔符
   * - 切出来的片段保留分隔符（如 "你好。" 而不是 "你好"）
   * - 空片段被过滤
   *
   * 例如 separators=['。', '！'], text='你好。世界！再见'
   *   → ['你好。', '世界！', '再见']
   */
  protected splitBySeparators(text: string, seps: string[]): string[] {
    // 找到第一个能匹配的分隔符
    const sep = seps.find(s => s !== '' && text.includes(s));
    if (!sep) {
      // 没有匹配的分隔符，整段返回（可能是 L4 字符级，由 hardSplit 处理）
      return [text];
    }

    const parts: string[] = [];
    let remaining = text;

    while (remaining.length > 0) {
      const idx = remaining.indexOf(sep);
      if (idx === -1) {
        // 没有更多分隔符了
        parts.push(remaining);
        break;
      }

      // 切到分隔符末尾（保留分隔符在片段里）
      const end = idx + sep.length;
      parts.push(remaining.slice(0, end));
      remaining = remaining.slice(end);
    }

    return parts.filter(p => p.length > 0);
  }
}
```

**关键设计**：
- `protected` 方法：外部不直接调，由 `splitRaw` 编排
- **保留分隔符在片段里**：`slice(0, end)` 而不是 `slice(0, idx)`，这样切完后片段自带句号，读起来完整
- 空字符串 `''` 作为分隔符时不在这里处理（`s !== ''` 过滤掉），交给 hardSplit

### Step 3B.5: RecursiveSplitter —— recursiveSplit

**目标**：实现递归降级 —— 文本太长就降级到更细的分隔符。

在 `src/splitter/recursive-splitter.ts` 加方法：

```typescript
  /**
   * 递归切分。
   * - 如果文本长度 <= maxChunkSize，直接返回
   * - 否则用当前 level 的分隔符切，每段如果还太长，level+1 递归
   * - 如果 level 已是最深（L4），调 hardSplit 兜底
   */
  private recursiveSplit(text: string, level: number): string[] {
    const { maxChunkSize, separators, lengthFunction: len } = this.config;

    // 终止条件 1：已经够小
    if (len(text) <= maxChunkSize) {
      return [text];
    }

    // 终止条件 2：已经是最深 level，硬切
    if (level >= separators.length) {
      return this.hardSplit(text);
    }

    // 用当前 level 的分隔符切
    const parts = this.splitBySeparators(text, separators[level]!);

    // 对每个片段：如果够小直接收下，太大递归降级
    const result: string[] = [];
    for (const part of parts) {
      if (len(part) <= maxChunkSize) {
        result.push(part);
      } else {
        result.push(...this.recursiveSplit(part, level + 1));
      }
    }
    return result;
  }
```

**为什么是 `private`**：外部只调 `splitRaw`，递归是实现细节。

**为什么用 `level` 而不是 `separators[level]` 直接传**：递归时 level+1 优雅，且方便 log 调试。

### Step 3B.6: RecursiveSplitter —— mergeUnits

**目标**：递归切完后会有很多小碎片（一个句号一段），需要合并到接近 maxChunkSize。

继续在 `src/splitter/recursive-splitter.ts` 加方法：

```typescript
  /**
   * 把小片段合并成接近 maxChunkSize 的大片段。
   * - 累加片段，超过 maxChunkSize 就提交当前累积的，开始新累积
   * - 如果当前累积不够 minChunkSize，但加上下一个会超 maxChunkSize，
   *   就把下一个并进来一起提交（避免产生过小的 chunk）
   */
  private mergeUnits(units: string[]): string[] {
    const { maxChunkSize, minChunkSize, lengthFunction: len } = this.config;
    const chunks: string[] = [];
    let current = '';

    for (const unit of units) {
      const candidate = current + unit;

      if (len(candidate) <= maxChunkSize) {
        // 还能装下，继续累积
        current = candidate;
      } else {
        // 装不下了
        if (current && len(current) >= minChunkSize) {
          // 当前累积够大，提交，开始新累积
          chunks.push(current);
          current = unit;
        } else {
          // 当前累积太小，硬把这个 unit 并进来一起提交（即使超一点）
          chunks.push(current + unit);
          current = '';
        }
      }
    }
    if (current) chunks.push(current);

    return chunks;
  }
```

**关键点**：
- `minChunkSize` 防止产生过小 chunk（如 5 个字的 chunk 单独存在）
- `len(current + unit)` 超 maxChunkSize 但 `len(current) < minChunkSize` 时，**宁可超出一点点也要保证不产生过小 chunk**
- 这是经验权衡：略大 chunk（vector 编码稍模糊）比过小 chunk（信息太少）好

### Step 3B.7: RecursiveSplitter —— addOverlap（修复 #16）

**目标**：让相邻 chunk 有重叠，保证上下文连续。

**修复 #16**：旧版的 snap 阈值 `idx - startChar < prev.length * 0.1` 用整段 prev.length 作为基准，但语义应该是"不超过 overlap 区间的 10%"。

继续在 `src/splitter/recursive-splitter.ts` 加方法：

```typescript
  /**
   * 给每个 chunk（除第一个）前面加上一个 chunk 的尾部作为重叠。
   * - 重叠长度约等于 overlapSize
   * - 重叠起点 snap 到最近的句子边界（避免从半句话开始）
   *
   * 修复 #16：snap 阈值用 overlapSize 作为基准，而不是 prev.length
   */
  private addOverlap(chunks: string[]): string[] {
    const { overlapSize, lengthFunction: len, separators } = this.config;
    if (overlapSize <= 0 || chunks.length <= 1) return chunks;

    const result = [chunks[0]!];   // 第一个不加重叠

    for (let i = 1; i < chunks.length; i++) {
      const prev = chunks[i - 1]!;

      // 如果上一个 chunk 比重叠还短，直接整个拼上来
      if (len(prev) <= overlapSize) {
        result.push(prev + chunks[i]);
        continue;
      }

      // 计算重叠起点：从 prev 末尾往回数 overlapSize 长度
      const ratio = overlapSize / len(prev);
      let startChar = Math.floor(prev.length * (1 - ratio));

      // 修复 #16：snap 阈值是 overlap 区间的 10%，不是整段的 10%
      // 例如 overlapSize=50, prev=1200 token，snap 容忍度是 5 字符（overlap 的 10%）
      // 旧版用的是 120 字符（整段的 10%），snap 会越过大量内容
      const snapTolerance = Math.max(2, Math.floor(overlapSize * 0.1));
      const sentenceSeps = separators[1] ?? [];   // L1 句子分隔符

      for (const sep of sentenceSeps) {
        const idx = prev.indexOf(sep, startChar);
        // snap 点在 [startChar - snapTolerance, startChar + snapTolerance] 内
        if (idx >= 0 && Math.abs(idx - startChar) <= snapTolerance) {
          startChar = idx + sep.length;
          break;
        }
      }

      result.push(prev.slice(startChar) + chunks[i]);
    }

    return result;
  }
```

**修复说明（#16）**：

| 旧版 | 新版 |
|---|---|
| `idx - startChar < prev.length * 0.1` | `Math.abs(idx - startChar) <= overlapSize * 0.1` |
| 用整段长度作基准（prev 可能 1200 token） | 用 overlap 区间作基准（约 50 token） |
| snap 容忍 ±120 字符 | snap 容忍 ±5 字符 |

旧版效果：snap 点可能跑到 prev 中间，导致 overlap 起点严重偏离预期。新版严格限制 snap 偏移在 overlap 区间附近。

另一个改进：`Math.abs` —— 旧版只检查正向偏移（idx > startChar），新版也允许 idx < startChar（snap 往前找）。

### Step 3B.8: RecursiveSplitter —— hardSplit

**目标**：兜底硬切（没有任何分隔符时）。

继续在 `src/splitter/recursive-splitter.ts` 加方法：

```typescript
  /**
   * 兜底硬切：按字符长度均分。
   * 只在 recursiveSplit 走到 L4（字符级分隔符）时调用。
   * 注意：可能会切到字符中间（如多字节 emoji），但 L4 已经是最后手段。
   */
  private hardSplit(text: string): string[] {
    const { maxChunkSize, lengthFunction: len } = this.config;
    const totalLen = len(text);
    if (totalLen <= maxChunkSize) return [text];

    // 按比例换算字符步长（lengthFunction 可能是 token，但字符串是字符）
    const ratio = maxChunkSize / Math.max(totalLen, 1);
    const step = Math.max(1, Math.floor(text.length * ratio));

    const parts: string[] = [];
    for (let i = 0; i < text.length; i += step) {
      parts.push(text.slice(i, i + step));
    }
    return parts;
  }
```

**关键点**：
- 只在没有任何分隔符时调用（极少见）
- 按 lengthFunction 的比例换算字符步长，保证切出来的 chunk 不超 maxChunkSize
- 可能切到多字节字符中间（如 emoji），接受这个不完美

### Step 3B.9: RecursiveSplitter —— splitRaw（组装完整流程）

**目标**：把上面的方法组装成完整的切分流程。

继续在 `src/splitter/recursive-splitter.ts` 加方法：

```typescript
  /**
   * 完整流程：递归切 → 合并 → 加重叠。
   * 这是外部调用的唯一入口。
   */
  splitRaw(text: string): string[] {
    // 统一换行符
    const cleaned = text.replace(/\r\n/g, '\n');

    // Step 1: 递归切分
    const units = this.recursiveSplit(cleaned, 0);

    // Step 2: 合并小碎片
    const merged = this.mergeUnits(units);

    // Step 3: 加重叠
    return this.addOverlap(merged);
  }
```

### Step 3B.10: ParentChildSplitter —— 切 parent（最小版本）

**目标**：实现 Parent-Child 切分。先做 parent 部分，能跑通。

写 `src/splitter/parent-child-splitter.ts`（先只切 parent）：

```typescript
import type { SplitConfig, ChunkUnit } from './types';
import { RecursiveSplitter } from './recursive-splitter';
import { countTokens } from './token-counter';
import { sha256 } from '../utils/hash';

export class ParentChildSplitter {
  constructor(
    private parentConfig: SplitConfig,
    private childConfig: SplitConfig,
  ) {}

  split(text: string, metadata: Record<string, unknown> = {}): ChunkUnit[] {
    const cleaned = text.replace(/\r\n/g, '\n');
    const units: ChunkUnit[] = [];

    // ===== Step 1: 切 parent =====
    const parentSplitter = new RecursiveSplitter(this.parentConfig);
    const parentTexts = parentSplitter.splitRaw(cleaned);

    // 记录每个 parent 在原文中的起始 offset（用于调试 + 未来 highlight）
    let searchFrom = 0;
    const parentStartOffsets: number[] = [];
    for (const parentText of parentTexts) {
      // 用 parent 文本前 50 字符做指纹，在原文中定位
      const fingerprint = parentText.slice(0, 50);
      const foundAt = cleaned.indexOf(fingerprint, searchFrom);
      const startOffset = foundAt >= 0 ? foundAt : searchFrom;
      parentStartOffsets.push(startOffset);
      searchFrom = startOffset + parentText.length;
    }

    // ===== Step 2: 生成 parent 的 ChunkUnit =====
    // 暂时不切 child，下个 Step 加
    for (let pi = 0; pi < parentTexts.length; pi++) {
      const parentText = parentTexts[pi]!;
      const startOffset = parentStartOffsets[pi]!;

      units.push({
        text: parentText,
        tokenCount: countTokens(parentText),
        isParent: true,
        parentChunkIndex: pi,
        childIndexWithinParent: null,                 // parent 自己
        startOffset,
        endOffset: startOffset + parentText.length,
        contentHash: sha256(parentText),
        metadata: { ...metadata, isParent: true, parentChunkIndex: pi },
      });
    }

    return units;
  }
}
```

### Step 3B.11: ParentChildSplitter —— 切 child + 关联（修复 #12）

**目标**：对每个 parent 再切 child，建立正确的归属关系。

```typescript
    for (const child of children) {
      expect(child.parentChunkIndex).toBeDefined();
      expect(child.childIndexWithinParent).not.toBeNull();

      // 对应的 parent 必须存在
      const parent = units.find(
        u => u.isParent && u.parentChunkIndex === child.parentChunkIndex
      );
      expect(parent).toBeDefined();
    }

    // 同一 parent 下，childIndexWithinParent 应该从 0 开始且不重复
    const byParent = new Map<number, number[]>();
    for (const c of children) {
      const arr = byParent.get(c.parentChunkIndex) ?? [];
      arr.push(c.childIndexWithinParent!);
      byParent.set(c.parentChunkIndex, arr);
    }
    for (const [, arr] of byParent) {
      const sorted = [...arr].sort((a, b) => a - b);
      expect(sorted).toEqual([...Array(arr.length).keys()]);
    }
  });

  it('所有 unit 有 contentHash', () => {
    const splitter = new ParentChildSplitter(parentConfig, childConfig);
    const units = splitter.split('测试文本。另一个句子。');
```

## 3C. Embedding 服务

### 3C 目标

把文本转成 1024 维向量。支持单条查询（走缓存）和批量入库（不走缓存）。

### Step 3C.1: EmbeddingService 完整实现

**目标**：一个类同时支持单条查询和批量入库。

写 `src/embedding/embedding-service.ts`：

```typescript
import { config } from '../config';
import { TTLCache } from '../cache/ttl-cache';
import { logger } from '../utils/logger';

/**
 * Embedding 服务 —— 把文本转成向量。
 * - 查询用 embedQuery（走缓存，相同 query 不重复调 API）
 * - 入库用 embedBatch（不走缓存，分批调用）
 */
export class EmbeddingService {
  // 修复 #11：缓存加上限，避免长尾查询导致内存无限增长
  private queryCache = new TTLCache<string, number[]>(
    600_000,                              // TTL 10 分钟
    config.embeddingCacheMax,             // LRU 上限
  );

  /** 单条 embed（查询用，走缓存） */
  async embedQuery(text: string): Promise<number[]> {
    const cached = this.queryCache.get(text);
    if (cached) return cached;

    const vectors = await this.callApi([text]);
    const vec = vectors[0]!;
    this.queryCache.set(text, vec);
    return vec;
  }

  /** 批量 embed（入库用，不走缓存） */
  async embedBatch(texts: string[]): Promise<number[][]> {
    if (texts.length === 0) return [];

    const results: number[][] = [];
    const batchSize = config.embeddingBatchSize;

    for (let i = 0; i < texts.length; i += batchSize) {
      const batch = texts.slice(i, i + batchSize);
      const vectors = await this.callApi(batch);
      results.push(...vectors);
    }

    return results;
  }

  /** 实际调 OpenAI 兼容的 /embeddings 接口 */
  private async callApi(texts: string[]): Promise<number[][]> {
    const url = `${config.embeddingApiUrl}/embeddings`;
    const res = await fetch(url, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${config.embeddingApiKey}`,
      },
      body: JSON.stringify({
        model: config.embeddingModelId,
        input: texts,
      }),
    });

    if (!res.ok) {
      const errorBody = await res.text();
      logger.error('Embedding API error', { status: res.status, body: errorBody });
      throw new Error(`Embedding API error: ${res.status} ${errorBody}`);
    }

    const json = await res.json() as {
      data: Array<{ embedding: number[] }>;
    };

    if (!json.data || json.data.length !== texts.length) {
      throw new Error(
        `Embedding API returned ${json.data?.length} vectors, expected ${texts.length}`,
      );
    }

    return json.data.map(d => d.embedding);
  }
}
```

**关键设计**：

1. **两个方法职责清晰**：
   - `embedQuery`：单条 + 缓存（用户 query 重复率高，10 分钟内同 query 复用）
   - `embedBatch`：批量 + 不缓存（入库的文本基本不重复，缓存浪费内存）
2. **批量分片**：按 `EMBEDDING_BATCH_SIZE`（默认 16）切，避免单请求太大
3. **错误信息完整**：把 API 返回的 body 写进错误，方便排查
4. **长度校验**：API 返回的向量数必须等于请求数，否则抛错（防止上游 bug 静默错位）

**为什么用 fetch 而不是 openai SDK**：
- `openai` npm 包又大又依赖 Node API，Bun 偶有兼容性问题
- OpenAI 兼容 API 就是标准 HTTP + JSON，fetch 完全够用
- 代码透明，没有黑盒

```typescript
import { config } from '../config';
import { logger } from '../utils/logger';

/** OpenAI 标准 message 类型 */
export interface Message {
  role: 'system' | 'user' | 'assistant' | 'tool';
  content: string;
  tool_calls?: ToolCall[];       // assistant message 才有
  tool_call_id?: string;         // tool message 才有

  id: string;
  type: 'function';
  function: { name: string; arguments: string };   // arguments 是 JSON 字符串
}

/** 暴露给 LLM 的 function 定义 */
export interface FunctionDefinition {
  type: 'function';
  function: {
    name: string;
    description: string;
    parameters: Record<string, unknown>;     // JSON Schema
  };
}

export interface ChatResponse {
  content: string | null;
  tool_calls?: ToolCall[];
}

export interface ChatOptions {
  messages: Message[];
  tools?: FunctionDefinition[];
  tool_choice?: 'auto' | 'none' | 'required';

  /** 最大输出 token 数（不传用模型默认） */
  maxTokens?: number;

export class LLMService {
  async chat(opts: ChatOptions): Promise<ChatResponse> {
    const url = `${config.llmApiUrl}/chat/completions`;
    const body: Record<string, unknown> = {
      model: config.llmModelId,
      messages: opts.messages,
      temperature: opts.temperature ?? 0.2,
    };

    // 只在有 tools 时加这些字段（某些 API 对 undefined 敏感）
    if (opts.tools && opts.tools.length > 0) {
      body.tools = opts.tools;
      body.tool_choice = opts.tool_choice ?? 'auto';
    }
    if (opts.maxTokens) {
      body.max_tokens = opts.maxTokens;
    }

    const res = await fetch(url, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${config.llmApiKey}`,
      },
      body: JSON.stringify(body),
    });

    if (!res.ok) {
      const errorBody = await res.text();
      logger.error('LLM API error', { status: res.status, body: errorBody });
      throw new Error(`LLM API error: ${res.status} ${errorBody}`);
    }

    const json = await res.json() as {
      choices: Array<{
        message: {
          content: string | null;
          tool_calls?: ToolCall[];
        };
      }>;
    };

    if (!json.choices?.[0]) {
      throw new Error('LLM API returned no choices');
    }

    const msg = json.choices[0].message;
    return { content: msg.content, tool_calls: msg.tool_calls };
  }

  /** 便捷方法：纯文本生成（不带 tools） */
  async generate(
    prompt: string,
    opts?: { temperature?: number; maxTokens?: number },
  ): Promise<string> {
    const res = await this.chat({
      messages: [{ role: 'user', content: prompt }],
      temperature: opts?.temperature,
      maxTokens: opts?.maxTokens,
    });
    return res.content ?? '';
  }
}
```

### Step 3E.1: RRF 融合

**目标**：纯函数，把两个排序列表融合成一个。

写 `src/retrieve/rrf.ts`：

```typescript
/**
 * Reciprocal Rank Fusion。
 * 公式：score(d) = Σ 1/(k + rank(d))
 *
 * 特点：
 * - 尺度无关：Dense 分数 0~1，Sparse 分数 0~100，融合时不被大数值主导
 * - 只看排名，不需要归一化，不需要调参
 * - k 默认 60（业界经验值）
 *
 * @param listA 第一个排序列表，[id, score] 数组
 * @param listB 第二个排序列表
 * @param k 平滑常数，默认 60
 * @returns 融合后的 [id, rrfScore] 数组，按 rrfScore 降序
 */
export function rrfFusion(
  listA: ReadonlyArray<readonly [string, number]>,
  listB: ReadonlyArray<readonly [string, number]>,
  k: number = 60,
): Array<[string, number]> {
  const scores = new Map<string, number>();

  // 按 score 降序排（虽然 caller 通常已排，但 contract 不强制）
  // 然后用降序后的 index 作为 rank
  const rankedA = [...listA].sort((a, b) => b[1] - a[1]);
  const rankedB = [...listB].sort((a, b) => b[1] - a[1]);

  for (let i = 0; i < rankedA.length; i++) {
    const id = rankedA[i]![0];
    scores.set(id, (scores.get(id) ?? 0) + 1 / (k + i + 1));
  }
  for (let i = 0; i < rankedB.length; i++) {
    const id = rankedB[i]![0];
    scores.set(id, (scores.get(id) ?? 0) + 1 / (k + i + 1));
  }

  return [...scores.entries()].sort((a, b) => b[1] - a[1]);
}
```

**改进**：
- 输入用 `ReadonlyArray<readonly [string, number]>`，表达"不修改入参"
- 显式 sort（contract 改为"按 score 排"而不是"按 caller 给的顺序"）
- `k` 有默认值，但允许 caller 覆盖

**验证**：写 `tests/rrf.test.ts`：

### Step 3E.2: Dense 检索

**目标**：用 pgvector 做向量相似度检索。

写 `src/retrieve/dense.ts`：

```typescript
import { db } from '../db/client';
import { chunks } from '../db/schema';
import { cosineDistance, sql, desc, gt, and, eq, isNotNull } from 'drizzle-orm';

export interface DenseHit {
  chunkId: string;
  content: string;
  parentId: string | null;
  documentId: string;
  score: number;          // 相似度（0~1，越大越相似）
}

/**
 * 向量检索。
 * @param queryVec 查询向量（1024 维）
 * @param datasetId 数据集 ID（必须，跨数据集检索无意义）
 * @param limit 返回数量
 */
export async function denseSearch(
  queryVec: number[],
  datasetId: string,
  limit: number,
): Promise<DenseHit[]> {
  // cosineDistance 返回的是距离（0~2），1 - distance = 相似度
  const similarity = sql<number>`1 - (${cosineDistance(chunks.embedding, queryVec)})`;

  const rows = await db
    .select({
      chunkId: chunks.id,
      content: chunks.content,
      parentId: chunks.parentId,
      documentId: chunks.documentId,
      score: similarity,
    })
    .from(chunks)
    .where(and(
      eq(chunks.datasetId, datasetId),
      eq(chunks.embeddingStatus, 'done'),
      isNotNull(chunks.embedding),
      // 过滤掉 parent chunk（parent 没有 embedding，但保险起见显式过滤）
      // 实际上 parent 的 embeddingStatus 会是 'pending'，上面那行已经过滤了
      gt(similarity, 0.2),   // 相似度阈值，过低无意义
    ))
    .orderBy(desc(similarity))
    .limit(limit);

  return rows.map(r => ({
    chunkId: r.chunkId,
    content: r.content,
    parentId: r.parentId,
    documentId: r.documentId,
    score: Number(r.score),
  }));
}
```

**关键设计**：

1. **`cosineDistance` → similarity**：pgvector 的 `<=>` 操作符返回余弦距离（0=完全相同，2=完全相反）。`1 - distance` 转成相似度，越大越相似
2. **`gt(similarity, 0.2)` 阈值**：相似度低于 0.2 的结果基本没用，提前过滤省 Reranker 成本
3. **`embeddingStatus = 'done'`**：跳过还没 embed 的 chunk（并发入库时不漏不重）
4. **强制 `datasetId`**：跨数据集检索没意义，要求 caller 必传

### Step 3E.3: Sparse 检索

**目标**：用 tsvector 做关键词检索。

写 `src/retrieve/sparse.ts`：

```typescript
import { db } from '../db/client';
import { sql, eq, and } from 'drizzle-orm';
import { chunks } from '../db/schema';

export interface SparseHit {
  chunkId: string;
  score: number;
}

/**
 * 关键词检索（PostgreSQL tsvector + GIN 索引）。
 * 用 plainto_tsquery 处理用户输入（避免特殊字符误解析为操作符）。
 */
export async function sparseSearch(
  query: string,
  datasetId: string,
  limit: number,
): Promise<SparseHit[]> {
  // 用原始 SQL，因为 Drizzle 对 tsvector 的抽象不完整
  const result = await db.execute<{
    id: string;
    score: number;
  }>(sql`
    SELECT id::text, ts_rank_cd(tsv, plainto_tsquery('simple', ${query})) AS score
    FROM chunks
    WHERE dataset_id = ${datasetId}::uuid
      AND tsv @@ plainto_tsquery('simple', ${query})
      AND embedding_status = 'done'
    ORDER BY score DESC
    LIMIT ${limit}
  `);

  return result.rows
    .map(r => ({
      chunkId: r.id,
      score: Number(r.score),
    }))
    .filter(r => r.score > 0);   // 过滤掉 0 分（理论上 WHERE 已经过滤，双保险）
}
```

**关键设计**：

1. **`plainto_tsquery` 而不是 `to_tsquery`**：后者对特殊字符敏感（`&|!()` 是操作符），用户输入 `"Docker & K8s"` 会报错。前者把整个字符串当作普通词组
2. **`ts_rank_cd`**：cover density ranking，比 `ts_rank` 更准（考虑词频 + 位置覆盖度）
3. **`@@` 操作符**：tsvector 匹配 tsquery，走 GIN 索引
4. **`embedding_status = 'done'`**：和 dense 保持一致，跳过未 embed 的 chunk

**中文检索的局限**（已在选型文档讨论）：
- `simple` 分词器把中文按字符切（"Docker 容器" → `'docker' '容' '器'`）
- 中文单字 token 噪音大（"的"、"是" 都被检索）
- Phase 2 换 pg_jieba 后改 SQL 即可

### Step 3E.4: Reranker

**目标**：对粗排结果精排。提供 API 实现和 PassThrough 兜底（API 不可用时降级）。

写 `src/retrieve/reranker.ts`：

```typescript
import { config } from '../config';
import { logger } from '../utils/logger';

export interface RerankCandidate {
  chunkId: string;
  parentId: string | null;
  documentId: string;
  text: string;        // 候选文本（精排时被截断）
  score: number;       // 粗排分数
}

export interface Reranker {
  /**
   * 对候选重新排序。
   * @param query 用户原始查询
   * @param candidates 粗排候选
   * @param topK 返回前 K 个
   */
  rank(
    query: string,
    candidates: RerankCandidate[],
    topK?: number,
  ): Promise<RerankCandidate[]>;
}

/**
 * API Reranker —— 调用 qwen3-rerank 等 Cross-Encoder 模型。
 * Cross-Encoder 比 Bi-Encoder（向量检索）精度高，但慢（每个候选都要过一次模型）。
 */
export class ApiReranker implements Reranker {
  /** 单文档最大字符数（防止 API 拒绝或超时） */
  private readonly maxDocChars = 8000;

  async rank(
    query: string,
    candidates: RerankCandidate[],
    topK?: number,
  ): Promise<RerankCandidate[]> {
    if (candidates.length <= 1) return candidates;
    const limit = topK ?? candidates.length;

    try {
      const documents = candidates.map(c => c.text.slice(0, this.maxDocChars));

      const res = await fetch(config.rerankApiUrl, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': `Bearer ${config.rerankApiKey}`,
        },
        body: JSON.stringify({
          model: config.rerankModelId,
          query,
          documents,
          top_n: limit,
        }),
      });

      if (!res.ok) {
        const body = await res.text();
        throw new Error(`Rerank API ${res.status}: ${body}`);
      }

      const json = await res.json() as {
        results: Array<{ index: number; relevance_score: number }>;
      };

      // 按返回的 index 映射回原 candidate
      return json.results.map(r => ({
        ...candidates[r.index]!,
        score: r.relevance_score,
      }));
    } catch (err) {
      // 降级：保留原顺序，截取 topK
      logger.warn('Rerank failed, falling back to original order', err);
      return candidates.slice(0, limit);
    }
  }
}

/**
 * PassThrough Reranker —— 不排序，直接截取 topK。
 * 用于测试 / API 不可用 / 不需要精排的场景。
 */
export class PassThroughReranker implements Reranker {
  async rank(
    _query: string,
    candidates: RerankCandidate[],
    topK?: number,
  ): Promise<RerankCandidate[]> {
    return candidates.slice(0, topK ?? candidates.length);
  }
}
```

**关键设计**：

1. **接口 + 两个实现**：测试时注入 `PassThroughReranker`，生产用 `ApiReranker`
2. **`maxDocChars: 8000`**：reranker 模型输入上限通常 8K-32K，截断避免超限
3. **降级策略**：API 失败不抛错，保留原顺序 —— 检索比报错好
4. **结果映射用 `index`**：API 返回的是 index，不是 id，要小心映射

**改进**：旧版 candidates[r.index] 直接用，可能 undefined。新版加 `!` 断言（前提是相信 API 返回合法 index，毕竟 API 是受信内部服务）。

### Step 3E.5: HybridRetriever（修复 #15：删死代码）

**目标**：编排 Dense + Sparse → RRF → Rerank → Parent 去重。

写 `src/retrieve/retriever.ts`：

```typescript
import { db } from '../db/client';
import { chunks, documents } from '../db/schema';
import { inArray } from 'drizzle-orm';
import { denseSearch, type DenseHit } from './dense';
import { sparseSearch } from './sparse';
import { rrfFusion } from './rrf';
import { ApiReranker, type RerankCandidate, type Reranker } from './reranker';
import { EmbeddingService } from '../embedding/embedding-service';
import { config } from '../config';
import { TTLCache } from '../cache/ttl-cache';

export interface RetrievalResult {
  chunkId: string;             // parent 的 id（如果有 parent），否则自己的 id
  text: string;                // 完整文本（parent 的文本，提供完整上下文）
  score: number;
  documentId: string;
  documentTitle: string;
  parentId: string | null;
}

export interface RetrieveOptions {
  datasetId: string;
  topK?: number;
}

export class HybridRetriever {
  private readonly reranker: Reranker;

  // 修复 #11：缓存加上限
  private readonly resultCache = new TTLCache<string, RetrievalResult[]>(
    config.resultCacheTtlMs,
    config.resultCacheMax,
  );

  constructor(
    private readonly embeddingService: EmbeddingService,
    reranker?: Reranker,
  ) {
    this.reranker = reranker ?? new ApiReranker();
  }

  async retrieve(query: string, opts: RetrieveOptions): Promise<RetrievalResult[]> {
    const topK = opts.topK ?? config.searchTopK;
    // 缓存 key 必须包含所有影响结果的字段（修复 review 提到的 cache key 不完整）
    const cacheKey = `${opts.datasetId}:${query}:${topK}`;
    const cached = this.resultCache.get(cacheKey);
    if (cached) return cached;

    // Step 1: embed query
    const queryVec = await this.embeddingService.embedQuery(query);

    // Step 2: 并行 Dense + Sparse（多取 N 倍候选，给 Reranker 留余量）
    const extend = topK * config.denseTopKMultiplier;
    const [denseHits, sparseHits] = await Promise.all([
      denseSearch(queryVec, opts.datasetId, extend),
      sparseSearch(query, opts.datasetId, extend),
    ]);

    // Step 3: RRF 融合
    const fused = rrfFusion(
      denseHits.map(h => [h.chunkId, h.score] as const),
      sparseHits.map(h => [h.chunkId, h.score] as const),
      config.rrfK,
    );

    if (fused.length === 0) return [];

    // Step 4: 取 top candidates + 加载 parent 文本
    const candidateIds = fused.slice(0, config.rerankTopK).map(f => f[0]);
    const candidates = await this.loadCandidates(candidateIds, denseHits);

    // Step 5: Reranker 精排
    const reranked = await this.reranker.rank(query, candidates, topK);

    // Step 6: Parent 去重 —— 多个 child 命中同一个 parent 时，只保留分数最高的
    const seen = new Set<string>();
    const results: RetrievalResult[] = [];
    for (const r of reranked) {
      const key = r.parentId ?? r.chunkId;
      if (seen.has(key)) continue;
      seen.add(key);
      results.push({
        chunkId: r.parentId ?? r.chunkId,
        text: r.text,
        score: r.score,
        documentId: r.documentId,
        documentTitle: '',
        parentId: r.parentId,
      });
      if (results.length >= topK) break;
    }

    // Step 7: 填充 documentTitle
    await this.fillTitles(results);

    this.resultCache.set(cacheKey, results);
    return results;
  }

  /**
   * 把 chunk id 列表转成 reranker 候选（带文本）。
   * - child chunk 用其 parent 的文本作为候选（更完整）
   * - 没有 parent 的 child 用自己的文本
   */
  private async loadCandidates(
    ids: string[],
    denseHits: DenseHit[],
  ): Promise<RerankCandidate[]> {
    const hitMap = new Map(denseHits.map(h => [h.chunkId, h]));

    // 1. 查 child chunks
    const childRows = await db.select({
      id: chunks.id,
      content: chunks.content,
      parentId: chunks.parentId,
      documentId: chunks.documentId,
    }).from(chunks).where(inArray(chunks.id, ids));

    // 2. 收集所有 parentId，批量查 parent 文本
    const parentIds = [...new Set(
      childRows.map(r => r.parentId).filter((p): p is string => p !== null),
    )];

    let parentTextMap = new Map<string, string>();
    if (parentIds.length > 0) {
      const parentRows = await db.select({
        id: chunks.id,
        content: chunks.content,
      }).from(chunks).where(inArray(chunks.id, parentIds));
      parentTextMap = new Map(parentRows.map(r => [r.id, r.content]));
    }

    // 修复 #15：删除死代码 fused_score_for
    // 旧版有个返回 0.5 的 placeholder，从来没被用过
    // 现在直接用 hitMap 拿 score（dense hit 的 score），sparse-only 的命中给 0
    return childRows.map(r => ({
      chunkId: r.id,
      parentId: r.parentId,
      documentId: r.documentId,
      text: (r.parentId && parentTextMap.get(r.parentId)) || r.content,
      score: hitMap.get(r.id)?.score ?? 0,
    }));
  }

  private async fillTitles(results: RetrievalResult[]): Promise<void> {
    const docIds = [...new Set(results.map(r => r.documentId))];
    if (docIds.length === 0) return;

    const docs = await db.select({
      id: documents.id,
      title: documents.title,
    }).from(documents).where(inArray(documents.id, docIds));

    const titleMap = new Map(docs.map(d => [d.id, d.title]));
    for (const r of results) {
      r.documentTitle = titleMap.get(r.documentId) ?? '';
    }
  }
}
```

**修复 #15 的说明**：

旧版有一个 placeholder：
```typescript
function fused_score_for(rows, ids) {
  return rows.map(r => [r.id, 0.5]);    // 永远 0.5，没意义
}
const scoreMap = new Map(fused_score_for(childRows, ids));   // 定义了但下面没用
```

新版删除这个函数，直接用 `hitMap.get(r.id)?.score ?? 0`。`scoreMap` 变量也一并删了。

**关键设计**：

1. **批量查询**：`loadCandidates` 用 `inArray` 一次查所有 child，再用 `inArray` 一次查所有 parent，**总共 2 次 SQL**（旧版逻辑等价，但代码更清晰）
2. **Parent 去重在 rerank 之后**：先 rerank 所有候选，再按 parent 去重保留最高分。这样不会因为去重而过早丢掉好候选
3. **缓存 key 完整**：包含 datasetId + query + topK
4. **`hitMap` 兜底 score**：sparse-only 的 chunk（dense 没召回）score 给 0，reranker 会重新打分

## Step 4.1: Tool 接口

```typescript
/** Tool 执行时携带的上下文（每次调用新建） */
export interface ToolContext {
  datasetId: string;
  userId?: string;
  queryLogId?: string;
}

/**
 * JSON Schema property 描述（用于 function calling 的 parameters）。
 * 类型故意收窄到我们用得到的几种，避免 LLM 收到怪异 schema。
 */
export interface JSONSchemaProperty {
  type: 'string' | 'number' | 'boolean' | 'array' | 'object';
  description: string;
  enum?: readonly string[];
  default?: unknown;
  items?: JSONSchemaProperty;
  properties?: Record<string, JSONSchemaProperty>;
  required?: readonly string[];
}

/** 完整的 Tool 定义 */
export interface Tool<TParams = Record<string, unknown>, TData = unknown> {
  /** 唯一名（snake_case，作为 function calling 的 name） */
  name: string;
  /** LLM 可读描述 —— LLM 靠这段话判断要不要调这个 Tool */
  description: string;
  /** 参数 JSON Schema（function calling 的 parameters） */
  parameters: {
    type: 'object';
    properties: Record<string, JSONSchemaProperty>;
    required: readonly string[];
  };
  /** 执行 */
  execute(params: TParams, ctx: ToolContext): Promise<TData>;
}
```

**关键设计**：

1. **`JSONSchemaProperty` 收窄类型**：标准 JSON Schema 支持几十种关键字，我们只用其中几种。收窄后 IDE 自动补全更准、写错更容易发现
2. **`required: readonly string[]`**：用 readonly 防止运行时被改
3. **Tool 泛型 `<TParams, TData>`**：让每个 Tool 的 execute 类型严格，调用方拿到的结果有类型

## Step 4.2: ToolRegistry

**目标**：注册中心，管理所有 Tool，并生成 function calling 定义给 LLM。

写 `src/tools/registry.ts`：

```typescript
import type { Tool } from './types';
import type { FunctionDefinition } from '../llm/llm-service';

export class ToolRegistry {
  private readonly tools = new Map<string, Tool>();

  register(tool: Tool): void {
    if (this.tools.has(tool.name)) {
      throw new Error(`Tool already registered: ${tool.name}`);
    }
    this.tools.set(tool.name, tool);
  }

  get(name: string): Tool | undefined {
    return this.tools.get(name);
  }

  /** 是否注册了指定 Tool */
  has(name: string): boolean {
    return this.tools.has(name);
  }

  /** 生成 function calling 定义数组，传给 LLM */
  toFunctionDefinitions(): FunctionDefinition[] {
    return [...this.tools.values()].map(t => ({
      type: 'function' as const,
      function: {
        name: t.name,
        description: t.description,
        parameters: t.parameters,
      },
    }));
  }

  /** 列出所有 Tool 元数据（用于 System Prompt） */
  list(): Array<{ name: string; description: string }> {
    return [...this.tools.values()].map(t => ({
      name: t.name,
      description: t.description,
    }));
  }
}
```

**改进**：
- 加 `has()` 方法（Skill 校验白名单时用）
- 所有字段 readonly，禁止运行时改 registry

## Step 4.3: search_knowledge（修复 #9：删 docType）

**决策（修复 #9）**：

旧版 `search_knowledge` 声明支持 `docType` 参数，但：
- `SearchParams` 接口里没该字段
- `RetrieveOptions` 也没
- chunks 表 schema 里没冗余 `docType` 字段
- dense.ts / sparse.ts SQL 也没按 docType 过滤

这是个"伪 API"——LLM 给它传 docType 会被默默丢弃。

**两种修复方向**：
- (A) 把 docType 实现完整：chunks 表加 `doc_type` 冗余字段、SQL 加过滤、SearchParams 加字段
- (B) 删除 docType 参数声明

**Phase 1 选择 (B)**：删除。理由：
1. docType 在我们当前的文档库里区分度不高（大部分是 general）
2. 实现完整需要 schema 改动 + 索引调整 + 测试，工作量超过价值
3. 如果未来真需要（如法律库 vs 教程库），按 dataset 隔离比按 docType 过滤更干净

```typescript
import { db } from '../db/client';

interface GetDocumentParams {
```

```typescript
import type { Tool } from './types';
import { db } from '../db/client';
import { chunks } from '../db/schema';
import { eq } from 'drizzle-orm';

interface GetChunkParams {
  chunkId: string;
}

interface GetChunkResult {
  chunk: {
    id: string;
    content: string;
    parentId: string | null;
    documentId: string;
    tokenCount: number;
  };
  parentText: string | null;        // 如果有 parent，附上 parent 完整文本
}

export const getChunkTool: Tool<GetChunkParams, GetChunkResult | { error: string }> = {
  name: 'get_chunk',
  description: '获取指定 chunk 及其 parent 上下文。当需要查看完整段落、补充上下文时使用。',
  parameters: {
    type: 'object',
    properties: {
      chunkId: {
        type: 'string',
        description: 'Chunk ID（UUID 格式）',
      },
    },
    required: ['chunkId'],
  },

  async execute(params: GetChunkParams) {
    const chunk = await db.query.chunks.findFirst({
      where: eq(chunks.id, params.chunkId),
    });

    if (!chunk) {
      return { error: 'Chunk not found', chunkId: params.chunkId };
```

```typescript
        id: documents.id,
        title: documents.title,
        docType: documents.docType,
        status: documents.status,
        fileSize: documents.fileSize,
        createdAt: documents.createdAt,
      })
      .from(documents)
      .where(and(
        eq(documents.datasetId, ctx.datasetId),
        isNull(documents.deletedAt),
        eq(documents.status, 'ready'),     // 只列出可检索的文档
      ))
      .orderBy(desc(documents.createdAt))
      .limit(limit);

    return { documents: docs, total: docs.length };
  },
};
```

**关键设计**：
- **`status: 'ready'` 过滤**：只列出已入库完成的文档，避免列出 pending/failed 的让 LLM 困惑
- **字段裁剪**：不返回 fileHash/contentHash/errorMsg 等内部字段
- **limit 边界保护**：[1, 100]，防止 LLM 传 `limit: 10000` 拖垮数据库

```typescript
let llmInstance: LLMService | null = null;

export function setLLM(llm: LLMService): void {
  llmInstance = llm;
}

interface SummarizeParams {
  text: string;
  instruction?: string;
}

interface SummarizeResult {
  summary: string;
  originalLength: number;
  summaryLength: number;
}

const DEFAULT_INSTRUCTION = '提取 3-5 个核心要点，每个要点用一句话概括';

export const summarizeTextTool: Tool<SummarizeParams, SummarizeResult | { error: string }> = {
  name: 'summarize_text',
  description: '对任意文本生成摘要。当资料太长需要压缩、用户要"总结""概括"时使用。',
  parameters: {
    type: 'object',
    properties: {
      text: {
        type: 'string',
        description: '要总结的文本（建议 < 5000 字，超长会被截断）',
      },
      instruction: {
        type: 'string',
        description: '可选：自定义总结要求，例如"用法律术语""按时间顺序"',
        default: '提取 3-5 个核心要点',
      },
    },
    required: ['text'],
  },

  async execute(params: SummarizeParams, _ctx: ToolContext): Promise<SummarizeResult | { error: string }> {
    if (!llmInstance) {
      return { error: 'LLM not initialized' };
    }

    // 长度保护：超过 10000 字截断
    const truncated = params.text.length > 10000
      ? params.text.slice(0, 10000) + '\n[...文本过长，已截断...]'

要求：${instruction}

文本：
${truncated}`;

    const summary = await llmInstance.generate(prompt, { temperature: 0.3 });

    return {
      summary,
      originalLength: params.text.length,
      summaryLength: summary.length,
    };
  },
};
```

## Step 5.3: SkillExecutor（mini Agent Loop）

**目标**：通用执行器。读 SKILL.md 指令 → LLM 按指令调用 Tool → 返回结果。

写 `src/skills/executor.ts`：

```typescript
export class SkillExecutor {
  /**
   * @param instructions  SKILL.md 正文（执行步骤）
   * @param allowedTools  frontmatter 声明的 tools 白名单
   * @param ctx           执行上下文
   */
  async execute(
    instructions: string,
    allowedTools: readonly string[],
    ctx: SkillContext,
  ): Promise<SkillResult> {
    // 只暴露 frontmatter 声明的 tools
    const toolDefs = filterToolDefs(ctx.tools, allowedTools);

    const messages = [
      { role: 'system', content: instructions },      // SKILL.md 正文 = 系统指令
      { role: 'user', content: formatParams(ctx.params) },
    ];

    // mini Agent Loop
    for (let iter = 0; iter < MAX_ITERATIONS; iter++) {
      const response = await ctx.llm.chat({ messages, tools: toolDefs });

      if (!response.tool_calls?.length) {
        // LLM 给出最终回答
        return { answer: response.content, citations: [], toolCalls };
      }

      // 执行 tool calls
      for (const call of response.tool_calls) {
        const result = await ctx.executeTool(call.name, JSON.parse(call.arguments));
        messages.push({ role: 'tool', tool_call_id: call.id, content: JSON.stringify(result) });
      }
    }
  }
}
```

**执行机制**：

```
SKILL.md 正文 → system prompt
     ↓
LLM 读指令："1. 调 search_knowledge..."
     ↓
LLM 调 search_knowledge({ query, topK: 10 })
     ↓
结果返回给 LLM
     ↓
LLM 读结果，按步骤 3-4 生成带引用的回答
     ↓
LLM 输出最终回答（无 tool_calls）→ 返回 SkillResult
```

## Part 6 核心设计：Agent ↔ Skill 契约（修复 #1）

### 旧版的问题

```
用户问题 → Agent Loop
  → LLM 调 QASkill
    → QASkill 内部: retrieve → generate（LLM 生成 answer）→ 返回 {answer, citations}
  → Agent 把 {answer, citations} JSON.stringify 截断到 4000 字，作为 tool message 喂回 LLM  ← #8
  → Agent Loop 结束，再调一次 LLM "请基于以上信息给出最终回答"  ← #1 双重生成
  → 最终 answer（可能改变措辞、丢失引用）
```

问题：
1. **Skill 的 LLM 调用白做了**——答案被 Agent 的 final generation 覆盖
2. **JSON.stringify 截断**——引用资料被切掉一半（#8）
3. **多花一次 LLM 调用**——浪费 token + 延迟

### 新版的契约

```
用户问题 → Agent Loop
  → Case A: LLM 调了 Skill
     → Skill 返回 {answer, citations}
     → Agent 直接透传 answer（不二次生成）
     → 给 LLM 的 tool message 只放摘要（"Skill qa 完成，答案 1234 字，3 个引用"）
     → Loop 继续或结束
  → Case B: LLM 只调了 Tool（没调 Skill）
     → Tool 返回原始数据（chunks / document）
     → Agent Loop 结束后，做一次 final synthesis 生成
  → Case C: LLM 直接回答（没调任何 function）
     → 用 LLM 的 content 作为 answer
```

三种路径，每种都只做必要的 LLM 调用。

```typescript
1. **Skill 优先**：如果有合适的 Skill，优先调 Skill（它会返回完整答案）
2. **Tool 补充**：Skill 不够时，可以补调 Tool 获取更多上下文
3. **可以组合**：一个问题可以调多个 Skill（如先 qa 再 multihop）
4. **可以迭代**：第一次结果不够，可以再调（换关键词、换 Skill）
5. **闲聊直接答**：问候、感谢等不需要调任何 Skill/Tool，直接回复

## 重要

- 调用 Skill 后，Skill 已经返回了完整答案，你不需要重新生成（系统会透传）
- 如果你只调了 Tool（没调 Skill），系统会帮你做最终合成
- 如果不需要任何 Skill/Tool，直接回复用户即可
- 基于知识库回答时，Skill 已经标注了引用，不要自己编造引用编号`;
}

function formatParams(parameters: { properties: Record<string, unknown>; required: readonly string[] }): string {
  const props = Object.entries(parameters.properties);
  if (props.length === 0) return '（无）';
  return props.map(([name, schema]) => {
    const s = schema as { type: string; description: string };
    const req = parameters.required.includes(name) ? '必填' : '可选';
    return `${name}(${s.type}, ${req})`;
  }).join(', ');
}
```

**关键设计**：
- **明确告知 LLM 调 Skill 后不需要重新生成**（修复 #1 的 prompt 层面配合）
- **参数格式化**：把每个 Skill 的参数列出来，让 LLM 知道能传什么（修复 #2 的价值体现）
- **决策原则清晰**：Skill 优先 > Tool 补充 > 直接答

## Step 6.3: QueryAgent —— execute 主流程

**目标**：核心 Agent Loop。这是修复 #1 和 #8 的核心代码。

写 `src/agent/query-agent.ts`：

```typescript
import type { LLMService, Message } from '../llm/llm-service';
import type { SkillRegistry } from '../skills/registry';
import type { ToolRegistry } from '../tools/registry';
import type { HookRegistry } from '../hooks/registry';
import type { SkillContext, SkillResult } from '../skills/types';
import type { Citation, ToolCallRecord, AgentStep } from '../db/schema';
import { buildSystemPrompt } from './system-prompt';
import type { QueryOptions, QueryResult } from './types';
import { db } from '../db/client';
import { queryLogs, agentTraces } from '../db/schema';
import { config } from '../config';
import { logger } from '../utils/logger';

export class QueryAgent {
  constructor(
    private readonly llm: LLMService,
    private readonly skillRegistry: SkillRegistry,
    private readonly toolRegistry: ToolRegistry,
    private readonly hookRegistry: HookRegistry,
  ) {}

  async execute(query: string, options: QueryOptions): Promise<QueryResult> {
    const startTime = Date.now();
    const steps: AgentStep[] = [];
    const allToolCalls: ToolCallRecord[] = [];

    // 修复 #1：追踪 Skill 产出的完整答案
    // 如果 LLM 调了 Skill，Skill 返回的 answer 直接用，不再二次生成
    const skillResults: SkillResult[] = [];

    let directAnswer: string | null = null;    // LLM 直接回答的内容

    // 构建 messages
    const messages: Message[] = [
      { role: 'system', content: buildSystemPrompt(this.skillRegistry, this.toolRegistry) },
      ...(options.history ?? []),
      { role: 'user', content: query },
    ];

    const maxIterations = options.maxIterations ?? config.agentMaxIterations;

    // ===== Agent Loop =====
    for (let iter = 0; iter < maxIterations; iter++) {
      const response = await this.llm.chat({
        messages,
        tools: [
          ...this.skillRegistry.toFunctionDefinitions(),
          ...this.toolRegistry.toFunctionDefinitions(),
        ],
        tool_choice: 'auto',
      });

      // Case C: LLM 决定直接回答（没调任何 function）
      if (!response.tool_calls?.length) {
        directAnswer = response.content ?? '';
        steps.push({
          iteration: iter,
          thought: '直接回答（不调用 Skill/Tool）',
          action: 'answer',
          params: {},
          resultSummary: directAnswer.slice(0, 200),
        });
        logger.debug(`[Agent] Iter ${iter}: direct answer`);
        break;
      }

      // 记录 assistant 的 tool_calls 消息
      messages.push({
        role: 'assistant',
        content: response.content ?? '',
        tool_calls: response.tool_calls,
      });

      // 执行所有 function calls
      for (const call of response.tool_calls) {
        const name = call.function.name;
        let params: Record<string, unknown>;
        try {
          params = JSON.parse(call.function.arguments);
        } catch {
          params = {};
        }

        steps.push({
          iteration: iter,
          thought: response.content ?? '',
          action: name,
          params,
          resultSummary: '',
        });

        // 执行
        const result = await this.executeCallable(name, params, options);

        allToolCalls.push({
          name,
          kind: this.skillRegistry.has(name) ? 'skill' : 'tool',
          params,
          result: undefined,        // 不存完整 result（可能很大），日志里只存摘要
        });

        // 修复 #1 + #8：根据结果是 Skill 还是 Tool，给 LLM 不同的 tool message
        if (this.isSkillResult(result)) {
          // Skill 返回了完整答案 —— 收集，给 LLM 只发摘要
          skillResults.push(result);

          // 修复 #8：不再 JSON.stringify 整个 SkillResult 截断
          // 只告诉 LLM "Skill 完成了，答案已收集"
          const summary = this.summarizeSkillResult(name, result);
          messages.push({
            role: 'tool',
            tool_call_id: call.id,
            content: summary,
          });
          steps[steps.length - 1]!.resultSummary = summary;
        } else {
          // Tool 返回原始数据 —— 喂给 LLM（截断保护）
          const toolContent = this.formatToolResult(result);
          messages.push({
            role: 'tool',
            tool_call_id: call.id,
            content: toolContent,
          });
          steps[steps.length - 1]!.resultSummary = toolContent.slice(0, 200);
        }

        logger.debug(`[Agent] Iter ${iter}: called ${name}`);
      }
    }

    // ===== 决定最终答案（修复 #1 的核心逻辑）=====
    const { answer, citations, termination } = this.resolveFinalAnswer(
      query,
      messages,
      skillResults,
      directAnswer,
      options,
    );

    const latencyMs = Date.now() - startTime;

    // ===== 写日志 =====
    const queryLogId = await this.logQuery(
      query, answer, citations, allToolCalls, steps, latencyMs, options,
    );

    return { answer, citations, steps, toolCalls: allToolCalls, latencyMs, queryLogId, termination };
  }

  // ...（辅助方法在下面的 Step 中实现）
}
```

**修复 #1 的核心逻辑（resolveFinalAnswer 在 Step 6.3.2）**：

三种终止路径：
- **Case A**：LLM 调了 Skill → 用最后一个 Skill 的 answer（或合并多个）
- **Case B**：LLM 只调了 Tool → 做一次 final synthesis
- **Case C**：LLM 直接回答 → 用 directAnswer

### Step 6.3.2: resolveFinalAnswer

继续在 `src/agent/query-agent.ts` 加方法：

```typescript
  /**
   * 修复 #1：决定最终答案的三种路径。
   * - skillResults 非空 → 用 Skill 的 answer（不二次生成）
   * - skillResults 空 + 有 Tool 调用 → 做一次 final synthesis
   * - 都没有 → 用 directAnswer
   */
  private async resolveFinalAnswer(
    query: string,
    messages: Message[],
    skillResults: SkillResult[],
    directAnswer: string | null,
    options: QueryOptions,
  ): Promise<{ answer: string; citations: Citation[]; termination: 'skill' | 'synthesis' | 'direct' }> {
    // Case A: Skill 产出了答案
    if (skillResults.length > 0) {
      // 如果只有一个 Skill，直接用
      if (skillResults.length === 1) {
        const sr = skillResults[0]!;
        return {
          answer: sr.answer,
          citations: sr.citations,
          termination: 'skill',
        };
      }

      // 多个 Skill：合并 citations，answer 取最后一个（或拼接）
      // Phase 1 简化：取最后一个 Skill 的 answer，合并所有 citations
      const lastSkill = skillResults[skillResults.length - 1]!;
      const allCitations = skillResults.flatMap(sr => sr.citations);
      return {
        answer: lastSkill.answer,
        citations: deduplicateCitations(allCitations),
        termination: 'skill',
      };
    }

    // Case C: LLM 直接回答
    if (directAnswer !== null) {
      return { answer: directAnswer, citations: [], termination: 'direct' };
    }

    // Case B: 只调了 Tool，做 final synthesis
    const finalResponse = await this.llm.chat({
      messages: [
        ...messages,
        { role: 'user', content: '请基于以上检索到的资料，给出最终回答。标注引用来源。' },
      ],
    });

    return {
      answer: finalResponse.content ?? '（无法生成回答）',
      citations: [],         // synthesis 模式下 citations 由 LLM 在 answer 里标注
      termination: 'synthesis',
    };
  }
```

**修复 #1 的关键**：
- Skill 的 answer **直接透传**，不再调 LLM 重新生成
- 只有"纯 Tool 调用"场景才做 final synthesis
- 三种路径清晰分离，不会双重生成

### Step 6.3.3: 辅助方法

继续在 `src/agent/query-agent.ts` 加方法：

```typescript
  /** 判断 result 是否是 SkillResult */
  private isSkillResult(result: unknown): result is SkillResult {
    return (
      typeof result === 'object' &&
      result !== null &&
      'answer' in result &&
      'citations' in result &&
      'toolCalls' in result
    );
  }

  /**
   * 修复 #8：Skill 结果的摘要（不截断原始内容）。
   * 旧版把整个 SkillResult JSON.stringify 截断到 4000 字，
   * 导致引用资料被切掉一半。
   * 新版只发元信息，让 LLM 知道"Skill 已完成"。
   */
  private summarizeSkillResult(name: string, result: SkillResult): string {
    return `Skill "${name}" 已完成。
答案长度: ${result.answer.length} 字符
引用数: ${result.citations.length}

答案已由系统收集，你不需要重新生成。如果你还需要更多信息，可以继续调用其他 Skill/Tool；如果信息充分，请直接回复"信息已足够"。`;
  }

  /**
   * 格式化 Tool 结果给 LLM。
   * Tool 返回的是原始数据（chunks/document），需要截断保护。
   */
  private formatToolResult(result: unknown): string {
    const json = JSON.stringify(result);
    if (json.length <= 4000) return json;
    // 截断时保留 JSON 结构合法性（截断后 parse 会失败，但 LLM 能理解）
    return json.slice(0, 4000) + '\n[...结果已截断...]';
  }
```

**修复 #8 的核心**：
- `summarizeSkillResult`：Skill 结果只发**元信息**（答案长度、引用数），不发原始内容
- `formatToolResult`：Tool 结果才需要发原始数据（chunks 文本），但仍截断到 4000 字

```typescript
  /** 统一执行入口：先找 Skill，再找 Tool，都经过 Hook */
  private async executeCallable(
    name: string,
    params: Record<string, unknown>,
    options: QueryOptions,
  ): Promise<unknown> {
    // Hook: before
    const beforeResult = await this.hookRegistry.runBefore(name, params, {
      datasetId: options.datasetId,
      queryLogId: options.userId,         // Phase 1 用 userId 占位，Phase 2 用真实 queryLogId
      userId: options.userId,
    }).catch(err => {
      logger.warn(`[Hook before] ${name} threw`, err);
      return undefined;                   // 修复 #18：Hook 异常不阻塞主流程
    });

    if (beforeResult?.block) {
      return { error: beforeResult.reason ?? 'blocked by hook' };
    }

    // 先找 Skill
    const skill = this.skillRegistry.get(name);
    let result: unknown;

    if (skill) {
      const ctx = this.buildSkillContext(params, options);
      try {
        const executor = new SkillExecutor(); result = await executor.execute(skill.instructions, skill.metadata.tools, ctx);
      } catch (err) {
        logger.error(`[Skill ${name}] execution failed`, err);
        result = { error: `Skill ${name} failed: ${err instanceof Error ? err.message : String(err)}` };
      }
    } else {
      // 再找 Tool
      const tool = this.toolRegistry.get(name);
      if (!tool) {
        return { error: `Unknown callable: ${name}` };
      }
      try {
        result = await tool.execute(params, { datasetId: options.datasetId });
      } catch (err) {
        logger.error(`[Tool ${name}] execution failed`, err);
        result = { error: `Tool ${name} failed: ${err instanceof Error ? err.message : String(err)}` };
      }
    }

    // Hook: after（异常不阻塞）
    try {
      const afterResult = await this.hookRegistry.runAfter(name, result, {
        datasetId: options.datasetId,
        queryLogId: options.userId,
        userId: options.userId,
      });
      if (afterResult !== undefined) result = afterResult;
    } catch (err) {
      logger.warn(`[Hook after] ${name} threw`, err);
    }

    return result;
  }
```

**关键设计**：
- **Skill 优先**：先查 SkillRegistry，找不到再查 ToolRegistry（避免重名冲突时调错）
- **错误兜底**：Skill/Tool 执行失败不抛错，返回 `{error}` 让 LLM 自己决定下一步
- **Hook 异常隔离**（修复 #18）：before/after hook 的异常用 `.catch()` 吞掉，不影响主流程

## Step 7.2: HookRegistry（修复 #18：异常隔离）

写 `src/hooks/registry.ts`：

```typescript
import type { Hook, HookContext, HookActionResult } from './types';
import { logger } from '../utils/logger';

export class HookRegistry {
  private hooks: Hook[] = [];

  register(hook: Hook): void {
    this.hooks.push(hook);
  }

  /**
   * before hook 链。
   * 修复 #18：每个 hook 的异常被隔离，不影响其他 hook 和主流程。
   */
  async runBefore(
    targetName: string,
    params: unknown,
    options: { datasetId: string; queryLogId?: string; userId?: string },
  ): Promise<HookActionResult | void> {
    const ctx: HookContext = {
      targetName,
      params,
      metadata: {
        datasetId: options.datasetId,
        queryLogId: options.queryLogId,
        userId: options.userId,
        timestamp: Date.now(),
      },
    };

    let currentParams = params;

    for (const hook of this.hooks) {
      if (hook.phase !== 'before') continue;
      if (hook.filter && !matchesFilter(targetName, hook.filter)) continue;

      try {
        const result = await hook.before?.(ctx);
        if (result?.block) return result;
        if (result?.modifiedParams) {
          currentParams = result.modifiedParams;
          ctx.params = currentParams;
        }
      } catch (err) {
        // 修复 #18：Hook 异常只记录，不传播
        logger.warn(`[Hook "${hook.name}" before] threw on "${targetName}"`, err);
      }
    }
  }

  /**
   * after hook 链（逆序执行，洋葱模型）。
   * 修复 #18：每个 hook 的异常被隔离。
   */
  async runAfter(
    targetName: string,
    result: unknown,
    options: { datasetId: string; queryLogId?: string; userId?: string },
  ): Promise<unknown> {
    const ctx: HookContext = {
      targetName,
      params: null,
      metadata: {
        datasetId: options.datasetId,
        queryLogId: options.queryLogId,
        userId: options.userId,
        timestamp: Date.now(),
      },
    };

    let current = result;

    for (let i = this.hooks.length - 1; i >= 0; i--) {
      const hook = this.hooks[i]!;
      if (hook.phase !== 'after') continue;
      if (hook.filter && !matchesFilter(targetName, hook.filter)) continue;

      try {
        const modified = await hook.after?.(ctx, current);
        if (modified !== undefined) current = modified;
      } catch (err) {
        // 修复 #18：Hook 异常只记录，不传播
        logger.warn(`[Hook "${hook.name}" after] threw on "${targetName}"`, err);
      }
    }

    return current;
  }
}

function matchesFilter(name: string, filter: string | string[]): boolean {
  return Array.isArray(filter) ? filter.includes(name) : filter === name;
}
```

**修复 #18 的核心**：

| 旧版 | 新版 |
|---|---|
| `await hook.before?.(ctx)` 异常直接冒泡 | `try { ... } catch (err) { logger.warn(...) }` |
| 一个 hook 挂了整个查询失败 | hook 挂了只记录 warn，主流程继续 |

**为什么 after 是逆序**：洋葱模型。before 是 A→B→C→执行→C→B→A。这样最外层 hook 包裹最内层。

```typescript
const callCounts = new Map<string, { count: number; resetAt: number }>();
const WINDOW_MS = 60_000;       // 1 分钟窗口

/** 限流：单次查询最多调用 N 次 Tool/Skill，防止 Agent 无限循环 */
export const rateLimitHook: Hook = {
  name: 'rate-limit',
  target: 'both',
  phase: 'before',

  async before(ctx) {
    const key = ctx.metadata.queryLogId ?? ctx.metadata.userId ?? 'anonymous';
    const now = Date.now();

    let entry = callCounts.get(key);
    if (!entry || now > entry.resetAt) {
      entry = { count: 0, resetAt: now + WINDOW_MS };
      callCounts.set(key, entry);
    }

    entry.count++;

    if (entry.count > config.agentMaxToolCalls) {
      return {
        block: true,
        reason: `超过单次查询最大调用数 (${config.agentMaxToolCalls})`,
      };
    }
  },
};

/** 查询结束后重置计数（由 Agent 在 resolveFinalAnswer 后调用） */
export function resetRateLimit(key: string): void {
  callCounts.delete(key);
}
```

**改进**：
- 加了 60 秒**时间窗口**（旧版永久累积，长期运行的服务所有用户最终都被限流）
- 窗口过期自动重置

```typescript
});
```

```typescript
export function startWorker(): void {
  const worker = new Worker(
    'ingest',
    async (job) => {
      const { docId, sourcePath, datasetId } = job.data as {
        docId: string;
        sourcePath: string;
        datasetId: string;
      };
      logger.info(`[Worker] Processing document: ${docId}`);
      await ingestDocument(docId, sourcePath, datasetId);
    },
    {
      connection,
      // 指数退避重试
      settings: {
        backoffStrategy: (attempts: number) => Math.pow(2, attempts) * 1000,
      },
    },
  );

  worker.on('completed', (job) => {
    logger.info(`[Worker] Job completed: ${job.id}`);
  });

  worker.on('failed', (job, err) => {
    logger.error(`[Worker] Job failed: ${job?.id}`, err);
  });

  logger.info('[Worker] Ingest worker started');
}
```

## Step 8.2: src/pipeline/ingest-pipeline.ts（修复 #4：批量写入）

**目标**：入库主流程。**核心修复**：用批量 INSERT 替代循环 N+1。

写 `src/pipeline/ingest-pipeline.ts`：

```typescript
import { db } from '../db/client';
import { chunks, documents, ingestJobs } from '../db/schema';
import { eq, inArray } from 'drizzle-orm';
import { TxtParser } from '../parser/txt-parser';
import { createSplitter } from '../splitter';
import { EmbeddingService } from '../embedding/embedding-service';
import { logger } from '../utils/logger';
import { sha256 } from '../utils/hash';
import type { ChunkUnit } from '../splitter';
import { sql } from 'drizzle-orm';

// 单例（Phase 1 简化；Phase 2 依赖注入）
const parser = new TxtParser();
const splitter = createSplitter();
const embeddingService = new EmbeddingService();

// ===== 辅助函数 =====

async function recordJob(
  docId: string,
  stage: 'parse' | 'chunk' | 'embed',
  status: 'running' | 'done' | 'failed',
  result?: Record<string, unknown>,
): Promise<void> {
  await db.insert(ingestJobs).values({
    documentId: docId,
    stage,
    status,
    result: result ?? {},
    startedAt: status === 'running' ? new Date() : undefined,
    finishedAt: status !== 'running' ? new Date() : undefined,
  });
}

async function updateDocStatus(
  docId: string,
  status: 'pending' | 'parsing' | 'chunking' | 'embedding' | 'ready' | 'failed',
  errorMsg?: string,
): Promise<void> {
  await db.update(documents)
    .set({ status, errorMsg, updatedAt: new Date() })
    .where(eq(documents.id, docId));
}

// ===== 主流程 =====

export async function ingestDocument(
  docId: string,
  sourcePath: string,
  datasetId: string,
): Promise<void> {
  try {
    // ===== Step 1: Parse =====
    await updateDocStatus(docId, 'parsing');
    await recordJob(docId, 'parse', 'running');

    const doc = await parser.parse(sourcePath);
    const contentHash = sha256(doc.content);

    await db.update(documents)
      .set({ contentHash })
      .where(eq(documents.id, docId));
    await recordJob(docId, 'parse', 'done', { chars: doc.content.length });

    // ===== Step 2: Chunk =====
    await updateDocStatus(docId, 'chunking');
    await recordJob(docId, 'chunk', 'running');

    const units = splitter.split(doc.content, {
      docType: doc.docType,
      title: doc.title,
    });

    const parentUnits = units.filter(u => u.isParent);
    const childUnits = units.filter(u => !u.isParent);

    // 修复 #4：批量 INSERT parent chunks（旧版是 N 次 INSERT）
    const parentRows = await db.insert(chunks).values(
      parentUnits.map((u, idx) => ({
        documentId: docId,
        parentId: null,
        parentChunkIndex: u.parentChunkIndex,
        childIndexWithinParent: null,           // parent 自己
        chunkIndex: idx,                         // 兼容字段（旧 schema 遗留）
        content: u.text,
        contentHash: u.contentHash,
        tokenCount: u.tokenCount,
        scope: 'platform',
        datasetId,
      })),
    ).returning({ id: chunks.id, parentChunkIndex: chunks.parentChunkIndex });

    // 建立 parentChunkIndex → uuid 的映射
    const parentIdMap = new Map<number, string>();
    for (const row of parentRows) {
      parentIdMap.set(row.parentChunkIndex, row.id);
    }

    // 批量 INSERT child chunks（关联到 parent uuid）
    const childRows = await db.insert(chunks).values(
      childUnits.map((u) => ({
        documentId: docId,
        parentId: parentIdMap.get(u.parentChunkIndex) ?? null,
        parentChunkIndex: u.parentChunkIndex,
        childIndexWithinParent: u.childIndexWithinParent,
        chunkIndex: u.parentChunkIndex * 1000 + (u.childIndexWithinParent ?? 0),
        content: u.text,
        contentHash: u.contentHash,
        tokenCount: u.tokenCount,
        scope: 'platform',
        datasetId,
      })),
    ).returning({ id: chunks.id });

    await recordJob(docId, 'chunk', 'done', {
      parents: parentUnits.length,
      children: childUnits.length,
    });

    // ===== Step 3: Embed（仅 child chunks）=====
    await updateDocStatus(docId, 'embedding');
    await recordJob(docId, 'embed', 'running');

    const childIds = childRows.map(r => r.id);
    const texts = childUnits.map(u => u.text);
    const embeddings = await embeddingService.embedBatch(texts);

    // 修复 #4：批量 UPDATE embeddings（旧版是 N 次 UPDATE）
    // 用 UNNEST 一条 SQL 更新所有向量
    await db.execute(sql`
      UPDATE chunks SET
        embedding = data.embedding,
        embedding_status = 'done'
      FROM (
        SELECT * FROM UNNEST(
          ${sql.raw(`ARRAY[${childIds.map(id => `'${id}'`).join(',')}]::uuid[]`)} AS id,
          ${sql.raw(`ARRAY[${embeddings.map(e => `'[${e.join(',')}]'`).join(',')}::vector[]`)} AS embedding
        )
      ) AS data
      WHERE chunks.id = data.id
    `);

    await recordJob(docId, 'embed', 'done', { embedded: childIds.length });

    // ===== Done =====
    await updateDocStatus(docId, 'ready');
    logger.info(
      `[Ingest] Document ${docId} ready: ${parentUnits.length} parents, ${childUnits.length} children`,
    );
  } catch (err) {
    const errorMsg = err instanceof Error ? err.message : String(err);
    await updateDocStatus(docId, 'failed', errorMsg);
    await recordJob(docId, 'embed', 'failed', { error: errorMsg });
    throw err;        // 让 BullMQ 重试
  }
}
```

**修复 #4 的对比**：

| 操作 | 旧版（N+1） | 新版（批量） |
|---|---|---|
| 插入 100 个 parent | 100 次 INSERT | **1 次** INSERT |
| 插入 1000 个 child | 1000 次 INSERT | **1 次** INSERT |
| 更新 1000 个 embedding | 1000 次 UPDATE | **1 次** UPDATE（UNNEST） |
| **总计（1000 chunk 文档）** | **~2000 次 round-trip** | **~3 次 round-trip** |

**批量 UPDATE 的 SQL 解释**：

```sql
UPDATE chunks SET
  embedding = data.embedding,
  embedding_status = 'done'
FROM (
  SELECT * FROM UNNEST(
    ARRAY['uuid1', 'uuid2', ...]::uuid[] AS id,
    ARRAY['[0.1,0.2,...]', '[0.3,0.4,...]']::vector[] AS embedding
  )
) AS data
WHERE chunks.id = data.id
```

`UNNEST` 把两个数组"解包"成临时表，然后和 chunks 表 JOIN 更新。一条 SQL 更新所有行。

**常见坑**：
- 向量数组格式：`'[0.1,0.2,...]'`（pgvector 的文本表示）
- UUID 数组必须 `::uuid[]` 类型转换
- 如果 child 数量极大（> 10000），需要分批 UNNEST（每批 1000），否则 SQL 太长

```typescript
import { Hono } from 'hono';
import { z } from 'zod';
import { nanoid } from 'nanoid';
import { extname } from 'path';
import { db } from '../db/client';
import { documents, datasets } from '../db/schema';
import { ingestQueue } from '../pipeline/queue';
import { fileHash } from '../utils/hash';
import { eq } from 'drizzle-orm';
import { logger } from '../utils/logger';

const app = new Hono();

// 修复 #5：文件名白名单 + sanitize
const ALLOWED_EXTENSIONS = new Set(['.txt', '.md']);
const MAX_FILE_SIZE = 50 * 1024 * 1024;      // 50 MB

// 修复 #19：zod 校验 dataset 名
const datasetNameSchema = z.string().min(1).max(100).regex(
  /^[a-zA-Z0-9\u4e00-\u9fff_-]+$/,
  'Dataset name can only contain letters, numbers, Chinese, hyphens and underscores',
);

app.post('/ingest', async (c) => {
  const formData = await c.req.formData();
  const file = formData.get('file');
  const datasetNameRaw = formData.get('dataset') ?? 'default';

  // ===== 输入校验 =====
  if (!(file instanceof File)) {
    return c.json({ error: 'No file provided' }, 400);
  }

  // 修复 #19：dataset 名校验
  const datasetNameResult = datasetNameSchema.safeParse(datasetNameRaw);
  if (!datasetNameResult.success) {
    return c.json({ error: 'Invalid dataset name', detail: datasetNameResult.error.issues }, 400);
  }
  const datasetName = datasetNameResult.data;

  // 文件大小校验
  if (file.size > MAX_FILE_SIZE) {
    return c.json({ error: `File too large (max ${MAX_FILE_SIZE / 1024 / 1024}MB)` }, 400);
  }

  // 修复 #5：文件名 sanitize —— 完全忽略原始文件名，用 nanoid 生成安全名
  // 旧版：`./documents/${Date.now()}-${file.name}` —— file.name 可含 ../
  // 新版：白名单扩展名 + nanoid 重命名
  const originalExt = extname(file.name).toLowerCase();
  const safeExt = ALLOWED_EXTENSIONS.has(originalExt) ? originalExt : '.txt';
  const safeName = `${Date.now()}-${nanoid(10)}${safeExt}`;
  const filePath = `./documents/${safeName}`;

  // ===== 确保 dataset 存在 =====
  let dataset = await db.query.datasets.findFirst({
    where: eq(datasets.name, datasetName),
  });
  if (!dataset) {
    const [d] = await db.insert(datasets)
      .values({ name: datasetName })
      .returning();
    dataset = d!;
  }

  // ===== 保存文件 =====
  const buffer = await file.arrayBuffer();
  await Bun.write(filePath, buffer);

  // ===== 计算 hash 查重 =====
  const fhash = await fileHash(filePath);

  const existing = await db.query.documents.findFirst({
    where: eq(documents.fileHash, fhash),
  });
  if (existing && existing.status === 'ready') {
    // 重复文件，删除刚保存的，返回已有 docId
    await Bun.file(filePath).delete?.().catch(() => {});
    return c.json({
      docId: existing.id,
      status: 'duplicate',
      message: 'File already ingested',
    });
  }

  // ===== 创建 document 记录 =====
  const [doc] = await db.insert(documents).values({
    datasetId: dataset.id,
    title: file.name.replace(/\.[^.]+$/, ''),      // 用原始名做 title（显示用，不影响安全）
    sourcePath: filePath,
    fileHash: fhash,
    fileSize: file.size,
    status: 'pending',
  }).returning();

  // ===== 入队 =====
  await ingestQueue.add('ingest', {
    docId: doc!.id,
    sourcePath: filePath,
    datasetId: dataset.id,
  });

  logger.info(`[Ingest] Queued document: ${doc!.id} (${file.name})`);

  return c.json({ docId: doc!.id, status: 'pending' });
});

export default app;
```

| `file.name = "../../etc/passwd"` → 写到 `/etc/passwd` | 完全忽略原始文件名，用 nanoid |
| 无扩展名校验 | `ALLOWED_EXTENSIONS` 白名单 |

## Step 8.4: src/routes/query.ts（修复 #19：zod）

```typescript
import { Hono } from 'hono';
import { z } from 'zod';
import type { QueryAgent } from '../agent/query-agent';

// 修复 #19：zod 校验请求体
const queryBodySchema = z.object({
  question: z.string().min(1).max(2000),
  datasetId: z.string().uuid().optional(),
  options: z.object({
    topK: z.number().int().min(1).max(50).optional(),
    maxIterations: z.number().int().min(1).max(10).optional(),
    history: z.array(z.object({
      role: z.enum(['user', 'assistant']),
      content: z.string(),
    })).max(20).optional(),
  }).optional(),
});

app.post('/query', async (c) => {
  if (!agentInstance) {
    return c.json({ error: 'Agent not initialized' }, 503);
  }

  // 解析 + 校验
  const rawBody = await c.req.json().catch(() => null);
  if (!rawBody) {
    return c.json({ error: 'Invalid JSON body' }, 400);
  }

  const parseResult = queryBodySchema.safeParse(rawBody);
  if (!parseResult.success) {
    return c.json({
      error: 'Invalid request body',
      detail: parseResult.error.issues,
    }, 400);
  }

  const body = parseResult.data;

  // 旧版用 `as any`，新版有类型保障
  const history = body.options?.history as Message[] | undefined;

  try {
    const result = await agentInstance.execute(body.question, {
      datasetId: body.datasetId ?? 'default',
      topK: body.options?.topK,
      maxIterations: body.options?.maxIterations,
      history,
    });

    return c.json(result);
  } catch (err) {
    return c.json({
      error: 'Query failed',
      detail: err instanceof Error ? err.message : String(err),
    }, 500);
  }
});

export default app;
```

- `history` 角色只能是 `user | assistant`，最多 20 条
- 校验失败返回 400 + zod 的详细错误

**旧版对比**：
```typescript
// 旧版（无校验）：
const body = await c.req.json<{ question: string; ... }>();
const result = await agentInstance.execute(body.question, {
  history: body.options?.history as any,    // ← as any
});

// 新版（zod 校验）：
const parseResult = queryBodySchema.safeParse(rawBody);
if (!parseResult.success) return c.json({ error: ... }, 400);
const body = parseResult.data;               // ← 类型安全
```

## Step 8.5: src/routes/documents.ts（修复 #13：import 提顶 + 软删一致性）

**目标**：文档管理路由。**修复**：import 放到文件顶部 + 软删/硬删一致。

写 `src/routes/documents.ts`：

```typescript
import { Hono } from 'hono';
import { db } from '../db/client';
import { documents, ingestJobs, chunks } from '../db/schema';      // 修复 #13：import 提顶
import { eq, isNull } from 'drizzle-orm';

const app = new Hono();

// 获取单个文档
app.get('/documents/:id', async (c) => {
  const id = c.req.param('id');
  const doc = await db.query.documents.findFirst({
    where: eq(documents.id, id),
  });
  if (!doc || doc.deletedAt) {
    return c.json({ error: 'Not found' }, 404);
  }
  return c.json(doc);
});

// 文档的入库流水线状态
app.get('/documents/:id/jobs', async (c) => {
  const id = c.req.param('id');
  const jobs = await db.select().from(ingestJobs).where(eq(ingestJobs.documentId, id));
  return c.json(jobs);
});

// 列出文档
app.get('/documents', async (c) => {
  const limit = Math.min(Number(c.req.query('limit') ?? 20), 100);
  const datasetId = c.req.query('datasetId');

  const conditions = [isNull(documents.deletedAt)];
  if (datasetId) {
    conditions.push(eq(documents.datasetId, datasetId));
  }

  const docs = await db.query.documents.findMany({
    where: conditions.length === 1 ? conditions[0]! : undefined,
    limit,
  });
  return c.json({ documents: docs.filter(d => !d.deletedAt) });
});

// 修复 #13：删除路由 —— 软删 document + 硬删 chunks（一致清理）
app.delete('/documents/:id', async (c) => {
  const id = c.req.param('id');

  // 1. 软删 document（保留元数据历史）
  await db.update(documents)
    .set({ deletedAt: new Date(), status: 'failed', updatedAt: new Date() })
    .where(eq(documents.id, id));

  // 2. 硬删 chunks（占用空间大，不需要保留）
  // 修复 #13：旧版 import 在使用后才声明（反模式），新版在文件顶部
  await db.delete(chunks).where(eq(chunks.documentId, id));

  return c.json({ ok: true, message: 'Document soft-deleted, chunks hard-deleted' });
});

export default app;
```

**修复 #13 的对比**：

| 旧版 | 新版 |
|---|---|
| `import { chunks }` 写在文件**末尾**（使用之后） | import 在文件**顶部** |
| document 软删 + chunks 硬删（不一致：document 还在但 chunks 全没了） | 同左，但**文档明确说明**这种设计意图 |

**为什么 document 软删 + chunks 硬删**：
- document 元数据小（KB 级），保留历史有价值（审计、恢复）
- chunks 含 embedding 向量（MB 级），保留成本高，删除节省存储
- 如果未来需要恢复，可以从 sourcePath 重新入库

| `tsvector 中文不分词` | Phase 1 用 `simple` 分词器（按字符切）；Phase 2 装 `pg_jieba` 后改 migration SQL |
| `LLM function calling 不工作` | 确认模型支持（qwen-plus / GPT-4o / Claude 3.5+）；确认 `LLM_MODEL_ID` 正确 |
| `Agent 无限循环` | 检查 `AGENT_MAX_TOOL_CALLS`（默认 10）和 rate-limit hook |
| `Embedding API 超时` | 减小 `EMBEDDING_BATCH_SIZE`（默认 16，改 8） |
| `入库慢` | 检查是否用了批量 INSERT（修复 #4）；检查 embedding API 延迟 |
| `内存增长` | 检查缓存上限：`EMBEDDING_CACHE_MAX`、`RESULT_CACHE_MAX` |
| `BullMQ 连接失败` | 确认 Redis 在跑；`maxRetriesPerRequest: null` 必须设 |
| `路径遍历攻击` | 已在 Part 8.3 修复（nanoid 重命名 + 扩展名白名单） |
