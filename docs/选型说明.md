# Knowledge Core 选型说明文档

> 每个技术选型都回答四个问题：**是什么？为什么选它？对比了什么？底层怎么运作？**
> 面向小白，尽量用类比和例子。

---

# 一、运行时与语言

## 1.1 Bun（不选 Node.js / Deno）

### 是什么？

**运行时** = 能执行 JavaScript/TypeScript 代码的程序。最出名的是 Node.js（2009年），Bun（2023年）是新出的替代品，用 Zig 语言重写了底层。

### 为什么选 Bun？

| 能力 | Node.js | Bun |
|---|---|---|
| 跑 TypeScript | 需要 ts-node 编译 | **原生支持，直接 bun run index.ts** |
| 包管理器 | 需要 npm/yarn/pnpm | **内置 bun install** |
| 测试框架 | 需要 jest/vitest | **内置 bun test** |
| 打包器 | 需要 webpack/esbuild | **内置 bun build** |
| 启动速度 | 慢 | **快 4-5 倍** |

一句话：Node.js 需要拼装一堆工具，Bun 全内置了。

### 底层原理

- Node.js 的引擎是 V8（Chrome 的 JS 引擎，C++ 写的）
- Bun 的引擎是 JavaScriptCore（Safari 的 JS 引擎），启动更快
- Bun 用 Zig 语言重写了系统调用层（文件读写、网络），减少 JS 和 C 之间的转换开销

### 为什么不选 Node.js？

跑 TypeScript 要装一堆工具，包管理器选择困难，启动慢。

### 为什么不选 Deno？

生态太小，很多 npm 包不兼容，标准库不成熟。

---

## 1.2 TypeScript（不选 JavaScript / Python）

### 是什么？

TypeScript = JavaScript + 类型系统。

JavaScript：`function add(a, b) { return a + b }`，传 add(1, "hello") 返回 "1hello"，运行时才暴露问题。

TypeScript：`function add(a: number, b: number): number`，编译时就检查 add(1, "hello") 报错。

### 为什么选 TypeScript？

1. Drizzle ORM 强依赖类型——定义表结构后自动推断查询结果类型
2. 大项目没类型 = 地雷阵——6张表、5个Tool、6个Skill，没类型无法维护
3. IDE 智能提示——VSCode 对 TS 支持极好

### 底层原理

```
写 .ts 文件 → tsc 编译器检查类型（类型错误则编译失败）→ 编译成 .js（擦除类型）→ Bun 运行
```

类型只在开发时检查，运行时不存在，不影响性能。

### 为什么不选 Python？

用户指定了 Node/Bun 技术栈。即使不指定，TypeScript 在 I/O 密集型场景（大量 API 调用 + 数据库读写）比 Python 更合适（V8 JIT 编译 vs Python 解释执行 + GIL 限制并发）。

### 为什么不选纯 JavaScript？

没有类型，函数参数传错运行时才发现，重构困难。

# 二、Web 框架

## 2.1 Hono（不选 Express / Fastify / NestJS）

### 是什么？

Web 框架 = 处理 HTTP 请求的工具库（收请求 → 路由 → 调函数 → 返回响应）。

### 为什么选 Hono？

| 框架 | 诞生年 | 速度 | Bun 原生 | 类型安全 | 重量 |
|---|---|---|---|---|---|
| Express | 2010 | 慢 | ❌ | 弱 | 轻 |
| Fastify | 2017 | 快 | 部分 | 中等 | 中 |
| NestJS | 2018 | 中等 | ❌ | 强 | **重** |
| **Hono** | 2022 | **最快** | **✅** | **强** | **轻** |

### 底层原理：为什么 Hono 快？

Express 基于 Node.js 的 http 模块。Hono 基于 Web Standard API（Request/Response 是浏览器标准的 Fetch API），Bun 原生实现了这个标准，零适配层。

### 为什么不选 Express？

2010年设计，基于回调，不支持 Web Standard API，性能比 Hono 慢 10 倍+。

### 为什么不选 NestJS？

模仿 Angular（Spring 式设计），强制装饰器 + 依赖注入容器，学习曲线陡峭。知识库底座不需要这么重。

### 为什么不选 Fastify？

基于 Node http 模块，和 Bun 不是原生整合，类型推断不如 Hono。

# 三、ORM

## 3.1 Drizzle ORM（不选 Prisma / TypeORM / 原生 SQL）

### 是什么？

ORM = 用代码对象操作数据库，不直接写 SQL。

不用ORM: `SELECT * FROM documents WHERE status = 'ready'`
用ORM: `db.select().from(documents).where(eq(documents.status, 'ready'))` — 返回类型自动推断

### 为什么选 Drizzle？

| 维度 | Drizzle | Prisma | TypeORM | 原生SQL |
|---|---|---|---|---|
| 类型推断 | 最强（schema即真相） | 强（靠代码生成） | 弱 | 无 |
| pgvector | **原生支持** | ❌ 需raw query | ❌ | 手写 |
| 运行时依赖 | 零 | Rust binary | 重 | 零 |
| SQL可见性 | 清晰 | 黑盒 | 不可控 | 你写的 |

### 底层原理：Schema-first

```typescript
// 你写的 schema 就是数据库真实结构
export const documents = pgTable('documents', {
  id: uuid('id').primaryKey().defaultRandom(),
  title: text('title').notNull(),
});
// Drizzle 自动推断类型
type Document = typeof documents.$inferSelect; // { id: string; title: string }
```

Prisma 需要写 .prisma DSL → 跑 prisma generate → 生成 client。Drizzle 直接 TypeScript，没有代码生成步骤。

### 为什么不选 Prisma？

不支持 pgvector（需要 raw query，类型安全失效），需要 Rust binary（部署复杂），代码生成黑盒。

### 为什么不选 TypeORM？

装饰器太重，类型推断弱，SQL 生成不可控。

# 四、数据库

## 4.1 PostgreSQL + pgvector（不选 MySQL / MongoDB / 专用向量库）

### 是什么？

PostgreSQL：开源关系型数据库，30年历史。pgvector：PostgreSQL 扩展，让 PostgreSQL 能存储和检索向量。

### 核心优势：一个数据库搞定三件事

```
PostgreSQL:
  ① 关系数据（documents, chunks 表）—— 传统 SQL
  ② 向量检索（pgvector）—— embedding 向量 + HNSW 索引
  ③ 全文检索（tsvector）—— 关键词搜索 + GIN 索引
```

不用 PostgreSQL 的话需要三个系统：MySQL + Qdrant + Elasticsearch。

### 底层原理

关系型数据库：数据存在表里（行列），B-Tree 索引加速查询，事务（ACID）保证完整性。

pgvector：新增 vector 列类型（存浮点数组），用余弦距离算相似度：
```sql
SELECT * FROM chunks ORDER BY embedding <=> '[0.1, 0.5, ...]' LIMIT 5;
```

### 为什么不选 MySQL？

不支持 pgvector，全文检索对中文支持差，不支持向量类型。

### 为什么不选 MongoDB？

知识库需要大量关联查询（chunk → document → dataset），关系型更擅长。MongoDB 向量检索需要 Atlas（付费）。

### 为什么不选 Qdrant/Milvus？

百万级以下 pgvector 完全够用，专用向量库增加运维成本。数据一致性需要跨系统同步。到亿级再迁移也不迟。

---

## 4.2 HNSW 索引（不选 IVF / 暴力扫描）

### 是什么？

向量检索时暴力扫描（每个向量都算一遍）太慢。HNSW 是一种图索引，快速近似找到最相似的向量。

### 为什么选 HNSW？

| 索引 | 查询速度 | 召回率 | 适合规模 |
|---|---|---|---|
| 暴力扫描 | 极慢 | 100% | < 10万 |
| IVF | 快 | 90-95% | 大规模 |
| **HNSW** | **快** | **95-99%** | **中小规模** |

### 底层原理

HNSW = 分层可导航小世界图。类比地图缩放：Layer 2（最粗，几个核心节点）→ Layer 1（中等）→ Layer 0（最细，所有节点）。查询时从粗层大致定位，逐层缩小，不用遍历所有节点。

### 为什么不选 IVF？

IVF 需要先训练（聚类），数据变了要重新训练，召回率低于 HNSW。

# 五、稀疏检索（全文搜索）

## 5.1 PostgreSQL tsvector（不选 BM25 内存 / Elasticsearch）

### 是什么？

稀疏检索 = 关键词搜索。和向量检索（语义搜索）互补。向量检索找语义相近的内容，关键词检索精确匹配包含特定词的内容。

### 为什么选 tsvector？

PostgreSQL 内置全文检索，和 pgvector 同一个数据库同一张表，混合检索不需要跨系统。

### 底层原理

1. 分词："Docker是一个容器平台" → ['docker', '是', '一个', '容器', '平台']
2. 倒排索引（GIN）：记录每个词出现在哪些行（"docker" → [chunk #1, #5, #12]）
3. 查询：用户搜 "Docker 容器" → 找同时包含两个词的行
4. 排序（ts_rank_cd）：词频越高分数越高

### 为什么不选内存 BM25？

内存 BM25 重启后要重建索引，不持久化，数据量大时内存爆炸。tsvector 持久化在磁盘，原生 SQL 查询。

### 为什么不选 Elasticsearch？

太重（JVM，吃内存），多一个系统运维。pgvector + tsvector 已能做 Hybrid Search。

### 中文分词问题

PostgreSQL 默认 simple 分词器按空格切，中文不行。需要安装 pg_jieba 或 zhparser 扩展提供中文分词。

# 六、Embedding 模型

## 6.1 BGE-M3 / DashScope text-embedding-v3（不选 OpenAI / Cohere）

### 是什么？

Embedding = 把文本变成一组数字（向量），让计算机能计算"语义相似度"。

"Docker 容器化" → [0.12, -0.34, ..., 0.78]（1024个数字）
"容器虚拟化" → [0.11, -0.32, ..., 0.77]（和上面很接近）

### 底层原理

输入文本 → Tokenizer 分词 → Transformer 模型（多层 Self-Attention 计算词之间的关系）→ 池化得到整句表示 → 输出 1024 维向量。

模型通过海量文本的对比学习训练——让相似文本向量靠近，不相似远离。

### 为什么选 BGE-M3？

| 维度 | BGE-M3 | OpenAI | Cohere |
|---|---|---|---|
| 中文能力 | **最强** | 中等 | 中等 |
| 部署方式 | 开源（本地/API） | 仅API | 仅API |
| 数据安全 | ✅ 可本地 | ❌ 出境 | ❌ 出境 |
| 维度 | 1024 | 1536 | 1024 |
| 成本 | 本地免费/便宜 | 贵 | 贵 |

### 为什么不选 OpenAI？

中文不如 BGE-M3，数据出境，贵，维度 1536 比 1024 多 50%（存储和检索更慢）。

# 七、Reranker 精排

## 7.1 qwen3-rerank

### 是什么？

Reranker = 对向量检索结果重新排序的模型。粗排 20 条 → 精排选 5 条。

### 为什么需要？

向量检索是 Bi-Encoder（query 和 document 各自编码成向量再算距离），无法捕捉交互关系。Reranker 是 Cross-Encoder（query + document 拼一起输入模型），精度高得多。

### 底层原理

```
Bi-Encoder（向量检索）:
  Query → [向量A]          Document → [向量B]    → 算 A 和 B 的距离
  （query 和 doc 分开编码，快但不精确）

Cross-Encoder（Reranker）:
  [Query + Document] → 一起输入 Transformer → 输出相关度分数
  （query 和 doc 每个词都能交互，精确但慢）
```

流程：先用快的向量检索粗排 20 条 → 用准的 Reranker 精排选 5 条。

# 八、RRF 融合

## 8.1 Reciprocal Rank Fusion

### 是什么？

向量检索和关键词检索各返回排序列表，RRF 把两个列表融合成一个。

### 底层原理

公式：score(d) = Σ 1/(k + rank(d))，k=60

```
Dense 排名: A(1), B(2), C(3), D(4)
Sparse 排名: B(1), E(2), A(3), F(4)

A: 1/(60+1) + 1/(60+3) = 0.0323
B: 1/(60+2) + 1/(60+1) = 0.0325 ← 最高（两个列表都排前列）
```

### 为什么选 RRF？

尺度无关——Dense 分数 0~1，Sparse 分数可能 0~100，直接加权会被大数值主导。RRF 只看排名不看绝对分数，不需要归一化，不需要调参。

### 为什么不只 Dense 或不只 Sparse？

只 Dense：搜"第39条"可能匹配到"第40条"（语义相近但不精确）。
只 Sparse：搜"怎么用Docker"如果文档写"如何使用容器"就搜不到（关键词不匹配）。
混合：Dense 捕捉语义，Sparse 精确匹配，互补。

# 九、分块策略

## 9.1 Parent-Child 两级分块（不选固定大小 / 语义分块）

### 是什么？

两级切分：Parent（1200 token，无向量，上下文用）切成多个 Child（300 token，有向量，检索用）。

### 为什么需要？

矛盾：chunk 太小向量精准但上下文不够；chunk 太大上下文够但向量不精准。Parent-Child 解法：用 Child 检索（精准），命中后返回 Parent（完整上下文）。类比查字典——用关键词找到页码（Child），翻开后读整页（Parent）。

### 为什么不选固定大小？

一个 chunk 可能横跨两个主题，向量编码混杂，检索不精准。

### 为什么不选语义分块？

要先对每句话做 embedding 再算相邻句子相似度，很慢。Parent-Child + 递归结构化已够好。

## 9.2 递归结构化切分

### 是什么？

切分时优先按自然结构：段落 > 句子 > 子句 > 词 > 字符。绝不切半句话。

### 底层原理

按 \n\n 段落切 → 太大的段落降级按 句子切 → 还太大降级按 子句切 → 合并小碎片 → 加重叠（对齐到句号边界）。

# 十、Agent 架构

## 10.1 Function Calling 自主编排（不选规则路由 / LangChain）

### 是什么？

不硬编码 if/else 路由，而是 LLM 通过 function calling 自主选择调什么 Skill/Tool。

### 底层原理

LLM 支持 function calling：你传入可用函数的定义（name + description + parameters），LLM 读用户问题后返回结构化的函数调用（JSON），不是自由文本。

Agent Loop: LLM 决策 → 调用 Skill/Tool → 看结果 → 继续调或回答 → 循环。

### 为什么选 function calling？

| 维度 | 规则路由 | Function Calling |
|---|---|---|
| 新增Skill | 改路由代码 | 只注册，LLM自动发现 |
| 复合问题 | 无法处理 | LLM自动调多个Skill |
| 意图判断 | 关键词匹配（不准） | LLM语义理解（准） |

### 为什么不选规则路由？

意图分类不准（关键词漏匹配），无法组合多个 Skill，扩展要改代码。

### 为什么不选 LangChain Agent？

太重（大量抽象层，黑盒），Bun 兼容不确定。我们的 Agent Loop ~50行代码完全可控。

# 十一、Skill / Tool / Script 分层

### 为什么要分三层？

类比：Tool = 厨房刀具，Skill = 菜谱，Script = 菜谱步骤，Hook = 厨房规则。

分层让每个组件只管一件事：Tool 管查数据，Skill 管编排任务，Script 是 Skill 内部步骤。Tool 可被任何 Skill 复用，Script 可独立测试。

### 为什么 Skill 要有 script/ 子目录？

一个 Skill 可能有多个脚本（retrieve + generate + format），独立测试，可被其他 Skill 复用。

### SKILL.md 的作用

description 字段是 LLM 决策的依据。LLM 读这段话判断"这个用户问题适不适合用这个 Skill"。写得越清晰 LLM 选择越准确。

# 十二、任务队列与缓存

## 12.1 BullMQ

### 为什么需要？

文档入库（解析+分块+embedding）要几分钟，不能让 HTTP 请求阻塞。异步处理：上传后立即返回 → 后台 Worker 执行 → 完成后更新状态。

### 为什么选 BullMQ？

TypeScript 生态最成熟的队列，支持重试/延迟/优先级。Celery 是 Python 的，Temporal 太重。

## 12.2 TTL 缓存

### 为什么需要？

相同问题不重复 embedding（省 API 费用），热点问题缓存后 0ms 返回。

### 为什么选内存 TTL？

Phase 1 单机够用，零依赖。生产多实例时换 Redis。

# 十三、总结

| 层 | 选型 | 一句话理由 |
|---|---|---|
| Runtime | Bun | 原生TS、快、工具全内置 |
| 语言 | TypeScript | 类型安全、Drizzle强依赖 |
| Web框架 | Hono | 最快、Bun原生、类型安全 |
| ORM | Drizzle | pgvector原生、类型最强 |
| 数据库 | PostgreSQL | 关系+向量+全文三合一 |
| 向量索引 | pgvector+HNSW | 高召回、同库零运维 |
| 稀疏检索 | tsvector+GIN | 同库、持久化 |
| Embedding | BGE-M3 | 中文最强、多语言 |
| Reranker | qwen3-rerank | 中文精排、API可用 |
| 融合 | RRF | 尺度无关、无需调参 |
| 分块 | Parent-Child | 小搜大回、精准+完整 |
| Agent | Function Calling | LLM自主决策、无硬编码 |
| 架构 | Tool/Skill/Script/Hook | 分层解耦、可组合可扩展 |
| 队列 | BullMQ | TS生态、重试+异步 |
| 缓存 | 内存TTL | 零依赖、Phase 1够用 |

**核心原则**：
1. 能用一个系统就不开两个（PostgreSQL 搞定三件事）
2. LLM 能做的决策不硬编码（function calling 替代 if/else）
3. 分层解耦（Tool/Skill/Script 各司其职）
4. 类型安全（TypeScript + Drizzle）
5. 先简后繁（Phase 1 够用就不上重方案）
