# Agent 入驻流程

## 什么是入驻

Agent 入驻 = 首次进入项目时建立上下文的过程。

没有入驻直接开始工作，相当于一个人空降到 unfamiliar 公司直接写代码——不知道项目规范、不知道最近改了什么、不知道哪些地方有坑。

## 入驻步骤

```
Step 1: PROJECT.md          → "这个项目是什么？有哪些模块？用什么技术？"
Step 2: AGENTS.md           → "这个项目的铁律是什么？测试怎么跑？有哪些约定？"
Step 3: openspec/changelog/ → "最近改了什么？5 条足够了解趋势"
Step 4: docs/               → "涉及的业务规则是什么？"（如涉及）
Step 5: 目标源代码           → "具体代码怎么实现的？"
```

## 每步要获取什么

| 步骤 | 读取目标 | 关键问题 |
|------|---------|---------|
| PROJECT.md | 项目全局 | 项目叫什么？有哪些子模块？各模块的职责是什么？技术栈是什么？ |
| AGENTS.md | 工作规则 | 铁律是什么？各模块的测试命令？编码规范？禁止事项？ |
| changelog | 变更历史 | 最近 5 个变更都是什么？改了什么文件？有没有踩坑记录？ |
| docs/ | 业务规则 | 涉及哪些业务表/数据流？有什么业务约束？ |
| 源代码 | 实现细节 | 具体的代码逻辑、数据流、接口设计 |

## 不可跳级的理由

| 跳过 | 后果 |
|------|------|
| 跳过 PROJECT.md | 不知道项目有哪些模块，做了重复劳动或方向跑偏 |
| 跳过 AGENTS.md | 不知道铁律，必然踩已有坑 |
| 跳过 changelog | 不知道最近变更，可能和昨天的修改冲突 |
| 跳过 docs/ | 不理解业务规则，实现逻辑和业务语义矛盾 |

## 入驻完成标志

入驻完成的信号：

- ✅ 能说出项目名称和各模块的职责
- ✅ 知道各模块的测试/lint 命令
- ✅ 了解最近 3-5 个变更的大致内容
- ✅ 知道 AGENTS.md 中的铁律和关键约定
- ✅ 清楚自己接下来要做什么

完成入驻后，才能开始读目标源码或动手改代码。

## 特殊情况

### 项目没有 PROJECT.md 或 AGENTS.md

如果项目初始化文档体系尚未建立：

1. 使用此 Skill 帮用户生成 PROJECT.md 和 AGENTS.md 骨架
2. 参考 `references/templates.md` 中的模板
3. 通过读取 README.md、package.json、目录结构来填充基本信息

### 项目有自定义文档体系

如果项目不使用 OpenSpec 而有自己的文档约定：

1. 尊重已有约定，不强套 OpenSpec 格式
2. 将已有约定的要点总结到 AGENTS.md 中，方便其他 Agent 快速理解
