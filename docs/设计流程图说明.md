## 1. 文本切割 (Text Chunking)

### 1.1 功能概述

KB-Core 采用 **Parent-Child 双层递归切分** 策略,把上传的 TXT / Markdown 文档切成两层 chunk:

- **Parent chunk(≈600 tokens)**:保留较完整上下文,作为检索命中后回填的完整段落。
- **Child chunk(≈200 tokens)**:用于向量检索,提高召回精度。

切分后的 chunk 会被向量化(`text-embedding-v3`, 1024 维),存入 PostgreSQL `pgvector`,HNSW 索引加速后续检索。

### 1.2 关键配置

| 配置项 | 默认值 |
|---|---|
| `CHUNK_PARENT_TOKENS` | 600 |
| `CHUNK_CHILD_TOKENS` | 200 |
| `CHUNK_OVERLAP_TOKENS` | 30 |
| `EMBEDDING_BATCH_SIZE` | 10 |
| `EMBEDDING_DIM` | 1024 |
| 支持扩展名 | `.txt`, `.md` |
| 单文件上限 | 50 MB |
| UPDATE 批大小 | 25 行 |

### 1.3 文本切割总流程

```mermaid
flowchart TD
    A["用户上传 .txt / .md"] --> B["POST /ingest 接收文件"]
    B --> C{"文件类型与大小校验"}
    C -->|不通过| X1["返回 4xx 错误"]
    C -->|通过| D["保存文件<br/>本地或 OSS"]
    D --> E["插入 documents 记录<br/>status = pending"]
    E --> F["加入 BullMQ 任务队列"]
    F --> G[("BullMQ Queue: ingest")]
    G --> H["Worker 拉取任务"]
    H --> I["启动 ingestDocument"]

    I --> J["① parse 解析阶段"]
    J --> J1["读取文件<br/>规范化文本"]
    J1 --> J2{"是否法律文档"}
    J2 -->|是| J3["docType 标记为 law"]
    J2 -->|否| J4["docType 标记为 general"]
    J3 --> K
    J4 --> K

    K["② chunk 切分阶段"] --> L["ParentChildSplitter 启动"]

    L --> L1["parent 层切分<br/>递归按段落、句号"]
    L1 --> L2["合并过短单元<br/>补齐 overlap"]
    L2 --> M["child 层切分<br/>在每个 parent 内再切"]

    M --> N["产出 parents + children"]

    N --> O["③ 持久化 parents"]
    O --> P["构建 parentIdMap"]
    P --> Q["持久化 children<br/>关联 parentId"]
    Q --> R["④ embed 向量化阶段"]

    R --> R1["按 batch = 10 分批"]
    R1 --> R2["调用 Embedding API"]
    R2 --> R3{"成功?"}
    R3 -->|否| R4["重试 3 次<br/>指数退避"]
    R4 --> R2
    R3 -->|是| R5["写入 embedding 向量"]
    R5 --> S["文档状态置为 ready"]
    S --> T["前端可见"]
```

### 1.4 双层 chunk 数据结构

```mermaid
flowchart LR
    subgraph PG ["PostgreSQL chunks 表"]
        P1["parent #0<br/>约 600 tokens"]
        C1["child #0.0<br/>约 200 tokens"]
        C2["child #0.1<br/>约 200 tokens"]
        C3["child #0.2<br/>约 200 tokens"]
        P2["parent #1<br/>约 600 tokens"]
        C4["child #1.0<br/>约 200 tokens"]
    end

    P1 --- C1
    P1 --- C2
    P1 --- C3
    P2 --- C4

    subgraph VEC ["pgvector HNSW 索引"]
        V1["embedding 1024 维"]
        V2["embedding 1024 维"]
        V3["embedding 1024 维"]
    end

    C1 -.-> V1
    C2 -.-> V2
    C4 -.-> V3
```

**设计要点:**

- 每个 parent 内嵌多个 child,parent 提供完整上下文,child 用于精确检索
- 命中 child 后,用 parentId 反查 parent 拿完整段落回填
- 状态机: `pending` → `done` / `failed`

## 2. 问答 (Q&A / RAG)

### 2.1 功能概述

KB-Core 的问答是基于 **WebSocket + Agent Loop** 的流式 RAG,核心特征:

- **协议层**:WebSocket `/ws/query`,SSE 流式 token 推送。
- **决策层**:LLM 自主决定调 Skill 还是 Tool,最多 5 轮迭代。
- **检索层**:混合检索 = 向量检索(pgvector cosine)+ 全文检索(tsvector + ILIKE 中文)+ RRF 融合 + Reranker 重排序。
- **多轮对话**:前端传 `history`,QueryAgent 直接拼接到消息列表。
- **断线恢复**:每个 query 持久化为 queryJob,断线后可续传。

### 2.2 关键配置

| 配置项 | 默认值 |
|---|---|
| `SEARCH_TOP_K` | 10 |
| `DENSE_TOP_K_MULTIPLIER` | 3 |
| `RRF_K` | 60 |
| `RERANK_TOP_K` | 20 |
| `DENSE_MIN_SIMILARITY` | 0.65 |
| `RERANK_MIN_SCORE` | 0.5 |
| `AGENT_MAX_ITERATIONS` | 5 |
| `AGENT_MAX_TOOL_CALLS` | 15 |
| `RESULT_CACHE_TTL_MS` | 300000 (5 min) |
| Skill max iter | 3 |
| WS schema topK | 1-50 |
| WS schema history max | 20 条 |

### 2.3 问答总流程

```mermaid
sequenceDiagram
    autonumber
    participant U as 用户
    participant WS as WebSocket
    participant A as MainAgent
    participant Q as QueryAgent
    participant L as LLM
    participant T as Tools / Skills
    participant R as HybridRetriever
    participant RR as Qwen3-Rerank

    U->>WS: 发送问题 + 历史 + topK
    WS->>WS: 鉴权、参数校验<br/>创建 queryJob
    WS->>A: 启动问答任务
    A->>Q: 进入 Agent Loop

    loop 最多 5 轮迭代
        Q->>L: 流式调用 LLM
        L-->>Q: 流式 token<br/>finish_reason = stop 或 tool_calls

        alt finish_reason = stop
            Q->>Q: 直接答案 = content
        else finish_reason = tool_calls
            loop 遍历每个 tool_call
                alt 调用 Skill
                    T->>T: Skill 内部最多 3 轮
                    Note over T: 多次调检索 + 1 次 LLM 综合
                    T-->>Q: 返回 SkillResult
                else 调用 Tool 如 search_knowledge
                    T->>R: 检索 query + topK
                    par dense 轨
                        R->>R: pgvector 向量检索
                    and sparse 轨
                        R->>R: tsvector + ILIKE 全文检索
                    end
                    R->>R: RRF 融合
                    R->>RR: 重排序 top20
                    RR-->>R: rerank 分数
                    R->>R: 过滤 + 去重
                    R-->>T: topK chunks
                end
            end
        end
    end

    Q->>Q: resolveFinalAnswer 决定终止路径
    alt 终止 = skill
        Q-->>U: 透传 SkillResult 答案
    else 终止 = direct
        Q-->>U: 直接答案
    else 终止 = synthesis
        Q->>L: 再发一轮 LLM 综合
        L-->>Q: 综合答案
        Q-->>U: 流式输出答案
    end
```

### 2.4 检索内部流程(Hybrid Retriever)

```mermaid
flowchart TD
    A["接收检索请求<br/>query + topK + datasetId"] --> B["query 向量化<br/>命中 Embedding 缓存"]
    B --> C["extend = topK x 3"]
    C --> D["双轨并行检索"]

    D --> E["dense 轨<br/>pgvector 向量检索"]
    D --> F["sparse 轨<br/>ILIKE 中文 + tsvector 英文"]

    E --> E1["过滤未向量化 chunk"]
    E1 --> E2["cosine 相似度<br/>过滤阈值 0.65"]
    E2 --> E3["走 HNSW 索引"]

    F --> F1["提取关键词<br/>中文 2-12 字 + 4 字滑窗"]
    F1 --> F2{"包含英文?"}
    F2 -->|是| F3["tsvector 全文检索"]
    F2 -->|否| F4["ILIKE 模糊匹配"]
    F3 --> F5["合并去重"]
    F4 --> F5

    E3 --> G["RRF 倒数排名融合"]
    F5 --> G
    G --> H["取前 20 作为重排候选"]

    H --> I["候选文本用 parent 替换 child"]
    I --> J["调用 Qwen3-Rerank 重排序"]
    J --> J1{"API 成功?"}
    J1 -->|否| J2["fallback 原排序"]
    J1 -->|是| J3["得到 rerank 分数"]
    J2 --> K
    J3 --> K["过滤 + 去重<br/>保留分数不低于 0.5 的"]
    K --> L["同 parent 只保留 1 个"]
    L --> M["截取 topK"]
    M --> N["检索结果列表<br/>进 TTL 缓存"]
```

### 2.5 三种终止路径

```mermaid
flowchart LR
    A["QueryAgent Loop 结束"] --> B{"终止类型"}
    B -->|skill| C["透传 Skill 的答案"]
    B -->|direct| D["LLM 直接答案"]
    B -->|synthesis| E["再发一轮 LLM 综合"]

    C --> F["前端流式渲染 + citations"]
    D --> F
    E --> F
```

| 终止类型 | 触发条件 | 是否二次 LLM |
|---|---|---|
| `skill` | LLM 调用 Skill 且返回 SkillResult | 否,直接透传 |
| `direct` | LLM 直接回复(闲聊) | 否 |
| `synthesis` | LLM 只调用了 Tool,未命中 Skill | 是,再综合一轮 |

## 3. 知识图谱 (Knowledge Graph)

### 3.1 功能概述

KB-Core 使用 **Neo4j** 存储知识图谱,数据来源是 `app/data/kg-data.json`(Zod 校验后入库)。

- **节点类型(4 种)**: Flow(流程)、Law(法规)、Evidence(证据)、Case(案例)。
- **关系类型(8 种)**: NEXT / BRANCH_TO / APPLIES_TO / REQUIRES / MAY_REQUIRE / REFERS_TO / KEY_EVIDENCE / CITES / RELATED。
- **双向同步**:Law 节点的法条原文通过 splitter + embedding 写入 PostgreSQL chunks,关联字段 kg_node_id ↔ chunk_id。
- **前端可视化**: D3.js 力导向布局,4 个预设视图,支持拖拽 / 缩放 / 双击重定位。
- **与 RAG 协同**: mediation-advisor Skill 同时调图谱工具和检索工具,双轨综合。

### 3.2 关键配置

| 配置项 | 默认值 |
|---|---|
| `KG_ENABLED` | true |
| `NEO4J_URL` | `bolt://localhost:7687` |
| `NEO4J_USER` / `NEO4J_PASSWORD` | `neo4j` / `neo4j_dev_password` |
| `KG_DEFAULT_ROOT_LABOR` | `flow_labor_apply` |
| `KG_DEFAULT_ROOT_NEIGHBOR` | `flow_neighbor_register` |
| Driver `maxConnectionPoolSize` | 50 |
| Driver `connectionTimeout` | 5000ms |
| `kg_subgraph` 默认 `depth` | 2 (范围 1-3) |
| `kg_search_nodes` `limit` | 20 (1-50) |
| `kg_neighbors` `limit` | 50 (1-200) |
| `kg_path` `maxDepth` | 5 (1-10) |
| 节点颜色 | Law=蓝 / Case=绿 / Evidence=橙 / Flow=紫 |
| 节点形状 | Law=矩形 / Case=圆形 / Evidence=菱形 / Flow=圆角矩形 |

    C1 --> D["初始化 Neo4j"]
    D --> D1["建立唯一性约束"]

    D --> D3["建立全文索引"]

    D3 --> E["清除旧节点<br/>按 datasetId"]
    E --> F["按 Label 分组<br/>批量创建节点"]

    F --> F1["Flow 流程节点"]
    F --> F2["Law 法规节点"]
    F --> F3["Evidence 证据节点"]
    F --> F4["Case 案例节点"]

    F1 --> G["批量建立关系"]

### 3.4 边类型映射

| edge.label 关键字 | Neo4j relType | 视觉 |
|---|---|---|
| 下一步 / 同意调解 / 达成一致 / 申请司法确认 / 归档 | NEXT | 实线 |
| 包含 拒绝 或 未达成一致 | BRANCH_TO | 实线 / 虚线 |
| 法律依据 | APPLIES_TO | 实线 |
| 包含 关键证据 | KEY_EVIDENCE | 实线 |
| 需要核对 / 需要材料 或包含 需采集 / 调取 / 提供 | REQUIRES | 虚线 |
| 包含 可选 | MAY_REQUIRE | 虚线 |
| 包含 参考案例 | REFERS_TO | 实线 |
| 包含 援引 | CITES | 实线 |
| 其他 | RELATED | - |

## 总结

| 功能 | 入口 | 核心算法 | 存储 | 前端 |
|---|---|---|---|---|
| 文本切割 | POST /ingest | Parent-Child 双层递归 + 多级分隔符 | PostgreSQL chunks + pgvector HNSW | Documents 管理页 |
| 问答 | WS /ws/query | Agent Loop + Hybrid Retriever | 复用 chunks + queryJob | Chat 页(流式 + citations) |
| 知识图谱 | POST /kg/ingest + GET /kg/* | 关系映射 + Law → chunks 同步 | Neo4j + chunks.kg_node_id | KnowledgeGraph 页(D3 力导向) |

三大功能通过 kg_node_id 字段与 mediation-advisor Skill 实现双轨协同,使系统在 RAG 召回之上叠加了知识图谱的结构化推理能力。
