# Knowledge Core - V6 设计文档

> Agent 自主驱动。Skill 纯 SKILL.md 驱动——零代码，LLM 读 Markdown 指令执行。

---

# 一、核心理念

**传统 RAG**：`Query → 检索 → 生成`（固定管线，所有问题走同一条路）

**本设计**：`Query → Agent（LLM）自主决策 → 按需调用 Skill / Tool → 自动编排组合 → 回答`

```
用户："对比 Docker 和 K8s 在微服务场景的优缺点"

Agent 自主思考:
  1. 这是对比类问题
  2. 调用 compare skill
  3. SkillExecutor 读 compare/SKILL.md 指令，LLM 按步骤执行
  4. 信息够了 → 生成对比表格

用户："你好"

Agent 自主思考:
  1. 闲聊，不需要检索
  2. 直接回答

分层原则：Agent 层灵活（LLM 自主决策），Skill 层固化（SKILL.md 指令定义最佳实践流程）。
```

# 三、整体架构

```text
┌──────────────────────────────────────────────────────────┐
│                    Agent（LLM 自主驱动）                   │
│                                                          │
│  System Prompt（描述所有可用 Skill / Tool + 使用场景）     │
│       +                                                  │
│  Function Definitions（Skill + Tool 的 JSON Schema）     │
│       ↓                                                  │
│  Agent Loop:                                             │
│    THINK  → 我该调什么？                                  │
│    CALL   → function calling（Skill 或 Tool）            │
│    OBSERVE → 看结果                                       │
│    DECIDE → 继续调？还是回答？                            │
│    ... 循环直到 LLM 认为可以回答 ...                      │
│                                                          │
│  三种终止路径：                                            │
│    ① Skill 返回 answer → 直接透传（不二次生成）            │
│    ② 只调了 Tool → 最后做一次 synthesis 合成               │
│    ③ LLM 直接回答（没调任何 function）                     │
└──────────┬─────────────────────────┬─────────────────────┘
           │                         │
     调用 Skill                直接调用 Tool
           │                         │
┌──────────▼──────────────────┐  ┌───▼──────────────────────┐
│   Skill 层（纯 SKILL.md）    │  │   Tool 层（原子操作）      │
│                             │  │                          │
│  SkillLoader 启动时扫描      │  │  search_knowledge        │
│  src/skills/*/SKILL.md      │  │  get_document            │
│  解析 YAML frontmatter      │  │  get_chunk               │
│  → metadata + instructions  │  │  list_documents          │
│                             │  │  summarize_text          │
│  SkillExecutor 执行时：      │  │                          │
│    system = SKILL.md 正文    │  │  ← SkillExecutor 内部调用 │
│    LLM 按指令逐步调 Tool     │  │                          │
│                             │  └──────────────────────────┘
│  skills/qa/SKILL.md         │
│  skills/compare/SKILL.md    │  ┌──────────────────────────┐
│  skills/multihop/SKILL.md   │  │   Hooks（横切拦截）       │
│  ...（只有 .md，零代码）     │  │   before: 审计/限流       │
│                             │  │   after: 日志/后处理      │
│  基础设施（非 Skill）：       │  │   异常隔离（不影响主流程） │
│  loader.ts   扫描 + 解析     │  └──────────────────────────┘
│  executor.ts mini Agent Loop│
│  registry.ts 动态注册        │  ┌──────────────────────────┐
│  types.ts    类型定义        │  │   Data 层                 │
└─────────────────────────────┘  │   PostgreSQL + pgvector   │
                                 │   + tsvector              │
                                 └──────────────────────────┘
```

**关键变化**：
- Agent 没有"意图→Skill"硬编码映射。System Prompt 告诉 LLM 有哪些 Skill，LLM 自己判断。
- Skill 没有代码文件。SKILL.md 是唯一真相——frontmatter 定义 metadata，正文定义执行步骤。
- Agent 调用 Skill 时，SkillExecutor 读 SKILL.md 正文作为 LLM 系统指令，LLM 按步骤执行。

## 5.3 chunks（Parent-Child + 双索引 + 外键修复）

```typescript

export const chunks = pgTable('chunks', {
  id: uuid('id').primaryKey().defaultRandom(),
  documentId: uuid('document_id')
    .references(() => documents.id, { onDelete: 'cascade' }).notNull(),

  // 修复 #10：datasetId 有外键约束
  datasetId: uuid('dataset_id')
    .references(() => datasets.id, { onDelete: 'cascade' }).notNull(),

  parentId: uuid('parent_id'),        // self-FK 通过 migration SQL 添加

  // 修复 #12：拆为两个字段
  parentChunkIndex: integer('parent_chunk_index').notNull(),
  childIndexWithinParent: integer('child_index_within_parent'),  // null = parent 自身

  content: text('content').notNull(),
  contentHash: text('content_hash').notNull(),
  tokenCount: integer('token_count').notNull(),
  embedding: vector('embedding', { dimensions: 1024 }),
  embeddingStatus: embeddingStatusEnum('embedding_status').notNull().default('pending'),
  scope: text('scope').notNull().default('platform'),
  createdAt: timestamp('created_at').defaultNow().notNull(),
}, (table) => [
  index('chunk_embedding_idx').using('hnsw', table.embedding.op('vector_cosine_ops')),
  index('chunk_document_idx').on(table.documentId),
  index('chunk_parent_idx').on(table.parentId),
  index('chunk_dataset_idx').on(table.datasetId),
  uniqueIndex('chunk_doc_parent_child_uniq')
    .on(table.documentId, table.parentChunkIndex, table.childIndexWithinParent),
]);
```

# 十二、Skill 系统 ★（纯 SKILL.md 驱动）

## 12.1 设计理念

参考 OpenCode / Claude 的 Skill 模式：**零代码，纯 Markdown 驱动**。

| 维度 | 传统模式 | 纯 SKILL.md 模式 |
|---|---|---|
| Skill 文件 | SKILL.md + index.ts + script/*.ts | **只有 SKILL.md** |
| metadata 来源 | index.ts 硬编码 | SKILL.md YAML frontmatter |
| 执行逻辑 | TypeScript 代码 | SKILL.md 正文 = LLM 指令 |
| 改流程 | 改代码 + 重新编译 | 改 SKILL.md 即可 |
| 加新 Skill | 写 class + 注册 | 建目录 + 写 SKILL.md |

## 12.3 SKILL.md 格式

```markdown
---
name: qa
description: "基于知识库的事实问答。检索相关资料，LLM 生成带引用标注的回答。"
tools:
  - search_knowledge
parameters:
  type: object
  properties:
    query:
      type: string
      description: "用户的事实性问题"
  required:
    - query
---

# QA Skill

你的任务是根据知识库内容，回答用户的实际问题。

## 执行步骤

1. 从参数中取出 `query`
2. 调用 `search_knowledge` 工具，参数：{ "query": query, "topK": 10 }
3. 检查检索结果，如果为空回复"资料不足"
4. 根据资料生成回答，用 [1][2] 标注引用
5. 输出最终回答
```

**两部分**：
- **YAML frontmatter**（`---` 之间）→ SkillMetadata（name + description + tools + parameters）
- **Markdown 正文**（`---` 之后）→ instructions（LLM 执行指令）

## 12.4 Skill 类型定义

```typescript
interface SkillMetadata {
  name: string;
  description: string;
  tools: string[];
  parameters: {
    type: 'object';
    properties: Record<string, JSONSchemaProperty>;
    required: string[];
  };
}

interface Skill {
  metadata: SkillMetadata;        // 来自 SKILL.md frontmatter
  instructions: string;            // 来自 SKILL.md 正文
}

// 注意：没有 execute 方法！执行由 SkillExecutor 统一处理。
```

## 12.5 SkillLoader（扫描 + 解析）

```typescript
class SkillLoader {
  constructor(private skillsBaseDir: string) {}

  async loadAll(): Promise<Map<string, Skill>> {
    // 1. 扫描 skillsBaseDir 下所有子目录
    // 2. 每个子目录读 SKILL.md
    // 3. 正则提取 YAML frontmatter
    // 4. yaml.parse() → SkillMetadata
    // 5. 正文 → instructions
    // 6. 返回 { metadata, instructions }
  }
}
```

## 12.6 SkillExecutor（mini Agent Loop）

```typescript
class SkillExecutor {
  async execute(
    instructions: string,          // SKILL.md 正文
    allowedTools: readonly string[], // frontmatter 声明的 tools 白名单
    ctx: SkillContext,
  ): Promise<SkillResult> {
    // 1. SKILL.md 正文 → system prompt
    // 2. params → user message
    // 3. 只暴露 allowedTools 声明的 Tool
    // 4. mini Agent Loop:
    //    LLM 按指令调 Tool → 看结果 → 继续调或回答
    // 5. 返回 SkillResult
  }
}
```

**执行流程**：

```
Agent 调用 skill "qa"
  ↓
SkillExecutor.execute(qa/SKILL.md 正文, ['search_knowledge'], ctx)
  ↓
  system: "你的任务是根据知识库回答问题。步骤：1. 调 search_knowledge..."
  user:   "query: Docker 是什么"
  tools:  [search_knowledge]
  ↓
  LLM 读指令 → 调 search_knowledge({ query: "Docker 是什么", topK: 10 })
  ↓
  结果返回给 LLM
  ↓
  LLM 按步骤 4 生成带 [1][2] 引用的回答
  ↓
  返回 SkillResult { answer, citations: [], toolCalls }
```

## 12.7 SkillRegistry（动态注册）

```typescript
async function createSkillRegistry(skillsBaseDir: string): Promise<SkillRegistry> {
  const loader = new SkillLoader(skillsBaseDir);
  const skillMap = await loader.loadAll();
  const registry = new SkillRegistry();
  for (const skill of skillMap.values()) {
    registry.register(skill);
  }
  return registry;
}
```

## 12.8 6 个内置 Skill

| Skill | SKILL.md 指令概要 | 可用 Tool |
|---|---|---|
| qa | 检索 top 10 → 生成带引用回答 | search_knowledge |
| search | 检索 → 返回原文片段（不生成） | search_knowledge |
| summary | 检索 top 20 → 提取 3-5 要点 | search_knowledge |

# 十三、Agent ★（三种终止路径 + SkillExecutor 集成）

## 13.1 核心原则

Agent 是 **LLM + System Prompt + Function Calling Loop**。LLM 自主决策调什么 Skill/Tool。

## 13.2 三种终止路径（修复 #1 双重生成）

| 路径 | 条件 | 行为 |
|---|---|---|
| **skill** | LLM 调了 Skill | Skill 返回 answer，Agent **直接透传**（不二次生成） |
| **synthesis** | LLM 只调了 Tool（没调 Skill） | Agent 最后做一次合成生成 |
| **direct** | LLM 没调任何 function | 用 LLM 的 content 作为 answer |

## 13.3 Skill 结果处理（修复 #8 截断）

当 LLM 调用 Skill 时：
- SkillExecutor 返回 `{ answer, citations, toolCalls }`
- Agent **不** JSON.stringify 截断
- 只给 LLM 发摘要："Skill qa 完成，答案 1234 字，3 个引用"
- Agent 直接使用 Skill 的 answer 作为最终答案（不二次生成）

## 13.4 Agent Loop（核心代码结构）

```typescript
class QueryAgent {
  async execute(query, options): Promise<QueryResult> {
    // Agent Loop
    for (let iter = 0; iter < maxIterations; iter++) {
      const response = await llm.chat({ messages, tools: [...skills, ...tools] });

      if (!response.tool_calls?.length) {
        // Case C: LLM 直接回答
        directAnswer = response.content;
        break;
      }

      for (const call of response.tool_calls) {
        if (skillRegistry.has(call.name)) {
          // Case A: 调 Skill → SkillExecutor 执行
          const skill = skillRegistry.get(call.name);
          const executor = new SkillExecutor();
          const result = await executor.execute(
            skill.instructions,
            skill.metadata.tools,
            ctx,
          );
          skillResults.push(result);
          // 给 LLM 只发摘要（不截断原始内容）
          messages.push({ role: 'tool', content: summarize(result) });
        } else {
          // Case B: 调 Tool
          const result = await tool.execute(params, ctx);
          messages.push({ role: 'tool', content: JSON.stringify(result) });
        }
      }
    }

    // 决定最终答案
    if (skillResults.length > 0) {
      return { answer: skillResults[last].answer, termination: 'skill' };
    }
    if (directAnswer !== null) {
      return { answer: directAnswer, termination: 'direct' };
    }
    // synthesis 路径
    const final = await llm.chat({ messages: [...messages, "请给出最终回答"] });
    return { answer: final.content, termination: 'synthesis' };
  }
}
```

## 14.1 接口

```typescript
interface Hook {
  name: string;
  target: 'tool' | 'skill' | 'both';    // both = 同时拦截
  phase: 'before' | 'after';
  filter?: string | string[];
  before?(ctx: HookContext): Promise<HookActionResult | void>;
  after?(ctx: HookContext, result: unknown): Promise<unknown | void>;
}
```

## 14.2 异常隔离（修复 #18）

```typescript
// HookRegistry 内部用 try/catch 包裹每个 hook
try {
  const result = await hook.before?.(ctx);
  if (result?.block) return result;
} catch (err) {
  logger.warn(`[Hook "${hook.name}"] threw`, err);   // 只记录，不传播
}
```

**原则**：Hook 是旁路（审计、限流），不应该有能力搞挂主流程。

## 14.3 内置 Hooks

| Hook | 作用 |
|---|---|
| audit-before/after | 记录每次调用日志 + 耗时 |
| rate-limit | 60s 窗口内最多 N 次调用（防止 Agent 无限循环） |
