# Knowledge Core —— 基于 Agent 的智能知识库系统

> 技术分享文档 | 面向跨团队同事

---

## 一、为什么要做这个系统？

### 痛点

每个团队都有大量内部文档：技术文档、产品手册、运维手册、FAQ……

- **找信息难**：文档散落在各处，搜关键词经常搜不到（换个表述就找不到了）
- **理解成本高**：找到了相关文档，还要自己阅读、提炼、汇总
- **知识无法组合**：跨文档的对比、综合推理需要人工逐篇翻阅

### 目标

把任意格式的文档变成可问答的知识库，用自然语言直接获取答案，并附带来源引用。

```
用户："Docker 和 K8s 在微服务场景下有什么区别？"

系统：[自动检索相关资料 → 对比分析 → 生成结构化回答 + 引用来源]
```

## 二、核心概念速览（不熟悉的同事先看这里）

### RAG（检索增强生成）

传统大模型（LLM）只能靠训练数据回答问题，不知道你的内部文档。RAG 的思路：

```
用户问题 → 先从知识库检索相关资料 → 把资料喂给 LLM → LLM 基于资料回答
```

类比：开卷考试。LLM 是考生，知识库是允许带的参考书。

### Embedding（向量化）

把文本变成一组数字（向量），语义相近的文本向量也相近：

```
"Docker 容器化部署" → [0.12, -0.34, ..., 0.78]（1024 个数）
"容器虚拟化技术"    → [0.11, -0.32, ..., 0.77]（和上面很接近）
```

这样"容器化"和"虚拟化"虽然关键词不同，但向量距离很近，就能搜到。

### Function Calling（函数调用）

LLM 不只是输出自由文本，还能输出结构化的函数调用指令：

```json
{ "name": "search_knowledge", "arguments": { "query": "Docker 微服务", "topK": 10 } }
```

这让 LLM 能"操作"外部系统，而不只是"说话"。

## 三、架构总览

### 和传统 RAG 的区别

| 维度 | 传统 RAG | KB-Core |
|------|---------|---------|
| 流程 | 固定管线：检索 → 生成 | Agent 自主决策：按需编排 |
| 问题处理 | 所有问题走同一条路 | 不同问题用不同策略（Skill） |
| 扩展性 | 加新功能改代码 | 加一个 Markdown 文件即可 |
| 复合问题 | 无法处理 | Agent 自动组合多个 Skill |

### 分层架构

```
Agent 层（LLM 自主决策，调什么 Skill/Tool）
  └── Skill 层（纯 Markdown 驱动，固化最佳实践流程）
        └── Tool 层（原子操作：检索、查文档、摘要等）
              └── Data 层（PostgreSQL + pgvector 数据库）
```

**核心设计原则**：Agent 层灵活（LLM 自己判断该做什么），Skill 层固化（把最佳实践写成 Markdown 指令）。

### 一个问题的完整旅程

```
用户问："对比 Docker 和 K8s 在微服务场景的优缺点"

[Agent] LLM 读到 System Prompt 里 6 个 Skill 的描述
  → "这是对比类问题，调 compare skill"

[Skill] SkillExecutor 读取 compare/SKILL.md 的指令
  → "步骤 1：把对比对象拆出来 → Docker, K8s"
  → "步骤 2：分别检索"
  →   调 search_knowledge({ query: "Docker 微服务部署" })
  →   调 search_knowledge({ query: "K8s 微服务部署" })
  → "步骤 3：生成对比表格"

[Tool] search_knowledge 内部执行混合检索
  → Dense（向量语义搜索）+ Sparse（关键词搜索）→ RRF 融合 → Reranker 精排

[返回] 带 [1][2] 引用标注的结构化对比表格 → 用户
```

## 四、Agent 层 —— LLM 自主编排

### 核心机制

Agent = **LLM + System Prompt + Function Calling Loop**

```
┌─ Agent Loop ──────────────────────────────────────┐
│  THINK  → 我该调什么？                              │
│  CALL   → function calling（Skill 或 Tool）         │
│  OBSERVE → 看结果                                   │
│  DECIDE → 继续调？还是回答？                         │
│  ... 循环直到 LLM 认为可以回答 ...                   │
└───────────────────────────────────────────────────┘
```

- System Prompt 描述所有可用的 Skill 和 Tool（名称 + 描述 + 参数）
- LLM 自己判断该调什么，不需要硬编码 if/else 路由
- 新增 Skill 只需注册，LLM 自动发现

### 三种终止路径

| 路径 | 触发条件 | 行为 | 举例 |
|------|---------|------|------|
| **skill** | LLM 调了 Skill | Skill 答案直接透传（不二次生成） | 事实问答、对比分析 |
| **synthesis** | LLM 只调了 Tool | 最后做一次合成生成 | 直接检索后综合回答 |
| **direct** | LLM 没调任何 function | 用 LLM 的直接回答 | "你好"→ 闲聊回复 |

**为什么不二次生成？** Skill 内部已经完成了高质量的生成（带引用、结构化），Agent 再合成一次会丢信息且浪费 token。

## 五、Skill 层 —— 纯 Markdown 驱动

### 设计理念

这是系统最独特的设计：**Skill 没有代码文件，只有一个 SKILL.md**。

| 维度 | 传统模式 | 我们的模式 |
|------|---------|-----------|
| Skill 文件 | class + 代码逻辑 | 只有 SKILL.md |
| 执行逻辑 | TypeScript 代码 | Markdown 正文 = LLM 指令 |
| 改流程 | 改代码 + 编译部署 | 改 SKILL.md 即刻生效 |
| 加新 Skill | 写代码 + 注册 | 建目录 + 写 Markdown |

### SKILL.md 示例

```markdown
---
name: qa
description: "基于知识库的事实问答。检索相关资料，LLM 生成带引用标注的回答。"
tools:
  - search_knowledge
parameters:
  type: object
  properties:
    query:
      type: string
      description: "用户的事实性问题"
  required:
    - query
---

# QA Skill

你的任务是根据知识库内容，回答用户的实际问题。

## 执行步骤

1. 从参数中取出 query
2. 调用 search_knowledge 工具，参数 { "query": query, "topK": 10 }
3. 检查检索结果，如果为空回复"资料不足"
4. 根据资料生成回答，用 [1][2] 标注引用
5. 输出最终回答
```

**两部分**：
- **YAML frontmatter**（`---` 之间）→ 告诉 Agent：这个 Skill 叫什么、干什么、需要什么参数
- **Markdown 正文**（`---` 之后）→ 告诉 LLM：执行时按什么步骤来

### 6 个内置 Skill

| Skill | 用途 | 策略 |
|-------|------|------|
| **qa** | 事实问答 | 检索 top 10 → 带引用回答 |
| **search** | 原文检索 | 检索 → 只返回原文片段 |
| **summary** | 主题摘要 | 检索 top 20 → 提取 3-5 个要点 |
| **compare** | 对比分析 | 拆对象 → 分别检索 → 对比表格 |
| **chat** | 闲聊 | 直接回复（不走检索） |
| **multihop** | 多跳推理 | 迭代检索（最多 N 轮）→ 综合推理 |

### 执行机制

SkillExecutor 是 Skill 的运行时引擎，本质是一个 mini Agent Loop：

```
SKILL.md 正文 → 作为 system prompt
用户参数 → 作为 user message
只暴露 SKILL.md 声明的 tools（白名单控制）
→ LLM 读指令 → 逐步调 Tool → 生成结果
```

## 六、Tool 层 —— 原子操作

5 个 Tool，每个只管一件事：

| Tool | 职责 | 关键参数 |
|------|------|---------|
| `search_knowledge` | 混合检索知识库 | query, topK |
| `get_document` | 获取文档元数据 | documentId |
| `get_chunk` | 获取文本块 + 父级上下文 | chunkId |
| `list_documents` | 列出文档 | limit |
| `summarize_text` | LLM 文本摘要 | text, instruction |

Agent 和 SkillExecutor 都可以调 Tool，Tool 也可以被单独测试。

## 七、检索系统 —— Hybrid Search

### 检索流程

```
用户 Query
  │
  ├─→ Dense（向量语义检索）─── pgvector HNSW ──→ top K×3
  │                                                │
  ├─→ Sparse（关键词检索）─── tsvector GIN ──→ top K×3
  │                                                │
  └────────────────── RRF 融合 ────────────────────┘
                        │
                    Reranker 精排（Cross-Encoder）
                        │
                    去重（按 Parent Chunk）
                        │
                    最终 top K 结果
```

### 为什么需要混合检索？

| 只用 Dense（向量） | 只用 Sparse（关键词） | 混合 |
|---|---|---|
| 搜"怎么用 Docker" → 能匹配"如何使用容器"（语义近） | 搜"第39条" → 精确匹配第39条（关键词） | **两者都能搜到** |
| 搜"第39条" → 可能匹配"第40条"（语义近但不精确） | 搜"怎么用 Docker" → 文档写"如何使用容器"就搜不到 |  |

### RRF 融合

RRF（Reciprocal Rank Fusion）只看排名不看绝对分数：

```
score(d) = 1/(60 + rank_in_dense) + 1/(60 + rank_in_sparse)
```

好处：Dense 分数 0~1，Sparse 分数可能 0~100，直接加权会被大数值主导。RRF 只看排名，不需要归一化，不需要调参。

### Parent-Child 分块

矛盾：小块向量精准但上下文不够，大块上下文够但向量不精准。

解法：

```
Parent（1200 token，无向量，上下文用）
  ├── Child 1（300 token，有向量，检索用）
  ├── Child 2
  └── Child 3
```

检索时用 Child 向量精准匹配，命中后返回 Parent 提供完整上下文。

类比：查字典——用关键词找到页码（Child），翻开后读整页（Parent）。

## 八、数据入库 —— 异步流水线

```
文件上传
  → nanoid 重命名（防路径遍历攻击）
  → fileHash 查重（防止重复入库）
  → 丢进 BullMQ 队列 → HTTP 立即返回
  → 后台 Worker 异步处理：
      Parse（文件解析）→ Chunk（智能切分）→ Embed（向量化）
      每步记录 IngestJob（可追溯、可重试）
  → 状态更新为 ready
```

文档状态机：`pending → parsing → chunking → embedding → ready`（任何步骤失败 → `failed`）

## 九、技术选型一览

| 层面 | 选择 | 核心理由 |
|------|------|---------|
| 运行时 | **Bun** | 原生 TypeScript，快 4-5x，工具全内置 |
| 语言 | **TypeScript (strict)** | 类型安全，Drizzle 强类型推断 |
| Web 框架 | **Hono** | 最快，Bun 原生 Web Standard API |
| ORM | **Drizzle** | pgvector 原生支持，schema 即类型 |
| 数据库 | **PostgreSQL 16** | 一个库搞定三件事：关系 + 向量 + 全文检索 |
| Embedding | **BGE-M3 (1024 维)** | 中文最强，可本地部署 |
| Reranker | **qwen3-rerank** | Cross-Encoder 精排 |
| 向量索引 | **pgvector + HNSW** | 高召回率(95-99%)，复用 PG 连接 |
| 全文检索 | **tsvector + GIN** | 持久化，同库零运维 |
| 任务队列 | **BullMQ + Redis** | 成熟稳定，支持重试/优先级 |

**核心选型原则**：能用一个系统就不开两个（PostgreSQL 搞定关系、向量、全文三件事），避免引入不必要的运维复杂度。

## 十、Hooks —— 横切拦截

独立于主流程的旁路系统，用于审计和限流：

```typescript
interface Hook {
  name: string;
  target: 'tool' | 'skill' | 'both';   // 拦截什么
  phase: 'before' | 'after';            // 调用前还是调用后
  before?(ctx): Promise<HookActionResult | void>;
  after?(ctx, result): Promise<unknown | void>;
}
```

**关键设计**：每个 Hook 用 try/catch 隔离，Hook 抛异常不影响主流程。Hook 是旁路（审计、限流），不应该有能力搞挂主流程。

内置 Hooks：
- **audit**：记录每次 Skill/Tool 调用日志 + 耗时
- **rate-limit**：60s 滑动窗口限流（防止 Agent 无限循环）

## 十一、数据库设计

6 张核心表：

| 表 | 职责 |
|---|---|
| `datasets` | 知识库（多租户隔离） |
| `documents` | 文档元数据 + 状态机 |
| `chunks` | 文本块（Parent-Child 自引用 + 向量 + 全文索引） |
| `ingest_jobs` | 入库任务追踪（每阶段记录） |
| `query_logs` | 查询日志（含引用、Tool 调用记录） |
| `agent_traces` | Agent 决策全链路追踪 |

**chunks 表是核心**：

```sql
-- 向量索引（HNSW）用于语义检索
CREATE INDEX chunk_embedding_idx ON chunks USING hnsw(embedding vector_cosine_ops);

-- 全文索引（GIN）用于关键词检索
ALTER TABLE chunks ADD COLUMN tsv tsvector GENERATED ALWAYS AS (to_tsvector('simple', content)) STORED;
CREATE INDEX chunk_tsv_idx ON chunks USING GIN(tsv);

-- Parent-Child 自引用
ALTER TABLE chunks ADD CONSTRAINT chunks_parent_id_fkey
  FOREIGN KEY (parent_id) REFERENCES chunks(id) ON DELETE CASCADE;
```

## 十三、设计亮点总结

### 1. Skill = Markdown，零代码扩展
不需要写代码、不需要部署。写一个 SKILL.md 放到目录里，系统自动发现。改流程只需要改 Markdown。

### 2. Agent 无硬编码路由
没有 if/else 判断用户意图。System Prompt 描述所有 Skill，LLM 自己判断用哪个。新增 Skill 自动被发现，不需要改路由代码。

### 3. 三种终止路径避免浪费
Skill 已经生成好了就透传（不二次 LLM），闲聊就直接回答（不走检索），只有需要综合时才 synthesis。每条路径都是最优选择。

### 4. 一个 PostgreSQL 搞定三件事
关系数据 + 向量检索 + 全文检索，不需要额外部署 Qdrant / Elasticsearch。降低运维成本，简化数据一致性。

### 5. Hook 异常隔离
旁路系统不影响主流程。审计挂了不影响问答，限流挂了不影响数据入库。
